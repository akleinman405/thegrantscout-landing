## 4. SCHEMA DESIGN REVIEW

### Design Strengths ✅

1. **Clear Separation of Concerns**
   - Production vs source tables
   - Core entities vs analytics tables
   - Clean foreign key relationships

2. **Flexible JSONB Usage**
   - `intelligence_profile` field for portfolio analysis
   - `match_reason` and `score_components` for explainability
   - Future-proof for schema evolution

3. **Comprehensive Field Coverage**
   - All Build Plan requirements met
   - Financial, geographic, programmatic data
   - Contact and application process fields

4. **Strong Primary Keys**
   - EIN-based uniqueness
   - Auto-incrementing IDs where appropriate
   - Composite keys for relationships

### Design Weaknesses 🔴

1. **Array Storage Inconsistency** (CRITICAL)
   ```
   foundations.ntee_codes:     TEXT (should be ARRAY)
   foundations.program_areas:  TEXT (should be ARRAY)
   nonprofits.ntee_codes:      ARRAY ✅ (correct)
   nonprofits.program_areas:   ARRAY ✅ (correct)
   ```
   **Impact:** Cannot use array operators for NTEE matching on foundations table
   **Fix Required:** Migrate TEXT columns to ARRAY type

2. **Missing Indexes** (HIGH)
   ```
   board_members:           No indexes (will cause performance issues when populated)
   network_relationships:   No indexes
   f990_officers:          No indexes
   matches:                No indexes (match_score queries will be slow)
   ```
   **Impact:** Query performance degradation when tables grow
   **Fix Required:** Add indexes before population

3. **Soft Foreign Keys** (MEDIUM)
   ```
   matches.nonprofit_ein:      No FK constraint
   matches.foundation_ein:     No FK constraint
   matches.grant_id:          No FK constraint
   board_members.nonprofit_ein: No FK constraint
   ```
   **Impact:** No referential integrity enforcement
   **Fix Required:** Add FK constraints for data integrity

4. **Redundant BMF Partitions** (MEDIUM)
   ```
   irs_bmf:      1,898,175 records ✅
   irs_bmf_eo1:    270,505 records (duplicate)
   irs_bmf_eo2:    700,488 records (duplicate)
   irs_bmf_eo3:    922,562 records (duplicate)
   irs_bmf_eo4:      4,620 records (duplicate)
   ```
   **Impact:** 1.9M duplicate records, wasted storage
   **Fix Required:** DELETE eo1-4 partitions

5. **No Materialized Views** (LOW)
   - Expensive aggregations recalculated on every query
   - No pre-computed match scoring
   - Example: "Top 100 funders by state" query runs full table scan

   **Fix Required:** Create materialized views for common queries

#### 2. Build Automated Matching Engine (Est: 40 hours dev)
**Problem:** Beta reports created via 100% manual research
**Impact:** Not scalable beyond 3-5 clients
**Solution:** Implement 3-tier matching algorithm from Build Plan

**Phase A: Tier 1 Matching (Direct Funders)**
```sql
-- Find foundations that gave directly to client or peers
SELECT f.ein, f.name, COUNT(*) as grant_count,
       AVG(hg.grant_amount) as avg_grant
FROM foundations f
JOIN historical_grants hg ON f.ein = hg.foundation_ein
WHERE hg.recipient_ein = {client_ein}
   OR hg.recipient_ein IN (SELECT peer_ein FROM peer_orgs WHERE beta_org_id = {client_id})
GROUP BY f.ein, f.name
ORDER BY grant_count DESC;
```

**Phase B: Tier 2 Matching (Peer-Based)**
- Identify peer organizations (same NTEE + region)
- Find funders who give to ≥3 peers
- Score by peer overlap

**Phase C: Tier 3 Matching (Thematic/Geographic)**
- NTEE code similarity
- Geographic overlap (state_restrictions)
- Keyword matching in grant purposes
- Grant size fit

**Expected Outcome:**
- 500-1000 scored matches per client
- Match generation time: <5 minutes
- Explainable match reasoning
- Automated monthly updates

**Timeline:** 40 hours (20h algorithm, 10h testing, 10h integration)

## 7. DATA GAPS & LIMITATIONS

### What's Missing (Cannot Be Fixed)

1. **Pre-2020 Historical Grants**
   - Only 2020+ data available in XML format
   - Earlier years require PDF parsing (not feasible)

2. **Declined Applications**
   - Only successful grants visible
   - Cannot see rejection patterns
   - Limits success probability modeling

3. **Private Foundation Guidelines**
   - Most foundations don't publish detailed criteria
   - Application processes often opaque
   - Requires manual research

4. **Real-Time Deadline Updates**
   - Foundation websites change frequently
   - No API for deadline data
   - Requires periodic re-scraping

5. **Individual Grants**
   - 50-60% of pf_grants recipients are individuals
   - No EINs available by design
   - Cannot track or analyze

### What Can Be Improved

1. **NTEE Code Coverage**
   - 95% of IRS BMF has NTEE codes
   - 100% of foundations have NTEE (from BMF)
   - Enables precise programmatic matching

2. **Contact Information**
   - Foundation websites have email/phone
   - Can scrape during profile enrichment
   - Improves application preparation

3. **Grant Cycle Timing**
   - Multi-year data reveals patterns
   - Can calculate quarterly distributions
   - Helps with "best time to apply" guidance

4. **Peer Organization Identification**
   - NTEE codes enable similarity matching
   - Geographic data supports regional peers
   - Strengthens Tier 2 matching
