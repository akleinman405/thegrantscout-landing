# Lessons Learned: Phase 3 Fundraising Signals Analysis

**Date:** 2025-12-11
**Task:** Identify F990 signals predicting active fundraising activity

---

## 1. What Worked Well

### Proxy Signal Approach
- When direct fundraising fields weren't available, derived metrics proved effective
- Volunteer-to-employee ratio is a strong proxy for community engagement
- Grant dependency percentage accurately identifies organizations reliant on charitable giving
- Fiscal year end timing is a practical signal for "active campaign NOW"

### Validation Strategy
- Quick web checks on top 20 confirmed scoring model validity
- 67% hit rate (91% excluding technical failures) exceeded expectations
- Seeing actual donation infrastructure (NeonCRM, Givebutter) validated the approach

### Score Distribution
- Natural distribution across 0-10 range provides good segmentation
- Clear tier separation: top 18.5% score 7+, enables meaningful prioritization
- Reduces Phase 4 scraping pool by 66% while maintaining high-quality candidates

## 2. What Was Harder Than Expected

### Missing F990 Fields
- **Expected:** Schedule G data (special events, fundraising events)
- **Reality:** Not imported or not consistently filed
- **Impact:** Had to rely entirely on proxy signals

### Web Validation Limitations
- Many nonprofit sites use Wix, Squarespace, or Divi frameworks
- These render content via JavaScript - raw HTML shows only framework code
- Can't reliably confirm fundraising presence without headless browser

### Fiscal Year Data Quirks
- `tax_period_end` required join to `nonprofit_returns` table
- Some records have NULL or unusual months
- December (49.5%) and June (36.6%) dominate - standard nonprofit FYE months

## 3. What I Wish I Knew Going In

### Database Schema Realities
1. **No Schedule G table exists** - despite CLAUDE.md mentioning it as a possibility
2. **Fiscal year end** is in `nonprofit_returns`, not `prospects`
3. **YoY revenue growth** field exists but is 100% NULL for prime prospects
4. The `prospects` table is already a curated view - good for efficiency

### F990 Fundraising Signals Are Sparse
- IRS forms don't require granular fundraising reporting for most nonprofits
- Schedule G (fundraising events) is only required for events >$15K gross receipts
- Private foundations (990-PF) have more detailed grant data than charities (990/990-EZ)

## 4. Recommendations for Future Similar Tasks

### Before Starting
1. **Run column inventory first** - know what fields actually exist before designing hypotheses
2. **Check NULL rates** - some fields exist but are empty (like `yoy_revenue_growth`)
3. **Verify table relationships** - know which joins are needed

### For Fundraising Analysis Specifically
1. **Schedule G import would be valuable** - if we parse XML for it, we'd have:
   - Fundraising event revenue
   - Gaming revenue
   - Professional fundraiser compensation

2. **Consider external data sources:**
   - GuideStar/Candid for more detailed 990 fields
   - ProPublica Nonprofit Explorer API
   - Form 990 Schedule G from IRS directly

3. **Web validation needs headless browser** for accurate results on modern sites

### For Scoring Models
1. **Start with binary flags**, then convert to weighted points
2. **Test multiple thresholds** - our 6+ vs 7+ analysis was valuable
3. **Validate early** - spot-check top 10-20 before finalizing model
4. **Document the SQL** - makes replication and iteration easy
