## Issues Encountered & Retries

### 1. Wrong column name: `fg.purpose` (doesn't exist)

**What happened:** The prompt SQL referenced `fg.purpose` but the actual column in `fact_grants` is `purpose_text`. Query failed.

**Retry cost:** 1 extra query round-trip.

**Root cause:** Prompt was written without checking the schema first. The column name is documented in CLAUDE.md and data-dictionary.md.

**Fix for next time:** Always inspect schema before writing SQL in prompts, OR have Claude validate the SQL before running it. Could also add a "validate SQL columns" step to the prompt template.

---

### 2. MCP postgres "aborted transaction" state

**What happened:** After the `fg.purpose` error, the MCP postgres tool entered an aborted transaction state. Every subsequent query failed with "current transaction is aborted, commands ignored until end of transaction block."

**Retry cost:** 5 queries failed (all parallel sibling calls). Had to switch to psql via Bash for all remaining work.

**Root cause:** MCP postgres tool wraps queries in transactions. After an error, the transaction is poisoned and can't recover without a ROLLBACK, which the tool doesn't expose.

**Fix for next time:** Known issue (documented in CLAUDE.md gotchas and MEMORY.md). After any MCP postgres error, immediately switch to psql via Bash. Don't waste time retrying MCP. Consider starting with psql for multi-query sessions.

---

### 3. Shell escaping: `!~` operator in psql `-c` flag

**What happened:** The `!~` regex negation operator in PostgreSQL was interpreted by the shell when passed via `psql -c "..."`. The backslash-bang caused a syntax error.

**Retry cost:** 3 queries failed (the nonprofit count + prospect + campaign queries). Fixed by switching from `-c` flag to heredoc (`<<'EOSQL'`).

**Root cause:** Double-quoted strings in bash interpret `!` as history expansion. The `-c` flag passes the SQL through the shell.

**Fix for next time:** Always use heredocs (`<<'EOSQL'`) for psql queries containing regex operators, especially `!~`, `!~*`, or any `!` character. The single-quoted heredoc delimiter prevents all shell interpretation.

---

### 4. Temp tables lost between psql calls

**What happened:** Created `tmp_linkedin_orgs` in one psql call, then tried to JOIN to it in a separate psql call. Table didn't exist — each psql invocation is a separate session.

**Retry cost:** 1 failed query. Had to restructure as CTEs within a single query, or export to CSV and process in Python.

**Root cause:** `CREATE TEMP TABLE` only lives for the duration of the database session (connection). Each `psql` command opens a new connection.

**Fix for next time:** Two options:
1. **Single psql call:** Put all dependent queries in one heredoc
2. **CSV pipeline:** Export intermediate results to CSV, then process in Python

Option 2 (CSV pipeline) turned out to be more flexible and is the recommended pattern for complex multi-step data pulls.

---

### 5. `%%` escaping for LIKE patterns in psql `-c`

**What happened:** `LIKE 'B%'` in a psql `-c` string needed `'B%%'` because `%` is a format specifier in some contexts. Inconsistent behavior between `-c` and heredoc.

**Retry cost:** 0 (caught proactively after issue #3), but it added complexity.

**Fix for next time:** Another reason to always use heredocs. With `<<'EOSQL'`, no escaping is needed — SQL goes through verbatim.

## Recommended Best Practices

1. **Start with psql via Bash** for any session that will run 3+ queries. MCP postgres is fine for quick one-offs but fragile for real work.
2. **Always use heredocs** (`<<'EOSQL'`), never `-c` flags. Zero escaping issues.
3. **CSV pipeline pattern** for multi-step data pulls: export intermediate results to `/tmp/*.csv`, process in Python. More flexible than temp tables.
4. **Schema-check prompts** before writing SQL. Include `SELECT column_name FROM information_schema.columns WHERE ...` as a preamble step.
5. **After any MCP error**, don't retry MCP — switch to psql immediately.
