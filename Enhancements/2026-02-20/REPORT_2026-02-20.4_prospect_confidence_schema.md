## Executive Summary

This session investigated how VetsBoats foundation recommendations are stored, discovered that the database schema is well-designed but largely unpopulated, and designed a unified replacement table (`calc_client_prospects`) that consolidates three existing tables into one with auto-computed confidence tiers.

Key findings:
- **143 foundations** matched to VetsBoats via sibling analysis, but LASSO scores are all NULL and the recommendations table has zero rows for VetsBoats
- **125 foundations** discovered through a new veteran keyword search that were NOT in the existing match list (different discovery method = different prospects)
- **None of VetsBoats' 5 known funders** would appear in the 125 veteran + PF-funder + open-to-apps list
- Three agents deliberated and converged on a single-table design with auto-computed A-E confidence tiers and separate human-assigned report status

### Agent 2: Scoring Logic Analyst

Reverse-engineered the confidence criteria from VetsBoats' actual report data:

**Critical finding: ~50% of final report decisions overrode the algorithmic tier.** Overrides go both directions:

| Foundation | Algo Tier | Final Status | Direction | Reason |
|-----------|-----------|--------------|-----------|--------|
| Kovler Family | B (no targets) | IN REPORT | Upgraded | Enrichment found adaptive sailing funding |
| Barry Foundation | C (preselected) | IN REPORT | Upgraded | Scale + perfect alignment |
| Van Beuren | C (preselected) | IN REPORT | Upgraded | Sailing connection path |
| SF Challenge | C (preselected) | DROPPED | Downgraded | Frozen grantee list |
| 1687 Foundation | B (open+geo) | WATCH | Downgraded | Zero PF grants, 88% TX |
| Liberty Mutual | B (open+geo) | SKIP | Downgraded | "Excludes grantmaking agencies" |

Proposed 4-level confidence framework: HIGH (apply now), MODERATE (one question to resolve), LOW/CULTIVATION (relationship-building), DISQUALIFIED (skip).

### Agent 3: Schema Designer

Proposed a single unified table with auto-computed confidence tiers. Key design decisions:

1. **Single table**, not a star schema (under 12K rows, no performance reason to normalize)
2. **Confidence tier computed by trigger**, not manually assigned
3. **report_status separate from confidence_tier** -- tier = data, status = human decision
4. **source column** tracks discovery method (sibling analysis, keyword search, LASSO, manual)
5. **comparable_grants as JSONB** replaces proof_of_fit_grants table

## Proposed Schema: calc_client_prospects

### Table Structure

```sql
CREATE TABLE f990_2025.calc_client_prospects (
    -- IDENTITY
    client_ein          VARCHAR(9)   NOT NULL,
    foundation_ein      VARCHAR(9)   NOT NULL,

    -- DISCOVERY SOURCE
    source              VARCHAR(30)  NOT NULL DEFAULT 'sibling_analysis',
    source_details      TEXT,

    -- FOUNDATION INFO (denormalized)
    foundation_name     VARCHAR(255),
    foundation_state    VARCHAR(2),
    foundation_assets   BIGINT,

    -- SIBLING ANALYSIS (populated by pipeline)
    siblings_funded             INT DEFAULT 0,
    grants_to_siblings          INT DEFAULT 0,
    total_amount_to_siblings    BIGINT DEFAULT 0,
    median_grant_to_siblings    BIGINT DEFAULT 0,
    target_grants               INT DEFAULT 0,
    target_first_grants         INT DEFAULT 0,
    geo_grants                  INT DEFAULT 0,
    target_geo_grants           INT DEFAULT 0,
    gold_standard_grants        INT DEFAULT 0,
    most_recent_grant_year      INT,
    most_recent_target_year     INT,

    -- ELIGIBILITY FLAGS
    open_to_applicants  BOOLEAN,
    geo_eligible        BOOLEAN,
    pf_eligible         BOOLEAN,      -- NULL = unknown, TRUE/FALSE = confirmed

    -- MODEL SCORES
    lasso_score         NUMERIC(10,3),
    lasso_probability   NUMERIC(6,4),
    semantic_similarity NUMERIC(5,3),

    -- CONFIDENCE (auto-computed by trigger)
    confidence_tier     CHAR(1) NOT NULL DEFAULT 'U',
    confidence_score    SMALLINT,       -- 0-100 ranking within tier
    confidence_flags    TEXT[],         -- human-readable reasons array

    -- ENRICHMENT (populated manually or by enrichment pipeline)
    foundation_type     VARCHAR(30),    -- LOI-Ready, Open Application, Cultivation, etc.
    contacts            JSONB,
    application_url     TEXT,
    application_process TEXT,
    deadline            TEXT,

    -- REPORT FIELDS
    report_status       VARCHAR(20) DEFAULT 'pipeline',
    report_rank         SMALLINT,
    potential_award_min NUMERIC(12,2),
    potential_award_max NUMERIC(12,2),
    positioning_strategy TEXT,
    action_plan         TEXT,
    analyst_notes       TEXT,

    -- VERIFICATION
    verification_status VARCHAR(15),
    verified_at         TIMESTAMPTZ,

    -- COMPARABLE GRANTS
    comparable_grants   JSONB,

    -- TIMESTAMPS
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    reviewed_at         TIMESTAMPTZ,

    -- CONSTRAINTS
    PRIMARY KEY (client_ein, foundation_ein),
    CHECK (confidence_tier IN ('A', 'B', 'C', 'D', 'E', 'U')),
    CHECK (report_status IN ('pipeline','researching','in_report','watch','dropped','skip')),
    CHECK (source IN ('sibling_analysis','keyword_search','lasso_model','manual',
                       'client_referral','web_research'))
);
```

### Valid Values

**source:** How the foundation was discovered for this client.

| Value | Description |
|-------|-------------|
| `sibling_analysis` | Pipeline found via similar-org funding patterns |
| `keyword_search` | Found via mission/purpose keyword matching (e.g. "veteran sailing") |
| `lasso_model` | LASSO model scored as high-probability match |
| `manual` | Added manually by analyst |
| `client_referral` | Client identified this foundation |
| `web_research` | Found through web research or external tools |

**confidence_tier:** What the DATA says about this prospect (auto-computed).

| Tier | Label | Criteria |
|------|-------|----------|
| A | Definitely Pursue | Open + geo + target_grants > 0 + recent (3yr) |
| B | Probably Good | Open + geo + recent, no target grants |
| C | Worth Investigating | Has target grants but missing open or geo |
| D | Long Shot | Single signal only, weak evidence |
| E | Not Recommended | No qualifying signals |
| U | Unscored | Newly added, not yet evaluated |

**report_status:** What the HUMAN decided about this prospect.

| Status | Description |
|--------|-------------|
| `pipeline` | Raw output, not yet reviewed |
| `researching` | Currently being enriched/verified |
| `in_report` | Included in client report |
| `watch` | Good prospect, not in current report |
| `dropped` | Reviewed and rejected |
| `skip` | Clear disqualifier found |

## Confidence Tier Formula

### Tier Assignment Logic

```
IF open_to_apps AND geo_eligible AND target_grants > 0 AND recent (3yr):
    tier = A
ELIF open_to_apps AND geo_eligible AND recent:
    tier = B
ELIF target_grants > 0 AND recent:
    tier = C
ELIF (open_to_apps OR geo_eligible) AND grants_to_siblings > 0:
    tier = D
ELSE:
    tier = E
```

### Confidence Score (0-100, for ordering within tier)

| Signal | Points |
|--------|--------|
| Target grants (6 per grant, max 30) | 0-30 |
| Gold standard grants (15 per, max 15) | 0-15 |
| Open to applicants | 15 |
| Recency: within 1yr / 2yr / 3yr | 15 / 10 / 5 |
| Geographic match | 10 |
| LASSO score: >100 / >50 / >10 / >0 | 10 / 7 / 4 / 2 |
| PF eligible confirmed / PF ineligible | +5 / -10 |

### Auto-Computation via Trigger

Confidence tier, score, and flags are recomputed on every INSERT or UPDATE via a PostgreSQL trigger. When you update `open_to_applicants` after researching a foundation, the tier automatically adjusts.

### Why Tier and Status Are Separate

This is the most important design decision. Examples:

| Foundation | Tier | Status | Explanation |
|-----------|------|--------|-------------|
| Herzstein | A | in_report | Data and human agree |
| Jim Moran | A | skip | Data says great, but hard FL county restriction found |
| Barry Foundation | C | in_report | Data says barrier (preselected), but human included as cultivation target |
| SF Challenge | C | dropped | Data says barrier, enrichment confirmed (frozen grantee list) |

This separation lets us track "the algorithm said X but the reality was Y" -- essential for improving the algorithm over time.

## Notes

### Recommendations

1. **Implement the schema** -- Create the table, function, trigger, and run the migration. This is a clean consolidation with zero data loss.
2. **Backfill VetsBoats enrichment** -- The 18 prospect pipeline foundations should be updated with report_status and analyst_notes from the CSV.
3. **Add the 125 veteran keyword matches** -- Insert with `source = 'keyword_search'` for VetsBoats. These are net-new prospects not in the sibling analysis.
4. **Run LASSO model for VetsBoats** -- The model was never scored for VetsBoats (0 rows in fact_foundation_client_scores). This would add model scores to complement the sibling analysis.
5. **Foundation for Global Sports Development** -- Strongest single lead from the veteran keyword search. CA-based, open to apps, funds therapeutic programs for veterans through PFs. Recommend immediate enrichment.

### Key Insight

The VetsBoats case proved that **~50% of final report decisions override the algorithmic tier**. The algorithm is a useful pre-screen, but real confidence comes from enrichment. The proposed schema explicitly models this with separate `confidence_tier` (data) and `report_status` (human) fields, creating a feedback loop for algorithm improvement.

### Risk

The `source` CHECK constraint only allows predefined values. If a new discovery method emerges (e.g., "linkedin_research", "conference_lead"), it will require an ALTER TABLE. Consider whether to keep the CHECK or make it a soft convention.
