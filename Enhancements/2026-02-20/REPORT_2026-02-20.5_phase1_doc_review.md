## Executive Summary

This report synthesizes what seven primary documentation files and dozens of supporting reports say about the TheGrantScout database. The review found:

- **77 unique table/view names** referenced across all documentation (not all necessarily exist today)
- **20+ tables referenced in code that are NOT in the data dictionary** (the biggest gap)
- **12 significant contradictions** between documents on schema names, record counts, database host, column names, and table structures
- **The data dictionary was last updated 2026-02-13** and covers 35 objects; the database spec from 2025-12-08 is severely stale (documents a `public` schema that was likely replaced)
- **The Build Plan from 2025-11-21** describes a completely different data model (organizations_core, funders, grants_raw, etc.) that was never built as specified
- **process_client.py** reads/writes 7 tables: dim_clients, emb_nonprofit_missions, nonprofit_returns, fact_grants, calc_client_siblings, calc_client_sibling_grants, calc_client_foundation_scores, plus pf_returns and dim_foundations
- **build_cohorts_v3.sql** references 8 tables/views, at least 3 of which are not in the data dictionary (nonprofits_prospects2, foundation_prospects2, stg_foundation_state_dist)
- **Proposed calc_client_prospects table** (from REPORT .4 today) would consolidate three existing client tables

## Pipeline Flow: process_client.py

`process_client.py` is a single-file pipeline that processes one client at a time through foundation matching. Here is a step-by-step trace of what it reads and writes.

### Inputs

| Phase | Step | Operation | Table | Columns Used |
|-------|------|-----------|-------|-------------|
| Init | Get client | READ | `dim_clients` | id, name, ein, state, mission_text, database_revenue, matching_grant_keywords, target_grant_purpose |
| Init | Load model | N/A | (loads SentenceTransformer) | |

### Phase 1: Clear Existing Data

| Step | Operation | Table |
|------|-----------|-------|
| 1a | DELETE | `calc_client_foundation_scores` WHERE client_ein = X |
| 1b | DELETE | `calc_client_sibling_grants` WHERE client_ein = X |
| 1c | DELETE | `calc_client_siblings` WHERE client_ein = X |

### Phase 2: Find Siblings (Steps 2-3)

| Step | Operation | Table | Details |
|------|-----------|-------|---------|
| 2a | READ | `emb_nonprofit_missions` | mission_embedding_v (pgvector cosine distance) |
| 2b | READ (JOIN) | `nonprofit_returns` | total_revenue, tax_year |
| 2c | WRITE | `calc_client_siblings` | client_ein, sibling_ein, similarity, client_budget, sibling_budget, budget_ratio |
| 3a | READ | `fact_grants` | All grants where recipient_ein IN (siblings) |
| 3b | READ (subquery) | `fact_grants` | MIN(tax_year) for is_first_grant determination |
| 3c | WRITE | `calc_client_sibling_grants` | client_ein, foundation_ein, sibling_ein, grant_id, amount, tax_year, recipient_state, purpose_text, is_first_grant, purpose_quality |

### Phase 3: Semantic Similarity + Keywords (Steps 4-6)

| Step | Operation | Table | Details |
|------|-----------|-------|---------|
| 4a | READ | `calc_client_sibling_grants` | purpose_text WHERE purpose_quality = 'VALID' AND semantic_similarity IS NULL |
| 4b | UPDATE | `calc_client_sibling_grants` | SET semantic_similarity (computed in Python) |
| 5 | UPDATE | `calc_client_sibling_grants` | SET keyword_match (regex against purpose_text) |
| 6 | UPDATE | `calc_client_sibling_grants` | SET is_target_grant, target_grant_reason |

### Phase 4: Foundation Scores (Step 7)

| Step | Operation | Table | Details |
|------|-----------|-------|---------|
| 7a | READ | `calc_client_sibling_grants` | Aggregation by foundation_ein |
| 7b | READ (JOIN) | `dim_foundations` | name, state |
| 7c | READ (LATERAL) | `pf_returns` | total_assets_eoy_amt (most recent filing) |
| 7d | WRITE | `calc_client_foundation_scores` | 19 columns: client_ein, foundation_ein, foundation_name, foundation_state, foundation_total_assets, siblings_funded, grants_to_siblings, total_amount_to_siblings, median_grant_size_to_siblings, target_grants_to_siblings, target_first_grants, unknown_target_count, client_state, geo_grants_to_siblings, target_geo_grants, gold_standard_grants, most_recent_grant_year, most_recent_target_year, lasso_score (NULL) |

### Phase 5: Enrichment (Step 8)

| Step | Operation | Table | Details |
|------|-----------|-------|---------|
| 8a | READ (subquery) | `pf_returns` | only_contri_to_preselected_ind (most recent filing) |

## Email Pipeline Flow: build_cohorts_v3.sql

`build_cohorts_v3.sql` builds the foundation lists used for cold email campaigns to nonprofit prospects. It was implemented on 2026-02-17 as part of the email pipeline redesign.

### Input Tables

| Table | Role | Key Columns Used |
|-------|------|-----------------|
| `nonprofits_prospects2` | Source of target state+sector combos | state, ntee_code, contact_email |
| `fact_grants` | Grant history for aggregation | foundation_ein, recipient_ein, recipient_state, amount, tax_year |
| `dim_recipients` | NTEE code for recipients | ein, ntee_code, name |
| `dim_foundations` | Foundation names | ein, name |
| `calc_foundation_profiles` | Openness metrics | ein, accepts_applications, openness_score, new_recipients_5yr, unique_recipients_5yr, has_grant_history, last_active_year |
| `foundation_prospects2` | Foundation assets | ein, total_assets_eoy_amt |
| `stg_foundation_state_dist` | Per-state giving percentages | foundation_ein, recipient_state, state_pct |
| `mv_foundation_geo_relevance` | Geographic tier classification | foundation_ein, state, geo_tier, is_national_funder |
| `ref_foundation_email_exclusions` | Ban list | foundation_ein, exclusion_category |
| `nonprofit_returns` | For example grant revenue lookup | ein, total_revenue, tax_year |

### Output Tables

| Table | Created/Modified | Rows (reported) | Description |
|-------|-----------------|-----------------|-------------|
| `cohort_foundation_lists_v2` | DROP + CREATE TABLE AS | 21,667 | Top 25 foundations per state+sector combo, ranked by email_fit_score |
| `cohort_viability` | ALTER TABLE (add columns) | 882 | display_count and display_text columns added |

### Processing Steps

1. **Step 5 (main query):** Builds scored foundation list
   - Gets distinct state+sector combos from `nonprofits_prospects2` WHERE contact_email IS NOT NULL
   - Aggregates foundation giving from `fact_grants` for tax_year >= 2019 by recipient_state and NTEE sector
   - Joins `calc_foundation_profiles` for openness data
   - Joins `foundation_prospects2` for asset data
   - Joins `stg_foundation_state_dist` for state concentration
   - Joins `mv_foundation_geo_relevance` for geo tier
   - LEFT JOINs `ref_foundation_email_exclusions` to exclude banned foundations (with geographic monopoly carve-out)
   - Computes 5-dimension score: geo (0.30) + size_fit (0.25) + openness (0.20) + recency (0.15) + volume (0.10), times mega_penalty
   - Ranks by composite email_fit_score, keeps top 25 per combo

2. **Step 6 (example grants):** Three-tier cascade UPDATE
   - Tier 1: Same 3-char NTEE prefix + same state + 2021+
   - Tier 2: Same 2-char NTEE prefix + same state + 2019+
   - Tier 3: Same broad sector (1 char) + same state + 2019+
   - Deletes rows with no viable example grant

3. **Step 7 (cohort viability):** Adds display_count and display_text to `cohort_viability`
   - Caps display at 75, rounds to nearest 5
