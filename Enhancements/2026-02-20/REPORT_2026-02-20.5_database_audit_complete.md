## Executive Summary

This audit cataloged every object in the `thegrantscout` PostgreSQL database (17 GB) across 3 user schemas. Key findings:

- **79 base tables, 2 materialized views, 31 views, 5 custom functions, 10 triggers, 53 sequences** across f990_2025 (55 tables), filtered_campaign (10 tables), and grantscout_campaign (14 tables). The `public` schema has no user tables (only extension functions).
- **The matching pipeline is BROKEN.** Both `process_client.py` and the LASSO V6.1 scorer depend on embedding tables (`emb_nonprofit_missions`, `calc_foundation_avg_embedding`) that were dropped 2026-01-04. No new clients can be processed until embeddings are restored or the pipeline is rewritten.
- **Only 1 of 8 clients (PSMF) has been fully processed.** The other 7 have partial sibling data but zero LASSO scores, zero recommendations, and zero enrichment.
- **Data quality is strong at the source level** (100% grant purpose coverage, 8.3M cleaned grants) but **degrades through the pipeline** (56.1% recipient EIN coverage, 0% LASSO scores for 7 clients, hardcoded geo_eligible flag).
- **Documentation covers ~60% of tables.** The data dictionary was last updated 2026-02-13 and is missing all email pipeline tables (fp2, np2, cohort_*), all calc_client_* tables, and fact_foundation_client_scores.
- **Email coverage is the binding constraint** for outreach: 0.9% of nonprofits and 6.1% of foundations have email addresses.
- **11 empty tables** exist in grantscout_campaign (schema created ahead of implementation).

## Migration Readiness Assessment

### Proposed: calc_client_prospects Table

The proposal (from REPORT_2026-02-20.4) consolidates three tables into one:

| Source Table | Rows | PSMF Only? | Key Data |
|-------------|------|------------|----------|
| calc_client_foundation_scores | 4,816 | No (all 8 clients) | Sibling metrics, eligibility flags |
| calc_client_foundation_recommendations | 5 | Yes | Rank, contacts, strategy |
| calc_client_proof_of_fit_grants | 25 | Yes | Best-fit example grants |

### What Exists Now

- **All 3 source tables exist** with valid data and no orphaned records
- **Zero triggers/functions** for prospect confidence exist -- must be built from scratch
- **LASSO scores live in a separate table** (fact_foundation_client_scores) and are NOT in calc_client_foundation_scores
- **The geo_eligible flag is hardcoded TRUE** -- not useful for migration without fixing first

### What's Missing or Problematic

| Issue | Impact on Migration |
|-------|-------------------|
| lasso_score is NULL in all 4,816 rows of calc_client_foundation_scores | The proposed confidence tiers depend on LASSO scores. Must backfill from fact_foundation_client_scores (only PSMF has data). |
| geo_eligible is hardcoded TRUE | Confidence tier calculations that depend on geo_eligible will be meaningless. Must implement real geo checks first. |
| Only 5 recommendations and 25 proof-of-fit rows (all PSMF) | The consolidated table will be sparse for 7 of 8 clients. |
| VetsBoats recommendations live in CSV files, not the database | 18 enriched prospects from `5. Runs/VetsBoats/2026-02-06/DATA_2026-02-06_prospect_pipeline.csv` would need to be backfilled. |
| No ETL script found for the raw-to-clean transformation | Cannot rebuild dim/fact tables if needed. |

### Risks

1. **Data loss:** None if migration uses INSERT...SELECT from source tables. Source tables should be kept until migration is verified.
2. **LASSO score gap:** 7 of 8 clients have no LASSO scores. Migrating now would create a table that's 87.5% incomplete on the key scoring column.
3. **Dependency risk:** The LASSO scorer and process_client.py both depend on dropped embedding tables. Fixing the pipeline should happen BEFORE migration, not after.

### Recommendation

**Do not migrate yet.** The prerequisite steps are:

1. Restore or replace embedding tables
2. Re-run pipeline for all 8 clients
3. Run LASSO scorer for all 8 clients
4. Implement real geo_eligible checking
5. Then migrate to calc_client_prospects with complete data
