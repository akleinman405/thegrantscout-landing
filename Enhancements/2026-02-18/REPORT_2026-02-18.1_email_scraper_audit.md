## Executive Summary

The nonprofit email scraper has processed **16,147 of 293,778** sites (5.5%). It's finding emails at a **61% hit rate** among completed sites and producing **18,455 emails across 9,629 orgs**. The overall architecture is solid: async HTTP with rate limiting, confidence scoring, MX validation, and cross-org dedup cleanup.

However, the audit found **6 significant issues** that are reducing quality and completeness:

| Issue | Severity | Impact |
|-------|----------|--------|
| **52+ junk/placeholder emails** in DB | Medium | Pollutes data, embarrassing if emailed |
| **9 malformed emails** with `www.` in domain | Low | Invalid emails in DB |
| **0% name extraction** (0 of 18,455) | **High** | No name+email pairing for personalization |
| **39% false negative rate** (6,259 working sites, no email found) | **High** | Missing ~3,800 potential emails |
| **1,208 orgs at 5-email cap** losing best contacts | Medium | May lose ED/director emails |
| **No `mailto:` link extraction** (highest-signal source) | High | Missing structured email data and name context |

## False Negatives Analysis

**6,280 sites marked "not_found"** (no email extracted). This is 39% of all scraped sites.

### How hard did we try?

| Metric | Value |
|--------|-------|
| Sites with no email | 6,280 |
| Of those with 200 OK homepage | 6,259 (99.7%) |
| Average pages tried | 3.0 |
| Max pages tried | 3 (homepage + 2 additional) |

### Homepage Size Distribution (for not_found sites with 200 OK)

| Page Size | Sites | % |
|-----------|-------|---|
| 100KB+ (heavy, likely JS-rendered) | 2,930 | 46.8% |
| 20-100KB (normal page) | 2,457 | 39.3% |
| < 1KB (redirect/empty) | 365 | 5.8% |
| 5-20KB (light page) | 323 | 5.2% |
| 1-5KB (minimal) | 184 | 2.9% |

### Root Causes for Missed Emails

1. **JavaScript-rendered content (SPAs):** ~47% of not_found sites have 100KB+ pages but the email is rendered by JS. The scraper fetches raw HTML only, no JS execution.

2. **Email obfuscation:** Many nonprofits use CloudFlare email protection (`data-cfemail` attributes), JavaScript concatenation, CSS direction reversal, HTML entity encoding, or image-based email display.

3. **Contact forms only:** Many modern nonprofit sites use forms instead of displaying email addresses.

4. **Emails on deeper pages:** The scraper tries max 3 pages total (homepage + 2). Emails may be on `/board-of-directors`, `/our-team`, `/connect`, etc.

5. **Emails in page footer only:** Some sites only show email in the footer, which may be loaded via a template include or JS widget.

### Estimated Recoverable Emails

If we assume the same 61% find rate could be achieved on not_found sites with better extraction: ~3,800 additional emails could be recovered with JS rendering and obfuscation decoding.

## Name+Email Pairing Gap

### Current State: Zero Name Extraction

| Field | Total Records | Populated |
|-------|--------------|-----------|
| `person_name` | 18,455 | **0 (0%)** |
| `person_title` | 18,455 | **0 (0%)** |

The `web_emails` schema has `person_name` and `person_title` columns, but the scraper never writes to them. The `save_scrape_result` method in `db.py` doesn't include these fields in the INSERT statement, and the scraper's `EmailResult` dataclass doesn't have name/title fields.

### Where Names ARE Available

Names are often present in the same HTML near the email in these patterns:

1. **`mailto:` anchor text**: `<a href="mailto:john@org.com">John Smith, Executive Director</a>`
2. **Adjacent HTML**: `<p>John Smith<br>john@org.com</p>` or `<div class="staff"><h3>John Smith</h3><p>ED</p><p>john@org.com</p></div>`
3. **Structured data**: JSON-LD `ContactPoint` and Schema.org markup
4. **IRS officer records**: The `officers` table has name+title for every 990 filer. Can cross-reference EIN + last name from email local part.

### Impact

Without name+email pairing:
- Cold emails can't be personalized ("Dear John" vs "Dear Sir/Madam")
- Can't identify the right person to contact (ED vs webmaster)
- Can't filter by role (only want decision-makers, not admin@)
- Significantly reduces email response rates

## Code Architecture Review

### Strengths

1. **Clean async architecture**: `aiohttp` with semaphore-based concurrency control
2. **Per-domain rate limiting**: Prevents hammering individual servers
3. **Comprehensive exclusion patterns**: 50+ patterns covering platforms, CMS vendors, tracking
4. **Confidence scoring**: Multi-factor scoring (domain match, prefix priority, personal name pattern)
5. **Post-scrape cleanup**: Cross-org dedup, MX validation
6. **Graceful shutdown**: SIGINT/SIGTERM handling
7. **SSL fallback**: Tries HTTP if HTTPS fails
8. **HTML cleaning**: Strips `<script>`, `<style>`, `<noscript>` tags before extraction

### Weaknesses

1. **Regex-only extraction**: `_find_all_emails()` uses a single regex on cleaned HTML. No `mailto:` link parsing, no structured data extraction.
2. **No name extraction at all**: The `EmailResult` dataclass doesn't even have name fields.
3. **Static page limit**: Max 2 additional pages beyond homepage. For large nonprofit sites, this misses deep pages.
4. **No JavaScript rendering**: Sites using React, Vue, Angular, or even jQuery AJAX won't have emails extracted.
5. **No obfuscation decoding**: CloudFlare `data-cfemail`, ROT13, HTML entities, JS concatenation are not handled.
6. **Contact page discovery is simple**: Only checks link text for patterns. Doesn't look at nav menus, footer links, or sitemap.xml.
7. **User-Agent is fixed per session**: Headers are randomized once per `scrape_batch` call, not per request.
8. **`_clean_html` is expensive**: Parses full HTML with BeautifulSoup just to strip tags, then `_find_all_emails` runs regex on the string. Could strip tags with regex instead, or parse once and extract from soup.

## Enhancement Recommendations

### P1: Fix Junk Patterns + www. Strip (1-2 hours)

**Impact:** Remove ~70+ junk emails, prevent future ones

Add to `EXCLUDE_PATTERNS`:
- Placeholder domains: `@company\.com$`, `@yoursite\.com$`, `@yourdomain\.`, `@mysite\.com$`, `@domain\.com$`, `@site\.com$`, `@emailserver\.com$`, `@contoh\.com$`, `@ejemplo\.com$`, `@company-domain\.com$`
- Theme/plugin junk: `@enfold-restaurant\.com`, `@events\.frontend$`
- Invalid TLDs: `\.(frontend|tld)$`
- Exact matches: `^email@`, `^yourname@`, `^youremail@`, `^address@domain\.`
- Platform staging: `\.azurewebsites\.net$`, `\.netlify\.app$`, `\.herokuapp\.com$`, `\.vercel\.app$`

Also strip `www.` from email domains:
```python
if '@www.' in email:
    email = email.replace('@www.', '@')
```

### P1: `mailto:` Link Extraction + Name Pairing (4-6 hours)

**Impact:** Populate `person_name`/`person_title` for ~30-50% of personal emails, find emails regex misses

1. Parse `mailto:` links with BeautifulSoup: extract anchor text (often contains name/title)
2. Look for name in adjacent elements (parent/sibling of email-containing element)
3. Add `person_name` and `person_title` to `EmailResult` dataclass
4. Update `save_scrape_result` to write name/title fields
5. Cross-reference with IRS `officers` table for EIN+lastname matching

### P2: CloudFlare Email Decoding (1-2 hours)

**Impact:** Recover emails from ~5-15% of not_found sites

CloudFlare encodes emails as hex XOR. Simple decode:
```python
def decode_cf_email(encoded):
    r = int(encoded[:2], 16)
    return ''.join(chr(int(encoded[i:i+2], 16) ^ r) for i in range(2, len(encoded), 2))
```

### P2: HTML Entity Decoding (30 min)

**Impact:** Recover emails using `&#` encoding

Extract text from BeautifulSoup parsed tree (which auto-decodes entities) instead of regex on raw HTML string.

### P2: Increase Crawl Depth (30 min)

**Impact:** Find emails on deeper pages

Increase from 2 to 4 additional pages. Add to `DIRECT_PAGE_PATHS`: `/board-of-directors`, `/our-board`, `/governance`, `/directory`, `/people`, `/connect`, `/executive-team`

### P2: Smarter 5-Email Cap (30 min)

**Impact:** Keep best contact emails instead of losing them to team page noise

Always keep top role email + top personal email. Fill remaining 3-6 slots from others. Consider increasing cap to 8.

### P3: JS Concatenation Patterns (2-3 hours)

**Impact:** Recover emails from JS-obfuscated sites

Catch patterns like: `"info" + "@" + "org" + "." + "com"` and `String.fromCharCode(64)` without full JS execution.

### P3: User-Agent Rotation Per Request (10 min)

**Impact:** Reduce blocking rate

Rotate User-Agent per individual HTTP request instead of once per batch.

### P3: Confidence Scoring Fixes (1 hour)

- Only penalize short local parts when domain doesn't match
- Add `hi`, `we`, `us` to ROLE_PREFIXES
- Weight generic domain penalty by email context

### P4: Full JS Rendering with Playwright (8-12 hours)

**Impact:** High but heavy dependency. Would recover emails from SPA sites. Consider as a separate "deep scrape" pass for high-value not_found sites only.
