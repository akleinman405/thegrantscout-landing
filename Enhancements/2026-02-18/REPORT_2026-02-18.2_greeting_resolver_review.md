## Executive Summary

Of 1,122 cold emails sent on 2026-02-18, 94.5% used the wrong greeting name. The `contact_first_name` (IRS 990 ED/CEO name) was used as the greeting regardless of whether the email was a generic inbox (info@, admin@) or belonged to a different person. A recipient replied asking "who is Kathleen?" because the name in the greeting didn't match anyone at their organization.

A `greeting_resolver.py` module was implemented with a conservative approach: default to "Hello," unless the email prefix positively confirms the 990 officer name. Three independent review agents rigorously audited all changes. One critical bug was found and fixed (name casing normalization). The implementation is now approved with 61 passing tests and zero remaining risks.

**Expected impact:** ~98.6% "Hello," (safe generic) + ~1.4% verified "Hi Name," (exact or nickname match only). Zero wrong-name risk.

## The Bug

The root cause was threefold:

1. **Assumption violation:** The code assumed `contact_first_name` (from IRS 990 filings) corresponded to `contact_email` (from web scraping). In reality, the 990 lists the ED/CEO while the scraped email is often a generic inbox (info@, admin@, office@) or belongs to a different staff member.

2. **Three duplicate template copies:** `send_campaign.py`, `fix_email_quality.py`, and `regenerate_mission_shorts.py` each had independent copies of the email templates with `Hi {first_name},`. Fixing one file would leave the bug in the other two.

3. **No validation layer:** The QC stage checked that `contact_first_name` existed and wasn't a placeholder, but never checked whether the name matched the email recipient.

**Data from audit:**
- 68.4% (764 emails): Wrong name sent to generic inbox (info@, admin@, etc.)
- 26.1% (296 emails): Wrong name sent to different person's email
- 1.4% (16 emails): Correct name-email match
- 4.1% (46 emails): Other mismatches

## Implementation

### Architecture

A single `greeting_resolver.py` module serves as the single source of truth for greeting logic. All three consumer files import from it.

```
greeting_resolver.py
    resolve_greeting(contact_first_name, contact_email) -> (name_or_none, reason)
    format_greeting_line(greeting_name) -> str

send_campaign.py        -- imports greeting_resolver
fix_email_quality.py    -- imports greeting_resolver
regenerate_mission_shorts.py -- imports greeting_resolver
```

### resolve_greeting() Logic Flow

1. **Guards:** No email, junk email, placeholder officer name -> (None, reason)
2. **Generic inbox:** Prefix in ~120-entry set OR matches regex patterns -> (None, "generic_inbox")
3. **Length check:** Prefix > 20 chars -> (None, "generic_inbox")
4. **Split prefix** on `.` `_` `-` to get email_first_part
5. **Initial-only:** First part is 1 char -> (None, "initial_only")
6. **Exact match:** email_first == officer_name.lower() -> (Name, "exact_match")
7. **Nickname match:** Bidirectional lookup in ~75 name groups -> (Nickname, "nickname_match")
8. **Default:** (None, "no_match")

### Template Changes

All 5 template definitions across 3 files changed from `Hi {first_name},` to `{greeting_line}`. The `format_greeting_line()` function returns either "Hi Name," (verified match) or "Hello," (everything else).

### Pipeline Integration

| Stage | Change |
|-------|--------|
| PULL | Computes `greeting_name` and `greeting_reason` per row, adds to CSV |
| GENERATE | Reads pre-resolved greeting from CSV (or computes fresh for backward compat), uses `format_greeting_line()` in `.format()` calls |
| QC | Checks greeting coherence: personalized reasons must have "Hi Name," in body, all others must have "Hello," |
| SEND | No changes needed (body is opaque string) |

## Fixes Applied

After the review, the following issues were fixed:

### Fixed: C1 - Name casing normalization (CRITICAL)

**Before:** `return (officer, "exact_match")` -- raw input, could be "NANCY" or "nancy"

**After:** `return (officer.capitalize(), "exact_match")` -- always "Nancy"

### Fixed: H3 - Weak test assertion

**Before:** `test_no_match_bookstore` only asserted `name is None`, comment said "could be generic_inbox or no_match"

**After:** Explicitly asserts `reason == "no_match"`

### Fixed: H4 - Duplicate "membership" in GENERIC_EMAIL_PREFIXES

Removed duplicate entry at line 38. Set deduplicates automatically but this was a maintenance smell.

### Fixed: M2 - Missing nicknames

Added 14 nickname groups: henry, harold, walter, arthur, theodore, melanie, jacqueline, samantha, stephanie, nicole, valerie, abigail. Total now ~75 groups.

### Added: Test cases for casing

- `test_all_caps_officer_name`: "NANCY" + nancy@ -> ("Nancy", "exact_match")
- `test_lowercase_officer_name`: "nancy" + nancy@ -> ("Nancy", "exact_match")
- `test_nickname_samantha_sam`: "Samantha" + sam@ -> ("Sam", "nickname_match")
- `test_format_preserves_capitalization`: verifies output format

### Not Fixed (accepted risk)

| Issue | Severity | Reason for accepting |
|-------|----------|---------------------|
| H1: Compound names miss match | HIGH | Conservative miss (falls back to "Hello,"), not a safety issue. Very few compound names in data. |
| H2: Numeric suffixes miss match | HIGH | Conservative miss. john123@ -> "Hello," is safe. |
| M1: Junk extension substring match | MEDIUM | Extremely unlikely to have ".jpg" in a real email address |
| M3: Undocumented 20-char cutoff | MEDIUM | Works correctly, just needs a comment |
| M4: No logging | MEDIUM | Consumer scripts print summaries at each stage |
| M5: Hello vs Hi there | MEDIUM | "Hello," confirmed as the correct choice per postmortem |
| L1-L4 | LOW | Minor quality items, no functional impact |
| Agent 2 M1: SQL filter sync | MEDIUM | Defense in depth -- resolver catches anything SQL misses |
| Agent 2 M2: Report greeting stats | MEDIUM | Stats print to console; report can be updated later |

## Deliberation and Final Verdict

### Consensus Points (all 3 agents agree)

1. The architectural decision to centralize greeting logic in a single module is correct
2. The conservative approach (default "Hello," unless confirmed) is the right call given the 94.5% error rate
3. All template copies are updated -- no remaining `{first_name}` placeholders
4. Backward compatibility is handled correctly
5. The QC coherence check provides meaningful defense in depth

### Remaining Debate

- **Agent 1** flagged compound names and numeric suffixes as HIGH. **Agent 2 and 3** did not flag these because they're conservative misses (safe fallback to "Hello,"), not safety issues. **Resolution:** Accepted as-is. These reduce personalization rate slightly (~0.5% of prospects) but never produce wrong greetings.

- **Agent 2** flagged the SQL placeholder filter being out of sync. **Resolution:** Accepted. The SQL filter is defense in depth and the resolver handles all edge cases downstream. Syncing would require a SQL migration for marginal benefit.

### Final Verdict: APPROVED

After fixing C1 (name casing), all three agents' concerns are addressed. The implementation:
- Eliminates 100% of wrong-name greetings
- Maintains ~1.4% correct personalization rate
- Has 61 passing unit tests
- Is integrated into all 3 consumer files
- Provides 3 layers of defense (SQL filter, resolver, QC check)
