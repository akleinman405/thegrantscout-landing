## What Happened

On 2026-02-18, we sent 1,122 cold emails to nonprofit organizations. Every email opened with a personalized greeting like "Hi Kathleen," or "Hi Andrew,". A recipient replied asking "who is Kathleen?" because nobody named Kathleen worked there.

Audit results for all 1,122 sent emails:

| Category | Count | % | What the recipient saw |
|----------|-------|---|----------------------|
| Generic inbox + wrong name | 764 | 68.1% | "Hi Kathleen," arrives at info@org.com. Nobody knows a Kathleen. |
| Personal email + wrong name | 296 | 26.4% | "Hi Andrew," arrives at karen.gillis@org.com. Karen reads it. |
| First initial match (ambiguous) | 46 | 4.1% | Might be right, might not |
| Name actually matches email | 16 | 1.4% | Correctly personalized |

**94.5% of emails used the wrong name in the greeting.**

## What Went Wrong

### Root Cause 1: Two Unrelated Data Sources Treated as One

The email pipeline used two fields from the `nonprofits_prospects2` table:

| Field | Source | What it actually is |
|-------|--------|-------------------|
| `contact_first_name` | IRS 990 tax filing officer data | The ED/CEO who signed the tax return (could be 1-2 years old) |
| `contact_email` | CSV import from web scraping | The organization's public email address (generic inbox or random staff) |

**These two fields have no relationship to each other.** The name comes from whoever the IRS has on file as the responsible officer. The email comes from whatever address was scraped from the org's website. Nobody verified that the name matches the email recipient.

Example: The Festival Foundation has ED/CEO "KATHLEEN PAYE" on their 2023 990, but their scraped email is `mdepuy@cherryfestival.org` (M. Depuy, a different person). The pipeline sent "Hi Kathleen," to M. Depuy.

### Root Cause 2: No Validation Between Name and Email

The send pipeline (`send_campaign.py`) had three validation gates, but none checked whether the name matched the email:

| Gate | What it checked | What it missed |
|------|----------------|----------------|
| SQL WHERE clause (PULL) | Filtered out placeholder names like 'there', 'Hi' | Didn't check if name matches email |
| PLACEHOLDER_NAMES set (QC) | Rejected generic names like "director", "admin" | "Kathleen" passes because it's a real name |
| Quality flag check (SEND) | Checked for broken merge fields, em dashes | Never compared name to email address |

The pipeline's implicit assumption was: "if the name is a real first name (not a placeholder), it's safe to use." This is wrong because a real name can belong to a completely different person than the email recipient.

### Root Cause 3: Three Independent Template Copies

The greeting logic existed in three separate files, each with their own copy of the email templates:

| File | Template location | Format call |
|------|------------------|-------------|
| `send_campaign.py` | Lines 91-130 | Lines 538-550 |
| `fix_email_quality.py` | Lines ~133-157 | Lines 244, 254, 318, 342 |
| `regenerate_mission_shorts.py` | Line ~1197 | Line 1280 |

All three independently used `first_name=row["contact_first_name"]` to inject the 990 officer name directly into the greeting. Fixing only one file would leave the other two producing wrong greetings whenever they regenerate email bodies.

## Why Each Problem Existed

### Why were names extracted from 990 officer data?

The 990 filing is the only structured source of individual names for nonprofits in our database. When the email pipeline was built on 2026-02-16, extracting the ED/CEO name from the `all_officers` JSONB field achieved "88% first-name coverage." This sounded excellent, but coverage and correctness are different things. 88% of orgs had an extractable officer name; but that name matched the email recipient in only 1.4% of cases.

### Why weren't the names validated against emails?

The pipeline was built in phases. The name extraction (2026-02-16) and the email sending infrastructure (2026-02-17) were done in separate sessions. Name-email mismatch flagging was identified as a future improvement (listed as Priority #4 in the pipeline implementation report) but was never implemented before the first send.

### Why are 68% of emails going to generic inboxes?

Nonprofit contact emails in public data are overwhelmingly generic. Of the 4,356 sendable Tier A+B prospects: 63.1% have generic emails (info@, admin@, office@), 4.8% have generic variants, and only 32% have personal-looking email addresses. Even among personal addresses, 85.7% don't match the 990 officer name because the email belongs to whoever manages the website, not the ED/CEO.

### Why did Gage Academy get "Kathleen" when the ED is Mary Lloyd?

The 990 filing lists officers from the filing year (often 1-2 years old). Gage Academy's most recent filing showed Kathleen Allen as "Executive Director (Sep-Jun)" and Mary Lloyd as "Interim Executive Director (Jul-Sep)." The extraction logic found Kathleen Allen in the `all_officers` JSONB. But the email went to `info@gageacademy.org`, a generic inbox where nobody named Kathleen reads it anymore.

## The Fix and Why It's Complete

### The Fix: `resolve_greeting()` function

A new `greeting_resolver.py` module with a single function that determines the safe greeting for each email. The function returns `None` (use "Hello,") unless the email prefix **confirms** the 990 officer name.

### How it addresses each root cause:

**Root Cause 1 (unrelated data sources):** The function explicitly checks whether the email prefix matches the officer name. If `contact_email = info@org.com` and `contact_first_name = "Kathleen"`, it returns `(None, "generic_inbox")` because `info` is in the generic prefix set. If `contact_email = lqualls@org.com` and `contact_first_name = "Kathleen"`, it returns `(None, "no_match")` because `lqualls` doesn't match `Kathleen`. Only when the email prefix starts with the officer name (or a known nickname) does it return a name.

**Root Cause 2 (no validation):** The function IS the validation. It runs at PULL time (adding `greeting_name` and `greeting_reason` columns to the CSV), at GENERATE time (building the greeting line), and QC now checks `greeting_reason` to verify coherence. Three layers of defense:

| Layer | Check | Effect |
|-------|-------|--------|
| PULL | `resolve_greeting()` computes greeting | Wrong names never enter the CSV |
| GENERATE | `format_greeting_line()` builds "Hello," or "Hi Name," | Wrong names never enter the email body |
| QC | Checks `greeting_reason` matches email body | Catches any leaks in the pipeline |

**Root Cause 3 (three template copies):** All three files are updated to:
1. Import `format_greeting_line()` from the shared `greeting_resolver.py` module
2. Use `{greeting_line}` placeholder in their template strings instead of `{first_name}`
3. Call `resolve_greeting()` before formatting

This means the greeting logic lives in exactly one place. Any future change to greeting behavior only needs to touch `greeting_resolver.py`.

### Why we removed the "extract name from email" feature

An early version of the plan proposed extracting names from email prefixes (e.g., `jennifer.smith@org.com` -> "Hi Jennifer,"). The review team tested this against actual data and found an **83.8% false positive rate**: prefixes like `bookstore@`, `battleship@`, `coalition@`, and `favicon@` all pass basic name heuristics. We would have sent "Hi Bookstore," and "Hi Battleship,". The only safe path is matching against the 990 officer name, which we already have.

### Coverage of all edge cases

| Scenario | Emails affected | Resolution | Greeting |
|----------|----------------|------------|----------|
| Generic inbox (info@, admin@, etc.) | 764 of 1,122 sent | Prefix in GENERIC_EMAIL_PREFIXES (~120 entries) | "Hello," |
| Department inbox (boxoffice@, admissions@) | ~30 of 1,122 | Added to generic set | "Hello," |
| Different person's personal email | 296 of 1,122 | Name doesn't match prefix -> no_match | "Hello," |
| Initial-only prefix (j.smith@) | ~20 of 1,122 | Can't verify name from one letter | "Hello," |
| Exact name match (nancy@org.com + "Nancy") | 16 of 1,122 | Prefix matches officer name | "Hi Nancy," |
| Nickname match (pam@org.com + "Pamela") | ~5 estimated | Bidirectional NICKNAME_MAP | "Hi Pam," |
| Stale officer data (Gage Academy) | Unknown count | Name doesn't match generic inbox prefix | "Hello," |
| Junk emails (Google Calendar, .webp) | ~3 of 1,122 | Domain pattern detection | "Hello," |
| Placeholder officer name ("there") | Filtered by SQL | Filtered before reaching resolver | Excluded |

### Why the fix is complete

1. **Single source of truth:** `greeting_resolver.py` is the only place greeting logic lives
2. **All code paths covered:** All 3 files that format email bodies use the shared module
3. **Defense in depth:** SQL filter -> resolve_greeting() -> format_greeting_line() -> QC check
4. **Conservative by default:** Returns "Hello," unless there's positive confirmation the name matches
5. **No guessing:** Removed the risky "extract name from email" heuristic
6. **Tested against real data:** Every edge case tested against the actual 1,122 sent emails

## Damage Assessment

### What was sent

- **1,122 emails** sent on 2026-02-18, all with potentially wrong name greetings
- **764** went to generic inboxes where the wrong name looks like obvious spam
- **296** went to personal emails of the wrong person, which is the most damaging
- **16** were correctly personalized (these were fine)

### Expected impact

- Recipients at generic inboxes likely ignored or deleted the email. The wrong name reduces credibility but doesn't identify a specific person.
- Recipients at personal emails who noticed the wrong name would reasonably conclude this is automated spam with bad data. Worst case: they report as spam, which hurts domain reputation.
- The "who is Kathleen?" response was relatively benign. It shows the recipient engaged enough to reply rather than silently marking as spam.

### Mitigation

- Future sends will use "Hello," for ~98.6% of emails and verified "Hi Name," for ~1.4%
- No way to un-send the 1,122 emails already delivered
- Domain reputation impact is likely minimal given the small volume and warm-up phase
