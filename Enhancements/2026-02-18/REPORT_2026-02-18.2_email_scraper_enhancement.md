## Executive Summary

## Implementation Details

### Step 1: Junk Patterns + @www. Strip

**Files:** `email_scraper.py` (JUNK_PATTERNS + EXCLUDE_PATTERNS)

| Category | Patterns Added | Examples |
|----------|---------------|----------|
| Placeholder domains | 10 | `@company.com`, `@yoursite.com`, `@contoh.com` |

| Platform staging | 4 | `.azurewebsites.net`, `.netlify.app`, `.herokuapp.com` |

The `@www.` strip is applied in three places: `_extract_from_mailto()`, `_extract_from_cfemail()`, `_extract_from_json_ld()`, and the regex pass in `_find_all_emails()`.

### Step 2: EmailResult Fields

**Files:** `email_scraper.py` (dataclass)

Added 3 fields to `EmailResult`:
- `extraction_method: str = 'regex_scrape'` (values: mailto, cfemail, json_ld, regex_scrape)
- `person_name: Optional[str] = None`
- `person_title: Optional[str] = None`

Added `import json` for JSON-LD parsing.

### Step 3: Single-Parse Architecture

**Files:** `email_scraper.py` (`_clean_html`, `_find_all_emails`)

The old flow parsed HTML twice (once for cleaning, once for regex). The new flow:

```
raw_html
  -> BeautifulSoup(html)
  -> Pass 1: _extract_from_mailto(soup)     [before decompose]
  -> Pass 2: _extract_from_cfemail(soup)     [before decompose]
  -> Pass 3: _extract_from_json_ld(html)     [needs raw script tags]
  -> decompose script/style/noscript
  -> cleaned_text = soup.get_text()
  -> Pass 4: regex on cleaned_text + str(soup)
  -> dedup via seen set (structured results take priority)
  -> validate + sort by confidence
```

`_clean_html()` was refactored to return `tuple[BeautifulSoup, str]` but is no longer called by `_find_all_emails()` (which now does its own inline decompose). The method is retained for potential external use.

### Step 4: Extraction Methods

**Files:** `email_scraper.py` (4 new methods + 6 new class constants)

**4a. `_decode_cloudflare_email(encoded)`** - Static method. XOR cipher decoder copied from `05_extract_website_data.py` lines 203-215. Decodes `data-cfemail` hex attributes.

**4b. `_extract_from_mailto(soup, domain, source_page, source_url)`** - Finds all `<a href="mailto:...">` tags. Skips tags with `data-cfemail` (handled by Pass 2). Strips query params (?subject, ?cc). Extracts person name from anchor text using strict criteria:
1. Anchor text is NOT the email itself
2. Anchor text does NOT contain `@`
3. Anchor text is not a CTA phrase ("Email Us", "Contact Us", etc.)
4. Email local part is not role-based (info@, contact@, executivedirector@, etc.)
5. Name matches 2-3 capitalized words, each 3+ chars
6. No words in NAME_SKIP_WORDS (geographic, corporate, institutional terms)

Title extraction from anchor text uses pattern matching for Executive Director, President, CEO, etc.

**4c. `_extract_from_cfemail(soup, domain, source_page, source_url)`** - Finds elements with `data-cfemail` attribute, decodes using 4a.

**4d. `_extract_from_json_ld(html, domain, source_page, source_url)`** - Regex for `<script type="application/ld+json">` blocks on raw HTML. Recursively searches for `email` and `contactPoint` fields.

**New class constants:**

| Constant | Purpose |
|----------|---------|
| `NAME_PATTERN` | Regex: 2-3 capitalized words with optional middle initial |
| `TITLE_PATTERNS` | Regex: Executive Director, President, CEO, VP, etc. |
| `NAME_SKIP_WORDS` | Set: geographic, corporate, institutional terms to filter |
| `ROLE_LOCAL_PARTS` | Set: role-based email prefixes (no name extraction for these) |
| `NON_NAME_PHRASES` | Set: CTA anchor text to skip ("Email Us", "Click Here", etc.) |

### Step 5: Multi-Pass _find_all_emails()

**Files:** `email_scraper.py`

Complete rewrite of the method body. The old version was single-pass regex on cleaned HTML. The new version orchestrates all 4 extraction methods in priority order, deduplicates via a `seen` set (structured results added first take priority), then applies the existing validation pipeline (malformed TLD check, hash-based tracking check, confidence threshold).

### Step 8: Smart Email Cap

**Files:** `email_scraper.py` (`_smart_cap_emails`)

Replaced the simple `emails[:5]` cap with an intelligent `_smart_cap_emails(emails, max_total=8)`:
1. Always keep the best role email (info@, contact@, etc.)
2. Always keep the best personal email (john.smith@, etc.)
3. Fill remaining 6 slots by confidence score
4. Cap increased from 5 to 8

### Step 9: UA Rotation Per Request

**Files:** `email_scraper.py` (`_fetch_page`)

Moved User-Agent selection from session-level (`_get_headers()`, set once per batch) to per-request (`_fetch_page()`, rotated on every fetch). Applied to both the primary HTTPS request and the HTTP fallback in the SSL error handler.

### Step 10: Database Updates

**Files:** `db.py`

**10a. Upsert strategy:**
```sql
ON CONFLICT (ein, email) DO UPDATE SET
    confidence = GREATEST(web_emails.confidence, EXCLUDED.confidence),
    person_name = COALESCE(EXCLUDED.person_name, web_emails.person_name),
    person_title = COALESCE(EXCLUDED.person_title, web_emails.person_title),
    extraction_method = CASE
        WHEN EXCLUDED.confidence > web_emails.confidence
        THEN EXCLUDED.extraction_method
        ELSE web_emails.extraction_method
    END
```

This ensures: highest confidence always kept, person info is never overwritten with NULL, extraction method tracks the source of the best confidence.

**10b. INSERT columns:** Added `person_name`, `person_title`. Changed `extraction_method` from hardcoded `'regex_scrape'` to dynamic from EmailResult.

**10c. Batch commit:** Removed `self.conn.commit()` from `save_scrape_result()`. Single commit in `save_batch_results()`. Reduces DB round-trips from N to 1 per batch.

**10d. `fix_existing_emails(dry_run)`:** Scans all emails against new junk patterns, fixes @www. with savepoints (falls back to delete on unique constraint), reclassifies NULL email types.

**10e. `reset_not_found_for_rescrape(dry_run)`:** Resets `scrape_status` to NULL for all `not_found` rows.

### Critical Findings (Fixed)

**C1. Unicode escape deletion corrupts HTML**

The `UNICODE_ESCAPES` regex replaced `\u003c`/`\u003e` with empty string instead of decoding to `<`/`>`. This destroyed angle brackets in Wix-style JSON-embedded HTML, preventing mailto and cfemail extraction from finding `<a>` tags in those pages.

**Fix:** Replaced regex substitution with explicit `html.replace('\\u003c', '<')`.

**C2. `classify_email_type()` crashes on emails without `@`**

`local, domain = email.split('@', 1)` raises `ValueError` if the email has no `@`. This function is called from `db.py` on data already in the database, so malformed data from a prior version could crash the `classify` and `fix-existing` commands.

**Fix:** Added `if '@' not in email: return 'junk'` guard at the top.

## Known Limitations

1. **Name extraction is conservative.** Only extracts from mailto anchor text, not from proximity-based HTML context. This was intentional: the async scraper lacks the cached HTML that the `05_extract_website_data.py` pipeline uses for deep name extraction. Expected population rate: ~5% of personal emails.

2. **JSON-LD extraction requires raw script tags.** The extraction runs on raw HTML before BeautifulSoup decompose. If a site uses dynamically injected JSON-LD (via JavaScript), it won't be captured.

3. **No AT/DOT obfuscation support.** The `05_extract_website_data.py` reference file has AT/DOT pattern matching (`user [at] domain [dot] com`). This was intentionally omitted from the async scraper to avoid false positives in the noisier scraping context.

4. **Batch commit data loss window.** If the process is killed (SIGKILL, OOM) during `save_batch_results()`, the entire batch (up to 500 sites for overnight, 50 for scrape-emails) is lost. Graceful shutdown (SIGINT/SIGTERM) is handled correctly. The `reset_stuck_in_progress` mechanism provides recovery.

5. **Duplicate pattern lists.** `JUNK_PATTERNS` and `EXCLUDE_PATTERNS` overlap significantly. Updates to one must be mirrored to the other. A future refactor could consolidate them.

### Architecture Decision: Why Not Use 05_extract_website_data.py Directly?

The `05_extract_website_data.py` script operates on cached HTML files on disk (gzip-compressed). It is a CPU-bound batch extractor designed for the foundation pipeline's multi-phase workflow (fetch -> cache -> extract). The nonprofit email scraper is an I/O-bound async scraper that processes HTML in-memory without caching. Copying the extraction patterns (cloudflare decode, mailto, JSON-LD) while keeping the async architecture was the right tradeoff: we get the same extraction capabilities without the disk I/O overhead or the dependency on the caching pipeline.
