## Issues Found

### 1. URL-encoded email prefixes (78 emails)

Emails extracted from `href="mailto:%20info@org.com"` patterns where `%20` (URL-encoded space) was not decoded before extraction.

Examples: `%20info@setonkc.org`, `%20contact@thecedars.org`, `%20director@pawsgh.org`

**Fix:** Added `urllib.parse.unquote()` in `_find_all_emails()` after extraction. Reject emails still containing `%` or spaces after decode.

### 2. Run-together domain names (35 emails)

The email regex captured past the real TLD into adjacent text, producing malformed domains.

Examples: `dburch@voa.orgvolunteering`, `info@bloominplace.orgtelephone`, `bergers@nyassembly.govcongresswomanalexandria`

**Fix:** Added `MALFORMED_TLD_PATTERN` regex detecting known base TLDs (org/com/net/edu/gov) fused with trailing text, plus a `VALID_LONG_TLDS` whitelist for legitimate TLDs like `.community`, `.foundation`, `.consulting`.

### 3. Hash-based tracking emails (9 emails)

CMS-generated tracking pixel or session-ID emails with 20+ hex character local parts.

Examples: `222f7380240e4d88919888edfb10ea94@ycoe.org`, `54b18be503644303addb8a004282c3cd@sr.sailamx.com`

**Fix:** Added `HASH_LOCAL_PATTERN` regex rejecting emails where the local part is 20+ hex characters.

### 4. Google Calendar variant missed (21 emails)

The `@group.calendar.google.com` exclusion didn't catch `@group.v.calendar.google.com` (a variant Google Calendar address format).

**Fix:** Changed pattern from `@group\.calendar\.google\.com` to `\.calendar\.google\.com` (catches all variants).

### 5. Missing exclusions and placeholders (8 emails)

- `you@email.com`, `info@domain.com` (placeholder emails)
- `info@latinotype.com` (font foundry)
- `@automattic.com` (WordPress parent company)

**Fix:** Added all to both `EXCLUDE_PATTERNS` and `JUNK_PATTERNS`.

### 6. Misclassified role emails (108 emails)

22 common nonprofit role prefixes were missing from `ROLE_PREFIXES`, causing emails like `foundation@`, `fundraising@`, `alumni@`, `counselor@`, `security@` to be classified as `personal`.

**Fix:** Added: foundation, fundraising, advancement, scholarship, alumni, intake, enrollment, housing, training, facilities, maintenance, transportation, nutrition, wellness, safety, staffing, compliance, reception, pastor, chaplain, counselor, referee.

### 7. Stuck in_progress rows (500 rows)

500 EINs from the last overnight run were left in `in_progress` status (the batch was still running when the scraper was killed).

**Fix:** Reset to NULL. Also added `reset_stuck_in_progress()` to `DatabaseManager` and auto-reset at start of `run-overnight`.

## Automation Readiness

The `run-overnight` command is now fully self-maintaining for nightly cron jobs:

1. **Startup:** Resets any stuck `in_progress` rows from previous crashed runs
2. **Scraping:** Extracts emails with all new filters (script/style stripping, URL decode, malformed TLD rejection, hash rejection, extended exclusions)
3. **Save:** Classifies emails inline during database insert (email_type populated immediately)
4. **Shutdown:** Runs cross-org dedup + MX validation on all new emails before exit
5. **Graceful stop:** Handles SIGINT/SIGTERM, finishes current batch before cleanup

### Recommended cron setup

```bash
# Start at 11 PM, run 8 hours, stop at 7 AM
0 23 * * * cd /path/to/nonprofit_email_scraper && python3 cli.py run-overnight --max-hours 8 --concurrency 15 >> /var/log/scraper_cron.log 2>&1
```

### What still needs MX validation after each run

New emails from each overnight run will have `mx_valid = NULL` until the post-scrape MX validation step runs. This is handled automatically by the `run-overnight` cleanup phase.
