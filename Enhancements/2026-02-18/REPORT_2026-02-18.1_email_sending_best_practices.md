## Executive Summary

The key finding is that the current `send_campaign.py` has a solid foundation but lacks four critical production features: **structured logging**, **resumable checkpointing**, **proper signal handling**, and **capacity-aware domain rotation**. The recommended architecture at the end combines all seven areas into a concrete design that can be implemented incrementally.

## 1. Logging

### What Should Be Logged

Every email send should produce a log record with these fields:

| Field | Purpose | Example |
|-------|---------|---------|
| `timestamp` | When the action occurred | `2026-02-18T14:32:01.123Z` |
| `event` | What happened | `email_sent`, `email_failed`, `rate_limited` |
| `prospect_ein` | Who it was for | `464194065` |
| `to_email` | Recipient address | `jane@nonprofit.org` |
| `from_domain` | Which sending domain | `grants.thegrantscout.com` |
| `template` | Which template | `A` |
| `resend_message_id` | API response ID | `re_abc123` |
| `duration_ms` | How long the API call took | `342` |
| `attempt` | Retry attempt number | `1` |
| `error` | Error message (if any) | `rate_limit_exceeded` |
| `batch_id` | Campaign batch identifier | `v2_2026-02-18_001` |

Additionally, log these aggregate events periodically:

- **Batch start/end** with total counts, elapsed time, and domain breakdown
- **Rate limit hits** with backoff duration
- **Checkpoint saves** with count of records saved
- **Graceful shutdown** with reason and final state

### Structured Logging with structlog

Use `structlog` instead of the standard `logging` module. It produces JSON in production and pretty console output in development, and supports "canonical log lines" (one rich log entry per email instead of many sparse entries).

```python
import structlog
import sys

def configure_logging(log_file: str = None, json_mode: bool = True):
    """Configure structlog for production email sending."""
    processors = [
        structlog.contextvars.merge_contextvars,
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.StackInfoRenderer(),
        structlog.processors.format_exc_info,
    ]

    if json_mode:
        processors.append(structlog.processors.JSONRenderer())
    else:
        processors.append(structlog.dev.ConsoleRenderer())

    structlog.configure(
        processors=processors,
        wrapper_class=structlog.make_filtering_bound_logger(20),  # INFO+
        context_class=dict,
        logger_factory=structlog.PrintLoggerFactory(
            file=open(log_file, "a") if log_file else sys.stderr
        ),
        cache_logger_on_first_use=True,
    )

log = structlog.get_logger()

# Usage in send loop:
log.info("email_sent",
    prospect_ein=ein,
    to_email=to_addr,
    from_domain=subdomain,
    template=template,
    resend_id=msg_id,
    duration_ms=elapsed_ms,
    attempt=1,
)
```

### File-Based vs DB Logging: Use Both

| Layer | Format | Purpose | Retention |
|-------|--------|---------|-----------|
| **Structured log file** | JSON lines (.jsonl) | Debugging, audit trail, grep-able | 90 days, rotate at 10MB |
| **CSV send log** | CSV | Human-readable quick review, spreadsheet import | Per-campaign file |
| **Database** | `campaign_prospect_status` table | Deduplication, reporting, cross-campaign tracking | Permanent |

The JSON log file is the primary debugging tool. The CSV is a convenience export. The database is the system of record for "was this email sent?"

**Recommendation:** Keep the current CSV send log for human review. Add a `.jsonl` structured log file for debugging. The database remains the idempotency source of truth (already implemented via `campaign_prospect_status`).

## 2. Stop/Start (Checkpointing)

### The Core Problem

When sending thousands of emails, the script must survive:
- `Ctrl+C` (user interrupt)
- Machine crash / power loss
- API outage mid-batch
- Network timeout

And resume **exactly where it left off** without sending duplicates.

### Tracking: By Email Address + Campaign Vertical (not row ID)

The deduplication key should be `(email, campaign_vertical)` because:
- Row IDs change if you re-pull prospects
- Email addresses are the real-world unique identifier
- A prospect should only get one email per campaign vertical

The current `send_campaign.py` already uses `ON CONFLICT (email, vertical)` in the DB. This is the correct approach.

### Checkpoint Architecture: DB-Primary with CSV Backup

```
                     +------------------+
                     |   prospect list  |
                     | (GENERATE_CSV)   |
                     +--------+---------+
                              |
                    for each prospect:
                              |
                     +--------v---------+
          +--------->| already in DB?   |
          |   YES    | (sent or error)  |
          |          +--------+---------+
          |                   | NO
          |          +--------v---------+
          |          |   send email     |
          |          +--------+---------+
          |                   |
          |          +--------v---------+
          |          | write to DB      |<--- atomic: send + record
          |          | (commit)         |     in same try/except
          |          +--------+---------+
          |                   |
          |          +--------v---------+
          +----------| append to CSV    |<--- backup/convenience
                     | (flush)          |
                     +------------------+
```

### Implementation Pattern

```python
class SendCheckpointer:
    """Track sent emails for crash-safe resume."""

    def __init__(self, conn, campaign_vertical: str, csv_path: Path):
        self.conn = conn
        self.campaign_vertical = campaign_vertical
        self.csv_path = csv_path
        self._sent_emails: set = set()
        self._load_from_db()

    def _load_from_db(self):
        """Load already-sent emails from database on startup."""
        cur = self.conn.cursor()
        cur.execute("""
            SELECT email FROM campaign_prospect_status
            WHERE vertical = %s
              AND initial_status = 'sent'
              AND email IS NOT NULL
        """, (self.campaign_vertical,))
        self._sent_emails = {row[0] for row in cur.fetchall()}
        cur.close()

    def already_sent(self, email: str) -> bool:
        """Check if this email was already sent in this campaign."""
        return email.lower() in self._sent_emails

    def record_sent(self, email: str, ein: str, metadata: dict):
        """Record a successful send. DB is source of truth."""
        email_lower = email.lower()

        # 1. Write to DB (source of truth)
        cur = self.conn.cursor()
        cur.execute("""
            INSERT INTO campaign_prospect_status
                (ein, email, org_name, org_type, vertical,
                 campaign_status, initial_sent_at, initial_status,
                 initial_subject, initial_sender, source_file)
            VALUES (%s, %s, %s, 'nonprofit', %s,
                    'initial_sent', NOW(), 'sent',
                    %s, %s, 'send_campaign.py')
            ON CONFLICT (email, vertical) WHERE email IS NOT NULL
            DO NOTHING
        """, (ein, email_lower, metadata.get('org_name', ''),
              self.campaign_vertical,
              metadata.get('subject', ''),
              metadata.get('from_addr', '')))
        self.conn.commit()

        # 2. Update in-memory set
        self._sent_emails.add(email_lower)

        # 3. Append to CSV (backup, non-blocking)
        self._append_csv(email_lower, metadata)

    def _append_csv(self, email: str, metadata: dict):
        """Append one row to CSV send log. File is opened in append mode."""
        write_header = not self.csv_path.exists()
        with open(self.csv_path, "a", newline="") as f:
            writer = csv.DictWriter(f, fieldnames=SEND_LOG_FIELDS)
            if write_header:
                writer.writeheader()
            writer.writerow({**metadata, "contact_email": email})

    @property
    def sent_count(self) -> int:
        return len(self._sent_emails)
```

### Key Design Decisions

1. **DB is source of truth.** On restart, load the set of already-sent emails from the DB, not from the CSV. The CSV may be incomplete (crash during write), but the DB has transactional guarantees.

2. **In-memory set for speed.** Loading sent emails into a Python `set` means each "already sent?" check is O(1). For 10K emails, the set is ~1MB of memory.

3. **CSV is append-only.** Open the CSV in append mode and write one row after each send. Never rewrite the full CSV during sending (the current approach of `write_csv_file` at the end is vulnerable to data loss on crash).

4. **`DO NOTHING` on conflict.** If the DB already has a record for this email+vertical, skip silently. This makes the insert idempotent.

## 3. Idempotency

### Definition

An email send is **idempotent** if running the script twice with the same input produces the same outcome: each recipient gets exactly one email.

### Three Layers of Idempotency Protection

```
Layer 1: Pre-send check (fast, in-memory)
    "Is this email in my sent_emails set?" --> skip

Layer 2: DB constraint (durable, authoritative)
    "Does a row exist for (email, vertical)?" --> ON CONFLICT DO NOTHING

Layer 3: Resend's built-in dedup (API-level)
    Not available -- Resend does NOT provide client-side idempotency keys.
    If you call the API twice with the same payload, it sends two emails.
```

Since Resend does not support idempotency keys, **Layers 1 and 2 are critical**. The script must never call the Resend API for an email that has already been sent.

### Implementation Pattern

```python
def send_one_email(prospect: dict, checkpointer: SendCheckpointer,
                   resend_client, from_addr: str) -> dict:
    """Send a single email with full idempotency protection."""
    email = prospect["contact_email"].lower()

    # Layer 1: In-memory check (fast path)
    if checkpointer.already_sent(email):
        return {"status": "skipped_duplicate", "email": email}

    # Layer 2: Defensive DB check (belt + suspenders)
    # This is handled by the ON CONFLICT in record_sent()

    # Send via Resend API
    try:
        response = resend_client.Emails.send({
            "from": from_addr,
            "to": [email],
            "subject": prospect["subject_line"],
            "text": prospect["email_body"],
            "reply_to": "alec@thegrantscout.com",
        })
    except Exception as e:
        return {"status": "error", "email": email, "error": str(e)}

    # Extract message ID
    msg_id = (response.id if hasattr(response, 'id')
              else response.get('id', '') if isinstance(response, dict)
              else str(response))

    # Record in checkpoint (DB + CSV + memory)
    checkpointer.record_sent(email, prospect["prospect_ein"], {
        "org_name": prospect.get("organization_name", ""),
        "subject": prospect["subject_line"],
        "from_addr": from_addr,
        "resend_id": msg_id,
        "template": prospect["template_used"],
        "timestamp": datetime.now(timezone.utc).isoformat(),
    })

    return {"status": "sent", "email": email, "resend_id": msg_id}
```

### Cross-Run Safety

The `pull` command already excludes previously-contacted emails:

```sql
AND np.contact_email NOT IN (
    SELECT DISTINCT email FROM campaign_prospect_status
    WHERE email IS NOT NULL
)
```

This means a fresh `pull` will never include already-sent addresses. Combined with the in-memory check during `send`, this provides two independent guarantees against double-sends.

### Edge Case: Resend API returns success but DB write fails

This is the "dual-write problem." If the Resend API accepts the email but the DB commit fails (e.g., connection dropped), the email is sent but not recorded. On resume, the script will try to send it again.

**Mitigation:** This is unlikely but possible. Accept the risk for now. A future improvement would be to implement a local write-ahead log (WAL): write the intent to a local file before calling the API, then update the file with the result. On restart, replay the WAL.

## 4. Progress Reporting

### Real-Time Console Output with tqdm

`tqdm` provides automatic ETA calculation, throughput rates, and a compact progress bar with minimal overhead (~60ns per iteration).

```python
from tqdm import tqdm
from collections import Counter

def send_batch(sendable: list, checkpointer, resend_client, args):
    """Send emails with real-time progress reporting."""
    domain_counts = Counter()
    template_counts = Counter()
    error_count = 0
    skip_count = 0

    pbar = tqdm(
        sendable,
        desc="Sending emails",
        unit="email",
        ncols=100,
        bar_format=("{l_bar}{bar}| {n_fmt}/{total_fmt} "
                    "[{elapsed}<{remaining}, {rate_fmt}]"
                    " sent={postfix[sent]} err={postfix[err]}"
                    " skip={postfix[skip]}"),
        postfix={"sent": 0, "err": 0, "skip": 0},
    )

    for prospect in pbar:
        result = send_one_email(prospect, checkpointer, resend_client,
                                prospect["from_addr"])

        if result["status"] == "sent":
            domain_counts[prospect["subdomain"]] += 1
            template_counts[prospect["template_used"]] += 1
        elif result["status"] == "error":
            error_count += 1
        elif result["status"] == "skipped_duplicate":
            skip_count += 1

        # Update progress bar postfix
        pbar.set_postfix({
            "sent": checkpointer.sent_count,
            "err": error_count,
            "skip": skip_count,
        }, refresh=False)

        # Pacing delay
        if result["status"] == "sent":
            delay = random.uniform(8, 12)
            time.sleep(delay)

    return {
        "domain_counts": domain_counts,
        "template_counts": template_counts,
        "error_count": error_count,
        "skip_count": skip_count,
    }
```

## 5. Graceful Shutdown

### The Problem

When the user presses Ctrl+C or the process receives SIGTERM, the script must:
1. Finish the current email send (do not interrupt mid-API-call)
2. Record the result
3. Save all state (DB commit, CSV flush, log flush)
4. Print a final summary
5. Exit with a non-zero code so the caller knows it was interrupted

### Signal Handler with Flag Pattern

The flag-based approach is the recommended best practice. Do not raise exceptions in signal handlers (they can corrupt state). Instead, set a flag and check it in the main loop.

```python
import signal
import threading

class GracefulShutdown:
    """Cooperative shutdown manager for email sending."""

    def __init__(self):
        self._shutdown_requested = threading.Event()
        self._original_sigint = signal.getsignal(signal.SIGINT)
        self._original_sigterm = signal.getsignal(signal.SIGTERM)

    def install(self):
        """Install signal handlers. Call before the send loop."""
        signal.signal(signal.SIGINT, self._handler)
        signal.signal(signal.SIGTERM, self._handler)

    def restore(self):
        """Restore original signal handlers."""
        signal.signal(signal.SIGINT, self._original_sigint)
        signal.signal(signal.SIGTERM, self._original_sigterm)

    def _handler(self, signum, frame):
        sig_name = "SIGINT" if signum == signal.SIGINT else "SIGTERM"
        print(f"\n  Received {sig_name}. Finishing current email, then stopping...")
        self._shutdown_requested.set()

        # On second signal, force exit
        signal.signal(signum, self._force_handler)

    def _force_handler(self, signum, frame):
        print("\n  Second signal received. Force exiting.")
        sys.exit(1)

    @property
    def should_stop(self) -> bool:
        return self._shutdown_requested.is_set()
```

### Integration with Send Loop

```python
def cmd_send(args):
    shutdown = GracefulShutdown()
    shutdown.install()

    try:
        for prospect in sendable:
            # Check before each send
            if shutdown.should_stop:
                log.info("graceful_shutdown", reason="signal",
                         sent=sent_count, remaining=total - sent_count)
                break

            result = send_one_email(prospect, checkpointer, resend_client,
                                    prospect["from_addr"])

            # ... handle result ...

            # Check again after send (in case signal arrived during API call)
            if shutdown.should_stop:
                log.info("graceful_shutdown", reason="signal_post_send",
                         sent=sent_count, remaining=total - sent_count)
                break

            time.sleep(delay)

    finally:
        shutdown.restore()
        # Save final state (DB is already committed per-email)
        # Write final CSV summary
        write_final_summary(send_log, domain_counts, error_count)
        log.info("send_complete",
                 total_sent=sent_count,
                 total_errors=error_count,
                 interrupted=shutdown.should_stop)
```

### Key Behaviors

- **First Ctrl+C:** Sets flag, current email finishes, loop exits cleanly, state is saved.
- **Second Ctrl+C:** Forces immediate exit (for emergencies).
- **SIGTERM (from systemd/launchd):** Same behavior as first Ctrl+C.
- **The `finally` block always runs**, ensuring CSV is written and DB connection is closed.

### Comparison with Current Implementation

The current `send_campaign.py` catches `KeyboardInterrupt` at the outer `try/except` level (line 879). This works but has a flaw: if the interrupt arrives during `resend_mod.Emails.send()`, the API call may complete but the result is never recorded. The flag-based approach is safer because the signal handler only sets a flag, and the loop checks the flag at safe points.

## 6. Batch Processing vs Streaming

### Resend-Specific Context

Resend offers two sending modes:

| Mode | API | Limit | Rate Limit |
|------|-----|-------|------------|
| **Single send** | `Emails.send()` | 1 email per call | 2 req/sec across all endpoints |
| **Batch send** | `Emails.batch.send()` | Up to 100 emails per call | 2 req/sec (same pool) |

The batch API counts as **one request** against the 2 req/sec limit, so sending 100 emails via batch uses 1 request vs 100 requests for single sends.

### Recommendation: One-at-a-Time with Delays (Current Approach is Correct)

For cold email campaigns with domain warmup concerns, **one-at-a-time with 8-12 second delays is the right approach**. Here is why:

| Factor | Batch (100 at once) | Streaming (one-at-a-time) |
|--------|---------------------|---------------------------|
| Deliverability | Dangerous: 100 emails from same domain in <1 second | Safe: mimics human sending pattern |
| Checkpoint granularity | All-or-nothing per batch of 100 | Per-email (crash-safe) |
| Error handling | One failure affects entire batch | Isolated per email |
| Rate limit management | Easy to blow through limits | Natural throttling via delays |
| Domain rotation | All 100 go through same API call | Can rotate per email |
| Warmup safety | 100 emails in a burst is 5x daily limit for new domains | Gradual, spread over time |

**The batch API makes sense for transactional email** (e.g., sending 100 password reset emails), but for cold outreach it is counterproductive because receiving ESPs flag burst patterns.

### Delay Strategy

```python
def calculate_delay(base_min: float = 8.0, base_max: float = 12.0,
                    jitter: float = 2.0) -> float:
    """Calculate human-like delay between sends.

    Base: 8-12 seconds (uniform random)
    Jitter: +/- 2 seconds (normal distribution)
    Result: roughly 6-14 seconds, centered around 10
    """
    base = random.uniform(base_min, base_max)
    noise = random.gauss(0, jitter / 3)  # 99.7% within +/- jitter
    return max(3.0, base + noise)  # floor at 3 seconds
```

### When to Use Resend Batch API

Only if you need to send a large number of emails quickly and deliverability is not a concern (e.g., internal notifications, transactional messages). For TheGrantScout cold email campaigns: do not use batch.

## 7. Domain Rotation Strategy

### Current Approach: Simple Round-Robin

The current `send_campaign.py` uses round-robin:

```python
email["subdomain"] = SUBDOMAINS[i % len(SUBDOMAINS)]
```

This distributes emails evenly across 17 subdomains. It is simple and effective for the initial launch, but lacks awareness of domain health.

### Recommended: Capacity-Aware Rotation with Health Pools

Based on research from Instantly and Mailpool, the best practice is to organize domains into three pools based on health metrics:

```
+-------------------+     +-------------------+     +-------------------+
|     PRIMED        |     |     RAMPING       |     |     RESTING       |
|                   |     |                   |     |                   |
| Mature domains    |     | New domains or    |     | Domains with      |
| Health >= 85%     |     | recovering ones   |     | health issues     |
| Full daily cap    |     | Conservative cap  |     | Warmup-only       |
+-------------------+     +-------------------+     +-------------------+
        |                         |                         |
   80% of volume            20% of volume              0% campaign
```

### Health Score Formula

From the Instantly research, a composite health metric:

```
Health % = 0.5 * Placement + 0.3 * (100 - BounceRate) + 0.2 * (100 - SpamRate)
```

Where:
- **Placement** = % of test emails landing in primary inbox (requires external tool like GlockApps)
- **BounceRate** = % of emails bouncing (from Resend webhooks)
- **SpamRate** = % marked as spam (from Resend webhooks)

### Practical Implementation for TheGrantScout

Since you do not yet have placement testing infrastructure, start with a simpler capacity-aware model:

```python
from dataclasses import dataclass, field
from typing import Dict, List
import random

@dataclass
class SendingDomain:
    subdomain: str
    daily_cap: int = 30          # max emails per day
    sent_today: int = 0          # emails sent today
    total_bounces: int = 0       # lifetime bounces
    total_sent: int = 0          # lifetime sends
    pool: str = "ramping"        # primed | ramping | resting
    warmup_day: int = 0          # days since domain creation
    paused_until: str = None     # ISO timestamp if paused

    @property
    def bounce_rate(self) -> float:
        if self.total_sent == 0:
            return 0.0
        return self.total_bounces / self.total_sent

    @property
    def remaining_today(self) -> int:
        return max(0, self.daily_cap - self.sent_today)

    @property
    def is_available(self) -> bool:
        if self.pool == "resting":
            return False
        if self.paused_until:
            return datetime.now(timezone.utc).isoformat() > self.paused_until
        return self.remaining_today > 0


class DomainRotator:
    """Capacity-aware domain rotation for cold email campaigns."""

    def __init__(self, domains: List[SendingDomain]):
        self.domains = {d.subdomain: d for d in domains}

    def next_domain(self) -> SendingDomain | None:
        """Select next available domain using weighted random selection.

        Primed domains get 4x weight, ramping domains get 1x weight.
        Within each pool, domains with more remaining capacity are preferred.
        """
        available = [d for d in self.domains.values() if d.is_available]
        if not available:
            return None

        weights = []
        for d in available:
            pool_weight = 4.0 if d.pool == "primed" else 1.0
            capacity_weight = d.remaining_today / max(d.daily_cap, 1)
            weights.append(pool_weight * capacity_weight)

        return random.choices(available, weights=weights, k=1)[0]

    def record_send(self, subdomain: str):
        """Record a successful send for a domain."""
        d = self.domains[subdomain]
        d.sent_today += 1
        d.total_sent += 1

    def record_bounce(self, subdomain: str):
        """Record a bounce. Pause domain if bounce rate exceeds threshold."""
        d = self.domains[subdomain]
        d.total_bounces += 1
        if d.bounce_rate >= 0.02:  # 2% threshold
            d.pool = "resting"
            d.paused_until = (
                datetime.now(timezone.utc) + timedelta(hours=24)
            ).isoformat()

    def check_health(self):
        """Promote/demote domains based on health metrics."""
        for d in self.domains.values():
            if d.pool == "resting" and d.paused_until:
                if datetime.now(timezone.utc).isoformat() > d.paused_until:
                    d.pool = "ramping"
                    d.daily_cap = max(10, d.daily_cap // 2)
                    d.paused_until = None

            elif d.pool == "ramping" and d.warmup_day >= 14:
                if d.bounce_rate < 0.01 and d.total_sent >= 100:
                    d.pool = "primed"
                    d.daily_cap = 50

    def summary(self) -> dict:
        """Return rotation status summary."""
        return {
            "primed": [d.subdomain for d in self.domains.values()
                       if d.pool == "primed"],
            "ramping": [d.subdomain for d in self.domains.values()
                        if d.pool == "ramping"],
            "resting": [d.subdomain for d in self.domains.values()
                        if d.pool == "resting"],
            "total_remaining": sum(d.remaining_today
                                   for d in self.domains.values()
                                   if d.is_available),
        }
```

### Warmup Ramp Schedule

Based on research from Mailpool and Smartlead:

| Days Since Creation | Daily Cap Per Domain | Total Across 17 Domains |
|---------------------|----------------------|-------------------------|
| 1-3 | 20 | 340 |
| 4-7 | 30 | 510 |

### Send Loop Pseudocode

```python
def cmd_send(args):
    configure_logging(json_mode=not sys.stdout.isatty())
    shutdown = GracefulShutdown()
    shutdown.install()

    conn, cur = get_db_connection()
    checkpointer = SendCheckpointer(conn, CAMPAIGN_VERTICAL, SEND_LOG_CSV)
    rotator = DomainRotator(load_domain_config())

    sendable = read_csv_file(GENERATE_CSV)
    already_sent_count = 0

    log.info("send_start", total=len(sendable),
             already_sent=checkpointer.sent_count)

    pbar = tqdm(sendable, desc="Sending", unit="email")

    for prospect in pbar:
        if shutdown.should_stop:
            break

        email = prospect["contact_email"].lower()

        # Skip already sent
        if checkpointer.already_sent(email):
            already_sent_count += 1
            pbar.set_postfix(skip=already_sent_count)
            continue

        # Get next available domain
        domain = rotator.next_domain()
        if domain is None:
            log.warning("all_domains_exhausted")
            break

        # Send
        result = send_one_email(prospect, checkpointer, resend_client,
                                f"Alec Kleinman <alec@{domain.subdomain}.thegrantscout.com>")

        if result["status"] == "sent":
            rotator.record_send(domain.subdomain)

        # Delay
        if result["status"] == "sent":
            time.sleep(calculate_delay())

    shutdown.restore()
    write_final_summary(...)
```

## Notes

- The Resend API has a hard limit of 2 requests/second across all endpoints. The current 8-12 second delay between sends is well within this limit.
- Resend's batch API (up to 100 emails per call) should NOT be used for cold outreach due to deliverability concerns.
- Resend does not support client-side idempotency keys, making application-level deduplication essential.
- The listmonk open-source project (PostgreSQL-based) provides a good reference schema for campaign tracking, with campaign status enums (draft, running, scheduled, paused, cancelled, finished) and bounce tracking (soft, hard, complaint).
- Domain health monitoring requires Resend webhook integration to track bounces and complaints per domain.
