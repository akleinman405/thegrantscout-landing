# TheGrantScout Email Discovery System

How We Find Websites, Extract Emails, and Build Contact Lists  
for 242,000+ Nonprofits and Foundations

Technical Documentation \| February 2026 \| Version 1.0

# Executive Summary

TheGrantScout maintains a database of 813,698 organizations (143,184 foundations and 670,514 nonprofits) sourced from IRS 990 tax filings. To turn these records into actionable sales contacts, we built a 7-script automated pipeline that discovers website URLs, scrapes those websites for email addresses and staff information, validates the emails, and surfaces the single best contact email per organization.

## Key Numbers

| **Metric** | **Value** |
|----|----|
| Total organizations in database | 813,698 |
| Organizations with IRS-filed websites | 377,175 (46.4%) |
| Websites confirmed alive via HTTP check | 242,564 (64% of those with URLs) |
| Email extraction methods | 5 (mailto, Cloudflare decode, JSON-LD, AT/DOT, regex) |
| Email validation | Syntax check + MX DNS record verification |
| Pipeline scripts | 7 Python + 1 shell orchestrator |
| Database tables | 6 (+ 1 materialized view) |
| Overnight schedule | Weekdays 10pm-9am, weekends all day |
| Estimated full run | 8-10 overnight sessions for all 242K orgs |

## Pipeline at a Glance

The system works in two phases, each with multiple steps:

**Phase A: Find Websites (Scripts 01-03)**

- Clean & normalize raw URLs from IRS filings (remove junk, standardize format)

- Validate each URL with an HTTP request (is the site alive or dead?)

- Search DuckDuckGo for orgs that have no URL on file

**Phase B: Scrape Websites for Emails (Scripts 04-07)**

- Fetch homepage + up to 4 internal pages per org (contact, about, team, etc.)

- Extract all emails using 5 different detection methods

- Validate email syntax and verify the domain accepts mail (MX records)

- Pick the single best email per org and optionally push to campaign tables

# Phase A: Finding Websites

Before we can scrape for emails, we need to know each organization's website. IRS 990 forms include an optional website field, but the data is messy: about 53% of entries are junk (phone numbers, "N/A", "NONE", aggregator sites like GuideStar). Three scripts work together to clean, validate, and fill in missing URLs.

## Step 1: Clean & Normalize URLs

**Script:** 01_clean_org_urls.py

**Why this step exists:** IRS filings are filled out by humans, so the website field is wildly inconsistent. Some orgs enter phone numbers, some write "N/A", some put GuideStar links instead of their own website. We need to separate real URLs from garbage before doing anything else.

### What it does

- Reads all 813,698 rows from org_url_enrichment where url_source is NULL (not yet processed)

- **Junk detection:** Checks each website_raw value against 63 exact junk matches

- **Examples:** "N/A", "NONE", "0", "www.", "coming soon", "see schedule o", phone number patterns, etc.

- **Regex junk:** Plus 12 regex patterns for attorney general references, PO boxes, very short codes

- **Aggregator filter:** Filters out 24 aggregator domains (GuideStar, Charity Navigator, ProPublica, IRS.gov, etc.)

- **Social media:** Extracts Facebook and LinkedIn URLs to separate columns

- **Normalization:** Normalizes valid URLs: adds https://, lowercases domain, strips tracking params and fragments

### Results

| **Category**                     | **Count** | **Percentage** |
|----------------------------------|-----------|----------------|
| Real URLs (cleaned & normalized) | 377,175   | 46.4%          |
| Junk removed                     | 434,863   | 53.4%          |
| Social media extracted           | 1,559     | 0.2%           |

*The url_source column tracks provenance: "irs_filing" for real URLs, "junk_removed" for garbage, "social_extracted" for Facebook/LinkedIn-only entries.*

## Step 2: Validate URLs with HTTP Requests

**Script:** 02_validate_org_urls.py

**Why this step exists:** Just because an IRS filing lists a URL doesn't mean the website still exists. Organizations shut down, domains expire, websites move. We need to check which sites are actually alive before spending time scraping them.

### What it does

- Makes async HTTP HEAD requests to all 377K cleaned URLs (50 concurrent connections)

- If HEAD fails, falls back to a GET request (some servers block HEAD)

- Classifies responses into alive or dead:

<!-- -->

- **Alive:** HTTP 200-399 (success + redirects)

- **Bot-blocked (still alive):** HTTP 403, 405, 429, 503 (the site exists but blocked our request)

- **Dead:** DNS failure, connection refused, HTTP 404, timeouts

<!-- -->

- Captures redirect targets (the final URL after all redirects)

- Checkpoint/resume via state file (can stop and restart without redoing work)

### Results

| **Category**              | **Count** | **Percentage** |
|---------------------------|-----------|----------------|
| Alive (confirmed working) | 242,564   | 64.3%          |
| Dead (site gone)          | 55,936    | 14.8%          |
| Pending (not yet checked) | 78,701    | 20.9%          |

### Key Configuration

| **Setting** | **Value** | **Why** |
|----|----|----|
| Concurrent connections | 50 | Balance speed vs. not overwhelming servers |
| Timeout per request | 15 seconds | Catch dead sites quickly without waiting forever |
| Max redirects | 5 | Follow reasonable redirect chains but prevent infinite loops |
| SSL verification | Disabled | Many nonprofit sites have expired/self-signed certificates |

## Step 3: Search for Missing URLs

**Script:** 03_search_org_urls.py

**Why this step exists:** Over half of organizations either had no URL on their IRS filing or had junk values. Many of these orgs DO have websites -- they just didn't list them. We search DuckDuckGo to find them.

### What it does

- Searches DuckDuckGo for each org that has no URL (using org name + city + state)

- 3 parallel worker threads with rate limiting (2-5 second delays between requests)

- Iterative backoff when rate-limited (all workers pause together)

- Scores each search result with sophisticated domain-name matching:

<!-- -->

- **Full name match:** Domain contains the full org name (e.g., "macfound.org" for "MacArthur Foundation")

- **Multi-word match:** 2+ unique words from the org name appear in the domain

- **Acronym match:** First letters of org name match domain (e.g., "ACLU" -\> aclu.org)

- **Snippet verification:** Org name appears in the search result title or description

<!-- -->

- Filters out 120+ aggregator/directory domains (same as Step 1, plus job boards, news sites, etc.)

- Filters out .gov and .edu domains

- Ignores common stop words, surnames, and first names when matching (e.g., "Miller" is too ambiguous)

### Confidence Levels

Only "high" confidence matches are saved. Medium confidence was disabled after testing showed 0% accuracy (single-word domain matches are too ambiguous for nonprofit names).

### Results (test run: 100 foundations)

| **Metric**          | **Value**         |
|---------------------|-------------------|
| URLs found          | 25 (25% hit rate) |
| All high confidence | Yes (100%)        |
| False positives     | 0                 |

# Phase B: Scraping Websites for Emails

Once we have validated website URLs, we scrape them to extract email addresses, staff names and titles, and organization metadata. This is the core of the system -- turning a URL into an actionable contact. Four scripts handle this in sequence.

## Step 4: Fetch Website Pages

**Script:** 04_fetch_website_pages.py

**Why this step exists:** Emails aren't always on the homepage. Contact pages, about pages, team pages, and leadership pages are where organizations typically list their contact information. We fetch multiple pages per site to maximize our chance of finding emails.

### What it does

- Fetches the homepage for each org, then discovers and fetches up to 4 internal pages

- 25 concurrent async HTTP connections via httpx (HTTP/2 support)

- Per-domain rate limiting: max 1 request per second to any single domain

- Checks robots.txt before accessing each domain (polite scraping)

- Stores HTML on disk as gzip-compressed JSON files (~5:1 compression ratio)

### Pages Discovered Per Org

The script looks for links on the homepage that match these patterns:

| **Page Type** | **URL Patterns Searched** | **Link Text Patterns** |
|----|----|----|
| Contact | /contact, /contact-us, /get-in-touch | "Contact", "Reach Us", "Email Us" |
| About | /about, /about-us, /who-we-are, /mission | "About", "Our Story", "Our Mission" |
| Team | /team, /staff, /our-team, /people, /directory | "Team", "Staff", "Our People" |
| Leadership | /leadership, /board, /board-of-directors | "Board", "Leadership", "Officers" |
| Grants | /grants, /apply, /rfp, /funding | "Grants", "How to Apply", "Funding" |

### Overnight Scheduling

**Why overnight?** We scrape during off-hours to be considerate to the organizations whose sites we're visiting. Traffic during business hours could affect their site performance. The script has a built-in time window that automatically stops fetching outside the allowed window.

| **Day**       | **Scraping Window**            |
|---------------|--------------------------------|
| Monday-Friday | 10:00 PM to 9:00 AM (11 hours) |

| **Tier** | **Description** | **Count** | **Est. Runtime** |

| Tier 3 | Nonprofits | ~100,000+ | ~3-4 nights |

*Why foundations first? Foundations are our potential customers' funders. Having their contact info is directly valuable for grant prospecting. Tier 1 (open to applications) are the most actionable.*

The script tracks progress by updating scrape_status in the database. If interrupted (power failure, manual stop, end of time window), it picks up exactly where it left off the next night. Commits happen every 100 orgs. A graceful shutdown handler ensures all in-progress work is saved on Ctrl+C.

## Step 5: Extract Emails, Staff, and Metadata

**Script:** 05_extract_website_data.py (~660 lines)

**Why this step exists:** Raw HTML is useless on its own. This script parses every page to find email addresses (using 5 different detection methods because websites hide emails in different ways), identifies staff members, and captures organization metadata like social media links and mission statements.

### Email Extraction: 5 Methods

Websites display email addresses in many ways. Some are plain text, some are hidden behind obfuscation to prevent spam bots. We use 5 extraction methods, listed from most to least reliable:

**Method 1: Mailto Links (Highest Confidence)**

When a website has \<a href="mailto:info@example.org"\>, the organization intentionally published that email as a contact point. These are the most reliable because someone deliberately coded them into the page.

**Method 2: Cloudflare Email Decoding**

Many websites use Cloudflare's email protection, which encodes emails in the HTML as "data-cfemail" attributes using an XOR cipher. The email is invisible to simple scrapers but we decode it by applying the XOR key (stored in the first byte of the encoded string) to each subsequent byte pair.

**Method 3: JSON-LD Structured Data**

Some websites embed machine-readable structured data in \<script type="application/ld+json"\> tags. This is a web standard (schema.org) that search engines use. We parse these blocks looking for "Organization", "NGO", or "NonprofitOrganization" types and extract their "email" and "contactPoint" fields.

**Method 4: AT/DOT Obfuscation**

Some organizations write their email as "info \[at\] example \[dot\] org" to prevent spam bots. We detect these patterns and convert them back to real email addresses. We require bracket delimiters around "at" and "dot" to avoid false positives from regular English words like "foundation" (which contains "at" and could generate fake emails).

**Method 5: Regex Scan (Broadest, Lowest Confidence)**

A standard email regex scans all page text for anything that looks like an email address. This catches emails that weren't in links or structured data. It has the highest recall but also the most noise, which is why it's scored with lower confidence.

### Email Filtering

Emails matching any of these patterns are automatically discarded:

- noreply@, no-reply@, donotreply@, unsubscribe@

- careers@, jobs@, hr@ (job listings, not contacts)

- webmaster@, postmaster@, mailer-daemon@, bounce@

- Platform emails: @wix.com, @squarespace.com, @wordpress.com, @godaddy.com, @mailchimp.com

- Grant platform emails: @submittable.com, @fluxx.io, @instrumentl.com

- Test/placeholder: user@domain, email@example, test@test

- Image file extensions: .png, .jpg, .gif, .svg

### Email Classification

Every extracted email is classified into one of three types:

| **Type** | **Examples** | **What It Means** |
|----|----|----|
| General | info@, contact@, hello@, admin@, office@ | Generic organizational inbox -- good for initial contact |
| Role | grants@, development@, fundraising@, programs@, media@ | Department-specific -- great for targeted outreach |
| Person | jane.doe@, jsmith@ | Individual person -- highest value for direct contact |

### Confidence Scoring

Each email gets a confidence score from 0.00 to 1.00 based on multiple factors. This helps us rank emails and pick the best one per org.

| **Factor** | **Score** | **Why** |
|----|----|----|
| Email domain matches org website | +0.40 | info@macfound.org on macfound.org = clearly their email |
| Found via mailto: link | +0.10 | Intentionally published by the org |
| Found via JSON-LD structured data | +0.10 | Intentionally structured by the org |
| Found on contact or about page | +0.15 | Contact pages are where orgs put their real emails |
| Person-type email (first.last@) | +0.10 | Individual contacts are more valuable |
| General prefix + matching domain | +0.15 | info@ on the org's own domain is reliable |
| Domain does NOT match website | -0.30 | Email on a different domain is suspicious |

**Maximum possible score:** 0.80 (a general@ email, matching domain, from a mailto link on a contact page)

**Example:** media@hewlett.org found via mailto on hewlett.org's contact page scores 0.65 (domain match + mailto + contact page)

### Staff Extraction

On team, leadership, board, and about pages, the script looks for staff member cards or lists. It uses two strategies:

- **Strategy 1:** CSS selector matching: Looks for divs/list items with class names containing "team", "staff", "member", "person", "bio", "profile"

- **Strategy 2:** Heading patterns: Scans \<h2\>, \<h3\>, \<h4\>, \<strong\>, \<b\> tags for capitalized names, then looks for job titles in adjacent elements

Staff are classified by type:

| **Staff Type** | **Detection** | **Examples** |
|----|----|----|
| Executive | Title contains CEO, President, Executive Director, COO, CFO | Jane Doe, Executive Director |
| Board | Page type is "board" or title contains Board Member, Trustee, Chair | John Smith, Board Chair |
| Leadership | Title contains Director, VP, Vice President | Sarah Lee, Program Director |
| Advisory | Title contains Advisory, Advisor, Emeritus | Dr. James, Advisory Board |
| Staff | Everything else | Mike R., Program Associate |

### Metadata Extraction

In addition to emails and staff, the script extracts:

| **Data** | **Source** | **Use** |
|----|----|----|
| Mission text | Paragraphs containing "our mission", "dedicated to", etc. | Understanding what the org does |
| Meta description | \<meta name="description"\> tag | Short org summary |
| Social media URLs | Links to Twitter, LinkedIn, Facebook, Instagram, YouTube | Additional contact channels |
| Phone numbers | Regex + \<a href="tel:"\> links | Phone outreach |
| Physical address | \<address\> tags + JSON-LD PostalAddress | Location verification |
| CMS platform | WordPress, Squarespace, Wix, Drupal markers | Technical profile |
| Cloudflare detection | data-cfemail attributes, "cloudflare" in HTML | May affect scraping approach |
| JS rendering needed | React (#root), Next.js (\_\_NEXT_DATA\_\_), Vue (#app) | Future: sites needing browser rendering |

## Step 6: Validate Emails

**Script:** 06_validate_emails.py

**Why this step exists:** Just because text looks like an email doesn't mean it's real. We need to verify two things: (1) is the syntax valid per email standards, and (2) does the domain actually accept mail? Sending to invalid addresses hurts our sender reputation and wastes campaign resources.

### What it does

**Check 1 - Syntax Validation:** Uses the email-validator Python library to check that the email address follows RFC 5321 standards (valid characters, proper format, not too long).

**Check 2 - MX Record Validation:** Does a DNS lookup to check if the email's domain has MX (Mail Exchanger) records. MX records tell the internet which servers accept mail for that domain. If a domain has no MX records and no A records, mail can't be delivered there.

### Performance Optimizations

- 50 concurrent DNS lookups (async)

- Results cached per domain (most emails share a few common domains)

- Emails grouped by domain before processing to maximize cache hits

- Batch commits every 500 emails

### Smoke Test Results (5 foundations)

| **Check**       | **Result**  |
|-----------------|-------------|
| Syntax valid    | 8/8 (100%)  |
| MX valid        | 8/8 (100%)  |
| Processing time | 0.3 seconds |

## Step 7: Pick the Best Email Per Org

**Script:** 07_materialize_best_email.py

**Why this step exists:** An org might have 5+ emails on its website. For campaign purposes, we need to pick the single best one. This script refreshes a materialized view that ranks all emails and selects the winner.

### Ranking Logic

Emails are ranked by these criteria, in order:

**1.** Domain matches website (emails on the org's own domain always win)

**2.** MX records valid (confirmed deliverable)

**3.** Email type: Person \> Role \> General (individual contacts preferred)

**4.** Confidence score (highest wins)

*Only emails with syntax_valid = TRUE and domain_matches_website = TRUE are considered. This eliminates suspicious emails from third-party domains.*

### Campaign Backfill

With the --backfill flag, the script copies the best email into the campaign tables (grantscout_campaign.nonprofit_prospects and foundation_prospects). This bridges the enrichment pipeline with the email sending system.

# Automation & Scheduling

The entire Phase B pipeline (Steps 4-7) runs automatically overnight using macOS's built-in launchd scheduler. A shell script orchestrates the four Python scripts in sequence.

## Shell Orchestrator: run_scraper.sh

- Runs scripts in order: 04 (fetch) -\> 05 (extract) -\> 06 (validate) -\> 07 (materialize)

- Uses caffeinate to prevent macOS from sleeping during execution

- Logs all output to timestamped log files

- Records exit codes for each stage

- All stages run regardless of whether previous stages fail
