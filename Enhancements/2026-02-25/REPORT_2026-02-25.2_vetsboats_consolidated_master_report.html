Part I: Executive Summary

VetsBoats (EIN 464194065) is TheGrantScout’s most complex client engagement. A private foundation that files 990-PF returns, VetsBoats broke the standard pipeline at every stage and required manual replication of every automated step. Over 8 sessions spanning 80 days (December 4, 2025 through February 21, 2026), the engagement produced 75+ files and 4 client deliverables, uncovered 26 post-delivery errors, and ultimately found that 6 of 10 recommended foundations should have been removed.

The central lesson: grant history patterns alone cannot predict eligibility. The LASSO V6.1 model correctly identifies foundations that fund veterans, sailing, and adaptive recreation. It does not check whether those foundations accept applications from private foundations, whether their geographic restrictions exclude California, or whether their contact information actually works.

Part V: Thematic Deep Dives

A. The Data Pipeline

Why the Standard Pipeline Failed

The automated pipeline (4. Pipeline/process_client.py) requires three things VetsBoats did not have:

  1. Embedding tables (archived January 4, 2026 to free 52GB). Without embeddings, the pipeline cannot compute semantic similarity between VetsBoats’ mission and grant purposes.

  2. A nonprofit_returns record. VetsBoats files 990-PF, not 990 or 990-EZ. The client loading step expects nonprofit_returns.

  3. A completed client questionnaire. VetsBoats had not filled one out. The profile had to be reconstructed from a sales call transcript, a pre-call brief, and online verification.

Additionally, the LASSO V6.1 scoring model has no feature for “PF-to-PF eligibility.” This constraint required manual judgment at every enrichment and selection step.

Manual Replication

Every pipeline phase was replicated manually:

Pipeline Step Automated Approach Manual Approach Used
01: Load client Read questionnaire Reconstruct from sales call + ProPublica + GuideStar
02: Score foundations LASSO model with embeddings Sibling grant analysis + keyword matching + semantic scoring
03: Check enrichment Read cache No cache existed; all fresh research
04: Filter viable Automated viability rules Manual DB pre-screening + web research
05: Assemble opps Structured JSON Manual profile assembly from multiple sources
06: Generate narratives AI-generated from structured data Manual writing with DB-verified statistics
07-08: Build + render Template merge Manual markdown writing + formatting

DB-First Pre-Screening

The approach of eliminating candidates via database queries before expensive web research proved highly efficient. The 8 foundations eliminated in DB pre-screening would each have required 30-60 minutes of web research. Total time saved: approximately 4-8 hours.

Key pre-screening queries: - Check only_contri_to_preselected_ind for latest filing - Check application_submission_info and app_restrictions in pf_returns for explicit exclusion language - Check geographic concentration via fact_grants recipient_state distribution - Check for PF-to-PF grant history by joining recipient_ein to pf_returns

Root Cause: Three Missing Checks

The pipeline correctly identifies foundations that fund veterans, sailing, and adaptive recreation. It does NOT check for:

  1. Private foundation exclusion language. 4 of 6 removed foundations explicitly exclude PFs (Rees-Jones, Autzen, Heinz, Barker). Bill Simpson requires “verification of public charities status.”
  2. Geographic restriction specificity. 5 of 6 removed foundations are regional-only (Rees-Jones: North TX; Autzen: Oregon/Clark County WA; Heinz: southwestern PA; Barker: NYC/Michigan City; Van Beuren: Aquidneck Island).
  3. Contact information currency. Bill Simpson’s website email is a dead mailbox. The application form references the 2025 cycle, not 2026.

The fundamental problem: 990-PF data shows who a foundation has funded, not who a foundation will accept applications from. Grant history is real. Eligibility restrictions are also real. The pipeline only sees the first.

E. The Hard-Filter Discovery

The 340/4,553 Funnel

On February 20, three hard filters were applied to a pool of 4,553 candidate foundations:

Stage Count % of Pool
Total candidate pool (siblings + veteran keyword search) 4,553 100%

Passed ALL 3 filters (open + CA + PF-to-PF grants) 340 7.5%

Filter 3 (PF-to-PF grants) was the most restrictive due to 56.1% recipient EIN coverage in fact_grants. Foundations that fund PFs but whose grant records lack recipient EINs produce false negatives.

Website Verification: 14/15 NO

The hard-filter output was promising on paper but devastating in practice. When foundations were actually crawled:

  • Batch 1 (18 foundations): Only 3 viable after deep crawl (665 pages analyzed)
  • Batch 2 (20 foundations): 3 POSSIBLE_FIT, 4 WEAK_FIT, 13 NOT_A_FIT
  • Batch 3 (9 foundations): 1 STRONG_FIT (Louie Family), 2 POSSIBLE_FIT

Combined across all batches: 2 STRONG, 7 POSSIBLE, 8 WEAK, 32 NOT_A_FIT.

The PF-to-PF False Positive Problem

Many foundations technically pass the PF-to-PF filter because they have incidental grants to organizations that happen to also be private foundations. These are often: - Grants to community foundations or DAFs (technically PFs) - Grants to corporate foundations (technically PFs) - Grants where the recipient EIN maps to a PF but the grant was for a specific program

Having a PF-to-PF grant in the database does not mean the foundation is open to cold applications from external private foundations. The grants reflect existing relationships, not application policies.

Part VII: Lessons Learned and Action Items

Pipeline Architecture (P0 – Must Build)

L-01: PF Scoring Feature

What happened: The LASSO V6.1 model has no feature for PF-to-PF eligibility. VetsBoats was scored against 144 foundations using the same model designed for public charities.

Why it matters: For PF clients, the most important signal is whether a foundation has ever funded another private foundation. Without this feature, the model ranks foundations by sector/geography overlap regardless of PF eligibility.

What to build: Add pf_grant_count and pf_grant_pct features to calc_foundation_features. When the client is a PF, multiply the LASSO score by a PF eligibility modifier.

Priority: P0 | Effort: Medium (feature engineering + model retrain) | Source: Part V-A

L-02: Website Verification as Required Step

What happened: 6 of 10 recommended foundations had disqualifying information on their websites that was not in the 990-PF data. The pipeline’s scoring was accurate (these foundations do fund the right sectors) but the eligibility check was missing.

Why it matters: Every foundation recommendation that gets removed after website review represents wasted client time and eroded trust.

What to build: Add a website verification step between scoring (step 02) and enrichment (step 03). At minimum, check: (1) PF exclusion language, (2) geographic restriction specificity, (3) application status (open/closed/invite-only). Could use foundation scraper skill with structured eligibility prompts.

Priority: P0 | Effort: High (web scraping + classification) | Source: Parts V-D, V-E

L-03: Behavioral PF-to-PF Filter

What happened: The hard-filter used EXISTS (SELECT 1 FROM pf_returns pr WHERE pr.ein = fg.recipient_ein) to detect PF-to-PF grants. Many “matches” were incidental (grants to community foundations, DAFs, or corporate foundations that happen to be PFs).

Why it matters: Having a PF-to-PF grant in the database does not mean the foundation welcomes cold applications from external PFs. The grants reflect existing relationships, not open policies.

What to build: Weight PF-to-PF grants by recency, count, and whether the PF recipient is a “real” external PF (not a DAF, community foundation, or corporate foundation). Add an exclusion list for false-positive PF types.

Priority: P0 | Effort: Medium | Source: Part V-E

L-04: “Open to Applicants” Does Not Mean “Accepting Cold Applications”

What happened: Van Beuren passed the only_contri_to_preselected_ind = FALSE filter and even had an open application portal, but their geographic restriction (Aquidneck Island) made a VetsBoats application futile.

Why it matters: The IRS “preselected only” flag is a binary indicator of grantmaking policy. It does not capture geographic restrictions, sector limitations, or implicit barriers. A foundation can be “open to applicants” and still be unreachable.

What to build: Replace the binary open/closed filter with a multi-dimensional eligibility score: open + geographic fit + sector fit + no PF exclusion + active application process.

Priority: P0 | Effort: High | Source: Parts V-D, V-E

Data Quality (P1 – Should Build)

L-05: Cache Enrichment in DB

What happened: The 17 foundations researched during the February 6 run were not stored in a database table. When subsequent sessions needed the same data, it had to be re-researched.

Why it matters: Web research is expensive (30-60 minutes per foundation). Caching results in foundation_enrichment prevents redundant work across reports and sessions.

What to build: After every enrichment run, store verdicts (GO/CONDITIONAL/NO-GO), key findings, and source URLs in foundation_enrichment. The next report for the same client (or similar clients) checks this cache first.

Priority: P1 | Effort: Low | Source: Part V-A

L-06: SQL-Verify Every Cited Stat

What happened: AI-generated statistics diverged from database values across all 5 foundation profiles. 5 trends wrong, 4 repeat rates inflated, 2 amounts fabricated.

Why it matters: Every inaccurate number erodes client trust and TheGrantScout’s credibility. The errors were not random; they were systematically optimistic (higher amounts, steeper trends).

What to build: A verification pass that extracts every number from the report markdown and queries the database to confirm. Flag any value that differs from DB by more than 5%. This could be automated as 09_audit_report.py.

Priority: P1 | Effort: Medium | Source: Part V-C

L-07: Cross-Check 990-PF Contacts Against Websites

What happened: 5 contact errors across 3 foundations. Chairman was Vice Chair, Grants Manager was Operations Manager, email addresses were wrong.

Why it matters: Sending an email to “Dear Chairman Van Beuren” when the Chair is actually Barbara Van Beuren is embarrassing. Incorrect contact details waste client time.

What to build: For every contact included in a report, verify name + title against the foundation’s website. If the website contradicts 990-PF data, use the website (more current).

Priority: P1 | Effort: Low | Source: Part V-D

L-08: EIN Validation Step

What happened: 4 of 5 EINs in the February 21 supplemental report were wrong. The web research dossier provided EINs from Instrumentl/ProPublica that did not match the database.

Why it matters: Wrong EINs mean wrong data. The funder snapshot script pulled metrics for the wrong foundations until the EINs were corrected.

What to build: Before any pipeline step, verify every input EIN exists in dim_foundations or pf_returns. If not found, search by name ILIKE and present matches for human confirmation. Never proceed with an unverified EIN.

Priority: P1 | Effort: Low | Source: Part V-F

Report Generation (P1 – Should Build)

L-09: AI Stat Divergence Fix

What happened: The report generation process wrote statistics from contextual inference. All 5 giving trends, 4 of 5 repeat rates, and 2 grant amounts were wrong.

Why it matters: This is the single most frequent error category (E-02, 14 instances).

What to build: The report template should populate statistics directly from calc_foundation_profiles and fact_grants queries, not from AI inference. Use merge fields: {{foundation.giving_trend}}, {{foundation.repeat_rate}}, etc. The narrative can be AI-generated; the numbers must be DB-sourced.

Priority: P1 | Effort: Medium | Source: Part V-C

L-10: Automated Connection Paths via Officers Table

What happened: The Matt Goettelman to Ron Levin to Kovler connection was found manually through web research. The SFSF triple connectors were found by querying the officers table.

Why it matters: Connection paths are the highest-value intelligence in the report. They transform “cold application” into “warm introduction.”

What to build: A script that takes a client’s advisory board/staff names and searches the officers table for shared board memberships with target foundation officers. Output: “Person X at Client shares Board Y with Person Z at Foundation.”

Priority: P1 | Effort: Medium | Source: Part V-G

L-11: Template with Merge Fields

What happened: Report assembly took multiple sessions. Each foundation profile was written from scratch.

Why it matters: The Feb 21 supplemental report used the grant-report skill with DB-backed funder snapshots and was produced in a single session. The Feb 6 report took 3 sessions because it had no structured data input.

What to build: A report template with merge fields populated from pull_funder_snapshots.py output. The AI writes narratives and positioning strategies; all metrics come from the snapshot JSON.

Priority: P1 | Effort: Medium | Source: Part V-B

L-12: MX Record Checks

What happened: The Bill Simpson email bounce was the trigger for the entire audit cascade. An MX record check before delivery would have caught this.

Why it matters: A bouncing email is the single worst outcome for client trust. It makes the entire report look unreliable.

What to build: Before report delivery, verify MX records for every email domain included. Flag any domain without valid MX records. Already implemented in Feb 17 audit (signaturefd.com, billsimpsonfoundation.org, herzsteinfoundation.org, kovler-b.com, vbcfoundation.org all checked).

Priority: P1 | Effort: Low | Source: Part V-D

Process (P2 – Nice to Have)

L-13: Run 44-Test QA on Every Report

What happened: The QA rubric caught 24 errors on first pass. Score improved from 70.66 to 93.57 after one fix cycle.

Why it matters: +22.91 points in one pass demonstrates the review’s ROI. Without it, a report containing fabricated grant amounts and wrong trends would have been delivered.

What to build: Make the SOP_report_review.md process a mandatory step before any report delivery. The 44-test rubric already exists. Automate what can be automated (data integrity checks via SQL); keep human review for subjective items.

Priority: P2 | Effort: Low (already built, just enforce) | Source: Part V-C

L-14: Formalize Session State Files

What happened: Session state files (SESSION_STATE_*.md) preserved context across context window compactions and session boundaries. Each new session could pick up exactly where the last left off.

Why it matters: VetsBoats required 8 sessions over 80 days. Without session state, each session would start from zero.

What to build: Standardize the session state format: client context, database changes made, current pipeline stage, next steps, open questions. The /continuity and /resume skills partially address this.

Priority: P2 | Effort: Low | Source: Part V-B

L-15: Client Verification Step

What happened: Matt performed manual website verification of all 10 foundations. This client-side verification was more effective than any automated check because Matt could read eligibility language in context.

Why it matters: The client knows their own eligibility constraints better than any automated system. A structured verification step (checklist sent to client before final delivery) would catch errors like the Bill Simpson PF requirement.

What to build: A pre-delivery checklist sent to the client: “For each foundation, please verify: (1) their website is active, (2) they accept organizations like yours, (3) the geographic focus includes your area.”

Priority: P2 | Effort: Low | Source: Part V-D

L-16: Schedule_i as Data Source

What happened: The February 21 discovery searched schedule_i_grants for the first time and found 18 public charity grantmaker candidates that the standard PF-only pipeline would never surface.

Why it matters: Public charity grantmakers (community foundations, United Way affiliates, corporate giving programs) often have different eligibility criteria than private foundations. They may be more accessible to a PF client like VetsBoats.

What to build: Include schedule_i_grants in the standard candidate pool generation. For PF clients especially, public charity grantmakers may be the most viable path since they are less likely to have PF exclusion language.

Priority: P2 | Effort: Medium | Source: Part V-F

2. Kovler is_individual Data Bug

All 196 Kovler grants in pf_grants are flagged is_individual = TRUE, causing any query with WHERE is_individual = FALSE to return 0 grants. This affected the Feb 21 funder snapshot script. Investigation needed:

  • Is this a Kovler-specific data entry pattern? (Some foundations mark all grants as individual.)

3. EIN Correction Protocol

When web research provides an EIN for a foundation, verify it against the database before use:

SELECT ein, name, state, city
FROM f990_2025.dim_foundations
WHERE ein = '{input_ein}';

If no match, search by name:

SELECT ein, name, state, city
FROM f990_2025.dim_foundations
WHERE name ILIKE '%{foundation_name}%';

Present matches for human confirmation. Never proceed with an unverified EIN. The Feb 21 session had an 80% EIN error rate on input data.

4. “3 Truly Actionable > 5 Hedged”

The Feb 6 report included 5 foundations, but the Feb 20 audit revealed only 2 were fully viable and 1 was conditional. The Feb 21 supplemental report’s 5 foundations included 1 clear action item (SFSF) and 4 that required further verification.

For PF clients, the viable pool is so narrow that quality matters more than quantity. A report with 3 well-vetted, genuinely actionable foundations (with verified eligibility, working contacts, and realistic positioning) delivers more value than 5 foundations where 2-3 will be removed after the client checks the websites.

Calibration target: every foundation in the report should survive client-side website verification.

PF-to-PF Grant Detection

SELECT fg.foundation_ein,
       COUNT(DISTINCT fg.id) as pf_grant_count,
       COALESCE(SUM(fg.amount), 0) as pf_grant_total
FROM f990_2025.fact_grants fg
WHERE fg.foundation_ein = ANY($1)
  AND fg.recipient_ein IS NOT NULL
  AND EXISTS (
      SELECT 1 FROM f990_2025.pf_returns pr
      WHERE pr.ein = fg.recipient_ein
  )
GROUP BY fg.foundation_ein;

        ein, only_contri_to_preselected_ind

    WHERE ein = ANY($candidates)
