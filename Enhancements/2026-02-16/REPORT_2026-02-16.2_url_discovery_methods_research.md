## Executive Summary

The current email pipeline spec (SPEC_email_pipeline.md) only documents the February 2026 approach to URL discovery: DuckDuckGo search with domain-name matching (Script 03). But back in December 2025, three additional methods were built and one (pattern guessing) was never integrated into the new pipeline. Additionally, the February pipeline's DuckDuckGo search has a much lower hit rate (25%) than the December version (83.7%), likely because the February version only saves "high confidence" matches while December saved all confidence levels.

Key findings:

- **3 URL discovery methods from December never made it into the current pipeline**: EIN propagation, pattern guessing, and free-text URL extraction from 990-PF application fields
- **ProPublica API is a dead end** for URLs (confirmed via live API testing: no website field exists)
- **30,434 foundations** have application requirements text that may contain buried URLs and emails
- **The December scraper found URLs for 83.7% of foundations searched** vs the February pipeline's 25% test hit rate
- **Current URL source distribution**: 377,176 from IRS filing, 1,559 social media, 25 from DuckDuckGo high-confidence, 75 DuckDuckGo not-found

## December 2025 Approach

Three scripts operated directly on `f990_2025.pf_returns` (foundations only, 294K rows at the time):

### Script 1: url_ntee_phone_setup.py + url_ntee_phone_executor.py (Dec 9)

**What it did:** Internal enrichment using data already in the database.

| Method | How It Works | URLs Added |
|--------|-------------|------------|
| Junk detection | Filtered "N/A", "NONE", "WWW", phone numbers from website_url field | Cleaned 195,830 junk values (66.5%) |
| EIN propagation | Copied valid URLs from one filing year to other years of same EIN | +2,735 |
| NTEE enrichment | Matched against IRS BMF for NTEE codes (not URLs, but ran alongside) | 278,455 NTEE codes (94.6%) |

**Starting point:** 41,181 valid URLs (14.0%)
**After internal enrichment:** 43,916 valid URLs (14.9%)

### Script 2: fast_url_scraper.py (Dec 10-11)

**What it did:** Parallel DuckDuckGo search for foundations without URLs.

| Metric | Value |
|--------|-------|
| Workers | 3 parallel threads |
| Rate limiting | 2-5 second random delay per worker |
| Find rate | **83.7%** (12,722 found out of 15,200 searched) |
| Confidence levels saved | High, medium, AND low |
| Overnight result | +27,183 URLs |
| Coverage after | 47,028 valid URLs (32.8%) |

The scraper saved all confidence levels as `url_source = 'scraped_high'`, `'scraped_medium'`, `'scraped_low'`. This is a key difference from the February approach.

### Script 3: free_url_enrichment.py (Dec 9)

**What it did:** Three additional methods (partially implemented):

| Method | Status | How It Works |
|--------|--------|-------------|
| EIN propagation | WORKED | Copy URLs across filing years for same EIN. Added 2,735 URLs. |
| ProPublica API | DEAD END | API has no website field. Function was a stub that never updated anything. |
| Pattern guessing | BUILT, unclear if run at scale | Generate candidate URLs from org name (e.g., "Smith Family Foundation" generates `smithfoundation.org`, `smith-family.org`, `sf.org`), then verify with HTTP HEAD. |

**Pattern guessing logic** (from `clean_name_for_url()` function):
1. Strip common suffixes: INC, FOUNDATION, FUND, TRUST, CHARITABLE, FAMILY, THE
2. Generate up to 5 candidate URLs:
   - Full name with dashes + `.org` (e.g., `john-smith.org`)
   - Full name + `foundation.org` (e.g., `john-smithfoundation.org`)
   - First word + `foundation.org` (e.g., `johnfoundation.org`)
   - Initials + `.org` and initials + `foundation.org` (e.g., `js.org`, `jsfoundation.org`)
   - First two words + `.org` (e.g., `john-smith.org`)
3. Try each candidate with HTTP HEAD request
4. Save first one that responds with status < 400

## February 2026 Approach (Current)

Three scripts in the current pipeline, operating on `f990_2025.org_url_enrichment` (813K rows, both foundations and nonprofits):

### Script 01: Clean and Normalize URLs

Same concept as December junk detection but more thorough:
- 63 exact junk patterns + 12 regex patterns (vs simpler checks in December)
- 24-domain aggregator blocklist
- Social media extraction to separate columns
- **Result:** 377,201 real URLs (46.4% of 813K)

### Script 02: Validate URLs with HTTP

Same concept as December but applied at larger scale:
- 50 concurrent async connections
- Checkpoint/resume via state file
- **Result:** 307,885 alive (81.6% of cleaned), 69,316 dead, 0 pending

### Script 03: DuckDuckGo Search for Missing URLs

Similar to December's `fast_url_scraper.py` but with stricter filtering:
- 3 parallel workers, 2-5s delays (same as December)
- Sophisticated domain-name matching: full name, multi-word, acronym, snippet verification
- 120+ aggregator/directory domain blocklist (vs December's ~20)
- **Only "high" confidence matches saved** (medium was disabled after 0% accuracy on test)
- **Test result:** 25% hit rate on 100 foundations (vs December's 83.7%)

## What Was Lost in the Transition

### 1. EIN Propagation (December had it, February doesn't)

December's `url_ntee_phone_executor.py` copied URLs from newer filings to older filings of the same EIN. This method added 2,735 URLs in December. The February pipeline doesn't do this because it uses `org_url_enrichment` (one row per EIN) rather than `pf_returns` (multiple rows per EIN per year). However, the initial seeding of `org_url_enrichment` should have captured the best URL per EIN from the source tables, so cross-year propagation may already be implicitly handled.

**Impact:** Likely minimal. Worth verifying that the initial seed query picked the best URL per EIN.

### 2. Pattern Guessing + HTTP Verification (December had it, February doesn't)

The `free_url_enrichment.py` script generates candidate URLs from org names and checks if they exist. This approach:
- Requires no search engine (no rate limiting concerns)
- Can run very fast (just HTTP HEAD requests)
- Works for foundations with predictable naming (e.g., `fordfoundation.org`, `macfound.org`)

**Impact:** Potentially significant. Could find URLs for thousands of foundations without any DuckDuckGo search. The method is complementary to search-based discovery.

### 3. Medium/Low Confidence DuckDuckGo Matches (December saved them, February discards them)

December's scraper saved all confidence levels and achieved an 83.7% find rate. February's Script 03 only saves "high" confidence matches and achieved 25% on its test run. The February code explicitly disabled medium confidence after testing showed "0% accuracy" on single-word domain matches.

**Impact:** This is the biggest factor in the hit rate difference. The December scraper was likely saving many correct but lower-confidence matches alongside some false positives. The February pipeline's strict filtering trades recall for precision.

**Possible middle ground:** Re-enable medium confidence but with additional verification (e.g., check that the found website's title or meta description mentions the org name).

### 4. Free-Text URL/Email Mining from 990-PF Fields (Discussed Feb 11, never built)

The Feb 11 session discovered that **30,434 foundations** (21%) have application requirements text (`app_requirements_txt` / `application_submission_info_grp`) that describes how to apply. The assistant noted: "Some even contain URLs and emails buried in the text." For example:
- "online application at www.example.org/grants"
- "submit via email to grants@foundation.org"

This data is already in the database. A simple regex pass could extract URLs and emails from these text fields without any web scraping.

**Impact:** Potentially high. 30K foundations is a large pool, and these are the foundations that are open to applications (the highest-value segment).

## ProPublica API Investigation

**Tested live on 2026-02-16** against Ford Foundation (EIN 131684331), Gates Foundation (562618866), and MacArthur Foundation (237093598).

**Result:** ProPublica's Nonprofit Explorer API returns BMF (Business Master File) data only: name, address, EIN, NTEE code, financials, filing PDFs. **No website field exists in the API response.** Zero of 35 returned fields contain URL/website/link data.

The `free_url_enrichment.py` script's ProPublica method was a stub. Line 188 in the code says: "ProPublica doesn't have website directly, but has strein for lookup." The function was never completed and never updated any records.

**Conclusion:** ProPublica API is not useful for URL discovery. Remove from any future plans.

## Untapped Data Sources in the Database

### 1. Application Requirements Text (30,434 foundations)

From the Feb 11 session analysis:

| Field | Foundations with Data | Quality |
|-------|---------------------|---------|
| Requirements (how to apply) | 30,434 (21%) | Contains submit instructions, sometimes with URLs/emails |
| Restrictions (who can apply) | 22,048 (15%) | Geographic/programmatic limits |
| Deadlines | 16,925 (12%) | Free text, not structured dates |

A regex pass on the requirements text for URL patterns (`http://`, `www.`, `.org`, `.com`) and email patterns could surface contact data that's already sitting in the database.

### 2. Officers Table (26.3M records, 140K+ unique foundations)

Cross-referencing officer names with web searches could find professional profiles with contact info. However, data quality issues make this harder than expected:
- 77% of names are ALL CAPS
- Titles truncated at 12 characters (15.5%)
- 87K+ records have institutional names in the person field
- Estimated 40-55% true positive match rate, 5-10% false positives

### 3. Grant Records with Recipient Website Cross-Reference

If Foundation A gave a grant to Org B, and Org B has a website in our database, we might find Foundation A mentioned on Org B's website (funder acknowledgment pages). This is speculative and labor-intensive.
