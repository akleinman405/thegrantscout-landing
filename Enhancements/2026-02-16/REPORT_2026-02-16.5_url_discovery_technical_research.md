## Executive Summary

We evaluated 30+ tools and techniques across 7 categories for discovering website URLs for the ~129K foundations in our database that currently lack validated URLs. The research reveals that **no single tool solves this problem** -- the best strategy is a layered pipeline combining cheap/free sources first, then progressively more expensive ones.

**Key findings:**

- **Candid/GuideStar Essentials API** is the single highest-value source: it returns website URLs for 1.6M+ nonprofits including foundations, derived from validated IRS 990 filings. Free trial available, pricing requires sales contact.
- **Serper.dev** (Google SERP API) offers the best price/performance for search-based discovery at $0.30-$1.00 per 1,000 queries, meaning 130K searches would cost $39-$130.
- **DNS-based domain guessing** (generate candidate domains from org names, verify via DNS before HTTP) is essentially free and can check 350K+ domains/second with MassDNS.
- **Common Crawl** is a free dataset of 100B+ web pages that can be searched for foundation mentions, but requires significant engineering effort.
- **WHOIS reverse lookups** are largely dead due to GDPR privacy redaction -- registrant names are hidden for most domains since 2018.

**Recommended budget:** $200-$650 one-time to fill the majority of gaps (detailed in Top 5 section).

## 1. Data Enrichment APIs

### 1.1 Candid/GuideStar API

**What it does:** Provides validated nonprofit data including website URLs, financials, people, and IRS compliance for 1.6M+ organizations, sourced from IRS 990 filings and direct nonprofit reporting.

**Foundation coverage/accuracy:** High. Candid is the merger of GuideStar and Foundation Center -- the definitive source for US nonprofit/foundation data. The Essentials API v1/v2 explicitly returns a website URL field for organizations required to file a 990. Coverage should be significantly better than raw IRS XML extraction because Candid validates and enriches the data.

**Cost:** Free trial available at developer.candid.org. Paid tiers require contacting sales. The Candid Premium search product is ~$292/month, but API pricing is separate and not publicly listed. Nonprofits with budgets under $1M can get free Foundation Directory access via the "Go for Gold" program.

**Effort to implement:** 1-2 days. REST API with good documentation, SDK available on GitHub (CandidOrg/Candid.GuideStarAPI.SDK).

**Rate limits / scale:** Not publicly documented. Would need to query by EIN for 129K foundations. API supports EIN-based lookups.

**Legal risk:** None. Official API with terms of service.

**Verdict:** HIGHEST PRIORITY. This is the most authoritative source for foundation website URLs in the US. The data is already cleaned and validated. Even if pricing is nontrivial, this should be the first source investigated because it may fill the largest portion of gaps with the highest accuracy. Start with the free trial to evaluate coverage.

## 2. Search Engine APIs

### 2.1 Google Custom Search JSON API

**What it does:** Programmatic Google search results via REST API.

**Foundation coverage/accuracy:** High. Google has the best web index. A query like "Smith Family Foundation website" will almost always return the correct URL if it exists. Expected hit rate: 70-85% for foundations that actually have websites.

**Cost:** 100 free queries/day. Then $5 per 1,000 queries. 130K queries = ~$650. **IMPORTANT:** Google has announced this API is closing to new customers and existing customers must transition by January 1, 2027.

**Effort to implement:** 0.5 days. Simple REST API with API key.

**Rate limits / scale:** 10,000 queries/day maximum. 130K queries would take 13 days minimum at full throttle.

**Legal risk:** None. Official API.

**Verdict:** GOOD BUT CLOSING. At $650 for 130K queries, this is cost-effective and high-accuracy. But the API is being deprecated (deadline: Jan 2027). If we move fast, this is viable. The 13-day minimum timeline at 10K/day is the main constraint.

### 2.5 DuckDuckGo (existing)

**What it does:** Free search engine with an unofficial API (duckduckgo-search Python package).

**Foundation coverage/accuracy:** Medium. Smaller index than Google. We measured 83.7% hit rate (Dec 2025, saving all confidence levels) and 25% (Feb 2026, saving only high-confidence).

**Cost:** Free.

**Effort to implement:** Already built.

**Rate limits / scale:** ~1 request/second before rate limiting. The lite backend may allow slightly higher throughput. At 1 req/sec, 130K queries = 36 hours. Rate limit errors return HTTP 202.

**Legal risk:** Medium. No official API. TOS may prohibit automated scraping. Rate limiting is aggressive.

**Maximizing throughput:** Use the `lite` backend (https://lite.duckduckgo.com), rotate user agents, add 1-2 second delays between requests, distribute across multiple IP addresses with proxy rotation.

**Verdict:** KEEP AS BASELINE. Free and already working, but slow and rate-limited. Use for gap-filling after paid APIs have been used. Consider lowering the confidence threshold back toward the Dec 2025 approach (save medium+high confidence, not just high) since the 25% vs 83.7% hit rate difference was primarily a threshold issue, not a DDG quality issue.

### 2.7 Optimal Search Query Format

For any search-based approach, the query format significantly impacts accuracy. Based on research:

**Best query format:** `"[Foundation Name]" foundation website`

**Variations to try:**
- `"Smith Family Foundation" official website` (most precise)
- `"Smith Family Foundation" .org` (targets .org domains)
- `Smith Family Foundation site:guidestar.org OR site:candid.org` (finds profiles that link to websites)
- `"EIN [number]" foundation` (for common names, use EIN to disambiguate)

**Tips:**
- Quote the full foundation name to avoid partial matches
- Add "foundation" if the org name does not already include it
- For common names (e.g., "Community Foundation"), add the state: `"Community Foundation of [State]"`
- Use the first result's domain only if it is NOT a directory site (guidestar, candid, propublica, etc.)

## 3. Domain Discovery Techniques

### 3.1 DNS-Based Domain Verification

**What it does:** Generates candidate domain names from foundation names (e.g., "Smith Foundation" becomes smithfoundation.org, thesmithfoundation.org, etc.) and verifies whether they resolve via DNS lookup -- much faster than HTTP HEAD requests.

**Foundation coverage/accuracy:** Low-Medium for discovery alone. The accuracy depends on the quality of the name-to-domain generation patterns. Expected unique hits (domains found via guessing that search engines miss): 5-15%. But this is essentially free throughput.

**Cost:** Free. DNS lookups use public resolvers.

**Effort to implement:** 1-2 days. Generate candidate domains + bulk DNS verification.

**Rate limits / scale:** MassDNS can resolve 350,000+ domains/second using public DNS resolvers. Our entire candidate list (say 5 patterns x 130K = 650K candidates) could be checked in under 2 seconds. Even Python's socket.getaddrinfo() with threading can do 1,000+/second.

**DNS vs HTTP HEAD speed comparison:**
- DNS lookup: ~5-20ms per query, UDP-based, no connection overhead
- HTTP HEAD: ~100-2000ms per query, requires TCP handshake + TLS negotiation
- DNS is 10-100x faster and uses far less bandwidth
- Recommended workflow: DNS first to filter candidates, then HTTP HEAD only on DNS-resolved domains to verify the website actually loads

**Legal risk:** None. DNS is a public protocol.

**Verdict:** STRONGLY RECOMMENDED AS A FREE FIRST PASS. The cost is zero, the speed is extraordinary, and even a 5% unique hit rate on 130K foundations = 6,500 new URLs for free. Run this before any paid API.

---

### 3.2 Domain Name Generation Patterns

**What it does:** Algorithmically generates candidate domain names from foundation names using common naming conventions.

**Foundation coverage/accuracy:** Depends on pattern quality. Research on nonprofit domain conventions shows:

**Most common patterns for foundations:**
1. `{name}.org` -- most foundations use .org (e.g., gatesfoundation.org)
2. `{name}foundation.org` -- if "foundation" not in name
3. `the{name}.org` -- common prefix (e.g., thesmithfoundation.org)
4. `{name}.com` -- some foundations use .com
5. `{name}fund.org` -- for "fund" variants
6. `{name}.net` -- less common fallback
7. `{firstname}{lastname}foundation.org` -- for personal name foundations
8. `{lastname}foundation.org` -- abbreviated personal name
9. `{initials}foundation.org` -- initialism (e.g., jmsfoundation.org)

**Name processing rules:**
- Remove "The", "Inc", "Corp", "Trust", articles
- Remove spaces, hyphens, apostrophes
- Lowercase everything
- Handle "&" -> "and"
- Try with and without "foundation" suffix

**Cost:** Free.

**Effort to implement:** 0.5 days for the generator, combine with DNS verification from 3.1.

**Legal risk:** None.

**Verdict:** RECOMMENDED. Combine with DNS verification (3.1) for a zero-cost first pass. The pattern generator already partially exists in our codebase.

---

### 3.3 WHOIS Reverse Lookups

**What it does:** Searches WHOIS registration records to find domains registered by a specific organization name. For example, searching "Smith Family Foundation" as a registrant name to find their registered domains.

**Foundation coverage/accuracy:** VERY LOW in 2026. Since GDPR took effect in 2018, most registrars redact registrant names in public WHOIS records. Entries typically show "Redacted for Privacy" instead of the actual organization name. This applies to all gTLD domains (.org, .com, .net) regardless of the registrant's location.

**Cost:** WhoisXML API offers reverse WHOIS lookups with a database of 25.5B+ WHOIS records (including historical pre-GDPR records). Annual subscription pricing (not publicly listed, requires sales contact). WhoisFreaks and Whoxy offer similar services.

**Effort to implement:** 1-2 days for API integration.

**Rate limits / scale:** WhoisXML supports bulk lookups of 500K domains simultaneously.

**Legal risk:** None for API usage. GDPR limits what data is actually available.

**Verdict:** NOT RECOMMENDED. GDPR has made reverse WHOIS lookups largely useless for domain discovery since 2018. Historical records (pre-2018) might yield some results for long-established foundations, but the cost-to-value ratio is poor. The money is better spent on search APIs.

---

### 3.4 Certificate Transparency Logs

**What it does:** Searches public Certificate Transparency (CT) logs for SSL/TLS certificates issued to domains. Every publicly-trusted certificate must be logged, so CT logs contain a record of almost every HTTPS domain.

**Foundation coverage/accuracy:** Medium for verification, low for discovery. CT logs are indexed by domain, not by organization name. You cannot search "Smith Foundation" and find their domain. You CAN search a candidate domain (e.g., smithfoundation.org) to verify it has an active SSL certificate, which confirms it is a real website.

**Cost:** Free. crt.sh (operated by Sectigo) provides free search and a JSON API. Also accessible via direct PostgreSQL connection.

**Effort to implement:** 0.5 days.

**Rate limits / scale:** crt.sh is a free service with no documented rate limits, but hammering it would be discourteous. Use it for verification, not discovery.

**Legal risk:** None.

**Verdict:** USEFUL AS A SUPPLEMENTARY VERIFICATION STEP. After generating candidate domains (3.2) and checking DNS (3.1), verify via CT logs that the domain has an SSL certificate. This adds a confidence signal that the domain hosts a real website. But this is not a discovery tool -- it cannot find domains by organization name.

## 4. Archive and Historical Data

### 4.1 Wayback Machine / Internet Archive

**What it does:** Archives web pages over time. The CDX API allows checking whether a given URL was ever archived, and the Availability API checks if a URL is currently accessible in the archive.

**Foundation coverage/accuracy:** Medium for historical verification. If a foundation had a website that went offline, the Wayback Machine may have archived it. But this requires already knowing the URL (or a candidate URL). It cannot discover URLs by organization name.

**Use case for us:** Verify candidate domain names by checking if they were ever archived as foundation websites. Also useful for recovering information from foundation websites that have gone offline.

**Cost:** Free. No API key required.

**Effort to implement:** 0.5 days.

**Rate limits / scale:** The CDX API supports pagination for bulk queries. Reasonable rate limiting expected. For 130K candidate URLs, spread queries over 1-2 days.

**Legal risk:** None.

**Verdict:** USEFUL AS A SUPPLEMENTARY VERIFICATION STEP. Like CT logs, this helps verify candidate domains, not discover new ones. Also useful for our product: even if a foundation's website is offline, the archived version may contain contact information.

---

### 4.2 Common Crawl

**What it does:** A free, open-source archive of the entire web containing 100B+ unique pages. The CDX index allows searching by URL pattern. The full dataset (stored on AWS S3) can be searched for content within pages.

**Foundation coverage/accuracy:** Potentially high, but requires significant engineering. Two use cases:

1. **URL verification via CDX index:** Search for candidate domains (e.g., `smithfoundation.org`) in the Common Crawl index to verify they exist. This is similar to Wayback Machine but with a potentially larger crawl footprint.

2. **Content search for foundation mentions:** Download and search the WARC files for pages that mention foundation names + EINs, then extract the URL of the page that mentions them. This could find directory listings, news articles, or other pages that link to foundation websites.

**Cost:** Free for data access (hosted on AWS Open Data). Compute costs for processing WARC files could be $50-$200 on AWS depending on approach.

**Effort to implement:** 3-7 days for CDX index approach. 2-4 weeks for full content search approach.

**Rate limits / scale:** The CDX index API at index.commoncrawl.org is freely queryable. The Columnar Index (on AWS S3) allows efficient filtering and aggregation.

**Python tools:**
- `cdx-toolkit` (PyPI): CLI and Python library for querying CDX
- `cdx-index-client` (GitHub/ikreymer): Parallel query client
- `comcrawl` (PyPI): Python library for Common Crawl search

**Legal risk:** None. Open data.

**Verdict:** HIGH POTENTIAL, HIGH EFFORT. The CDX index approach (verifying candidate domains) is worth pursuing as a free verification layer. The full content search approach could yield excellent results but requires 2-4 weeks of engineering. Consider as a Phase 2 project if cheaper methods leave significant gaps.

## 6. Web Scraping Infrastructure

### 6.1 Proxy Services for Search Engine Scraping
