## Phase Summary

**Goal:** Maximize the number of foundations with validated website URLs so we can scrape them for emails in Phase 2.

**Current state:** Of 143,184 foundations, 49,265 (34.4%) have an IRS-filed URL and 14,107 (9.9%) are HTTP-validated. 93,919 (65.6%) lack any IRS-filed URL. But many of those ~94K genuinely do not have websites (they reported "N/A" on their 990-PF filings). The realistic discoverable ceiling is ~40-60% of the gap, or ~37,000-56,000 new URLs.

**For the production pipeline** (assets >= $100K, open to apps, grants to orgs, 5+ recipients): only 66-95 foundations lack URLs. This is a trivially small number.

**Bottom line:** Phase 1 is largely a solved problem for production use. The bulk URL discovery effort is for expanding coverage to the broader foundation universe.

## What We Already Have (Quick Wins)

Three zero-cost improvements can be applied immediately using data already in our database:

### 1. Re-validate 4,233 Failed URLs ($0, 2-4 hours)

Many URLs failed HTTP validation but may be fixable: add `https://`, fix common typos, check for redirects. Expected recovery: 1,000-2,000 URLs.

### 2. Mine app_form_requirements for Buried URLs ($0, 2-3 hours)

540 foundations have URLs hidden in application text fields (e.g., "apply online at www.example.org/grants"). These are not in `website_url` but are in `app_form_requirements`, `app_restrictions`, or `app_submission_deadlines`. A regex pass extracts them.

**Top buried URLs found:**

| URL | Foundations |
|-----|------:|
| csascholars.org | 71 |
| jpmorgan.com/onlinegrants | 48 |
| bankofamerica.com/grantmaking | 28 |
| wellsfargo.com/private-foundations | 8 |
| truist.com/trusteed-foundations | 8 |

### 3. Re-tune DuckDuckGo Confidence Threshold ($0, 2 hours)

The December 2025 scraper achieved 83.7% hit rate saving all confidence levels. The February 2026 version dropped to 25% by saving only high-confidence matches. Re-enabling medium confidence (as December did) recovers most of the gap. The hit rate difference was a threshold issue, not a data quality issue.

## Implementation Pipeline

Execute in this order. Each step builds on the previous.

```
Step 1: Quick wins (free, 1 day)
  ├─ Re-validate 4,233 failed URLs
  ├─ Extract 540 buried URLs from app text
  └─ Verify EIN propagation captured in initial seed

Step 2: DNS domain guessing (free, 1-2 days)
  ├─ Build domain generator (8-10 naming patterns)
  ├─ MassDNS or Python DNS verification
  └─ HTTP HEAD verify DNS-positive domains

Step 3: Serper.dev Google search ($39-$130, 1 day)
  ├─ Sign up, run 2,500 free queries to measure hit rate
  ├─ If >50% hit rate: purchase credits for full 130K run
  └─ Parse results, filter aggregator domains

Step 4: Candid API trial (free, 1-2 days)
  ├─ Sign up at developer.candid.org
  ├─ Test 1,000 EIN lookups for URL coverage
  └─ If >30% new URLs: contact sales for bulk pricing

Step 5: DuckDuckGo re-tuned (free, 12-36 hrs)
  ├─ Lower confidence threshold to December levels
  ├─ Add proxy rotation for throughput
  └─ Run on remaining gaps after Steps 3-4

Step 6: Validate all results (free, 4-8 hrs)
  ├─ HTTP validate all candidate URLs
  ├─ Cross-reference against Common Crawl CDX
  └─ Store results in database
```
