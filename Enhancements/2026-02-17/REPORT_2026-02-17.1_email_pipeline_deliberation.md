### Foundation Selection Hard Rules

1. Never lead with the most obvious foundation in the state (state-by-state "do not lead with" list provided)
2. Ban invitation-only foundations: Ford, Mellon, Lilly (outside IN), Gates, Packard, Hewlett, Woodruff, MacArthur
3. Grant examples must pass a 3-part test: same TYPE of work, same STATE, within 10x BUDGET
4. Grant dollar amounts must be within 0.5x-5x of what recipient could realistically receive
5. Use exact foundation counts from the database (67, 111, 137), never rounded
6. No foundation appears in more than 1 email per batch

## Layer 3: Data Pipeline Changes

**Full report:** `REPORT_2026-02-17.1_data_pipeline_redesign.md`

### New Foundation Ranking Algorithm

Replaces `ORDER BY total_giving_5yr DESC` with a composite **email_fit_score**:

| Dimension | Weight | What It Measures |
|-----------|--------|-----------------|
| geo_score | 0.30 | Foundation's giving concentration in target state |
| size_fit_score | 0.25 | Whether median grant fits target org budget |
| openness_score | 0.20 | New-grantee rate (proves they fund new orgs) |
| recency_score | 0.15 | Activity in last 2-3 years |
| volume_score | 0.10 | Total giving, log-scaled (prevents raw dollars from dominating) |

Plus a **0.5x penalty** for foundations with $1B+ assets.

**Expected impact:** Gates drops from 184 cohorts to ~25. Lilly drops from 117 cohorts to ~4 (Indiana only). Bank of America Charitable ($300 median) drops from 383 cohorts to near zero.

### New Grant Example Selection

Replaces `ORDER BY amount DESC` (biggest grant wins) with a 4-tier cascade function `pick_example_grant()`:

| Tier | Match Criteria | Email Language |
|------|---------------|----------------|
| 1 | Same NTEE 3-char sub-code + revenue within 3x | "They funded [Name], which does similar work to yours" |
| 2 | Same NTEE 2-char sub-code, any size | "They funded [Name], a [sub-sector] organization" |
| 3 | Same broad sector + similar revenue | "They supported [Name] in [state]" |
| 4 | Same broad sector, any size | "They've given $X to [sector] organizations in [state]" (no "similar work" claim) |

## Notes

### What All 3 Agents Independently Agree On (Highest Confidence)

1. **Never say "free."** Both spam analysis AND recipient psychology converge. Highest confidence finding.
2. **The foundation ranking algorithm is the root cause.** Sorting by total_giving_5yr DESC makes every downstream problem inevitable.
3. **Add company identity.** Hiding TheGrantScout is counterproductive. Minimum viable identity: full name + company + opt-out.
4. **Specific numbers beat round numbers.** 67 > 200. 137 > 200. Always.
5. **Emails #15, #18, #19 are unsalvageable.** Kill, don't fix.

### The Single Most Important Change

If you could only do one thing: **rebuild the foundation ranking algorithm.** The composite email_fit_score with geographic concentration, size fit, openness, and mega-penalty would have prevented every single foundation mismatch in the first batch. Prompt-level fixes and content rules are band-aids if the data feeding the email writer is still sorted by raw giving volume.

### What These Reports Cannot Fix

- **Foundation application status is imprecise.** `accepts_applications = true` in the data doesn't mean "accepts applications from small orgs." Real verification requires checking websites.
- **Subject line optimization requires A/B testing.** Guidelines are provided but performance data needs real sends.
- **The 88% first-name extraction from all_officers has unknown error rate.** Needs audit of first 100 records.
- **Foundation niche matching requires mission-level data** that the cohort table currently lacks. The `example_recipient_mission` column needs to be populated.
