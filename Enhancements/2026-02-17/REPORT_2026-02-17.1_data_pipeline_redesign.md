## Executive Summary

The current `cohort_foundation_lists` table ranks foundations by `total_giving_5yr DESC` within each state+sector combo and picks the top 15. This single sort order causes every problem reported in the email reviews: mega-foundations dominate (Bank of America Charitable appears in 383 cohorts, Gates in 184, Ford in 174), grant examples are selected by size rather than relevance, geographic mismatches go undetected (Lilly Endowment at 0.18% Connecticut giving appearing in CT cohorts), and the email writer has no signal to distinguish a genuinely open $5M community foundation from a $62B invitation-only endowment.

The redesign replaces the single-dimension ranking with a composite **email_fit_score** that balances five factors: geographic concentration in the target state, grant-size appropriateness for typical recipients, new-grantee rate (proving they actually fund new organizations), recency of giving, and a hard penalty for mega-foundations. It also introduces: a `ref_foundation_email_exclusions` table to permanently ban invitation-only mega-foundations; a new example-grant selection algorithm that matches on NTEE sub-code and recipient revenue range; a `pre_email_quality_report` materialized view for pre-flight validation; and six prospect validation checks that would have caught every error in the first email batch.

**Expected impact:** The 3 worst emails (#1, #15, #19) would never have been generated. The 40% of emails citing mega-foundations would drop to 0%. Example grants would match recipients at the sub-sector level instead of broad sector, eliminating "domestic violence cited for a disability center" comparisons.

## Current State Diagnosis

### What the Pipeline Produces Today

The `build_cohorts_v2.sql` script:
1. Gets distinct state+sector combos from `nonprofits_prospects2` where contact_email is not null
2. Aggregates foundation giving from `fact_grants` for tax_year >= 2019, joining on `recipient_state` and `LEFT(ntee_code, 1)`
3. Filters to `accepts_applications = true` from `calc_foundation_profiles`
4. Ranks by `total_giving_5yr DESC`, keeps top 15
5. Picks example grants by `amount DESC` (biggest grant in that state+sector)

## 1. Redesigned cohort_foundation_lists

### Problem

The current ranking -- `ORDER BY total_giving_5yr DESC` -- means the same 60-70 mega-foundations appear across hundreds of cohorts. Bank of America Charitable (median grant: $300) appears in 383 cohorts. This is a DAF/corporate fund that gives $300 checks, not a strategic funder. Meanwhile, excellent mid-tier foundations like Agnes M. Lindsay Trust ($3.4M, 328 new recipients, openness 0.64) or Horizon Foundation ($4.8M, 161 new recipients, openness 0.70) never surface because they are dwarfed by raw dollar volume.

### Solution: Composite email_fit_score

Replace the single `total_giving_5yr DESC` ranking with a weighted composite score that captures what actually makes a foundation good to cite in a cold email:

**Five scoring dimensions:**

| Dimension | Weight | What It Measures | Why It Matters |
|-----------|--------|-----------------|---------------|
| geo_score | 0.30 | Foundation's giving concentration in target state | Lilly at 0.18% in CT vs Cummings at 80%+ in MA |
| size_fit_score | 0.25 | Whether median grant fits target org budget range | A $300 BofA check vs a $75K Cummings grant |
| openness_score | 0.20 | New-recipient rate + accepts_applications | Proves they fund new orgs, not just repeat grantees |
| recency_score | 0.15 | Activity in last 2-3 years weighted higher | Stale foundations waste recipient's time |
| volume_score | 0.10 | Total giving in that state+sector (log-scaled) | Still rewards active funders, but logarithmically |

**Mega-foundation penalty:** If `total_assets_eoy_amt >= 1,000,000,000` (billion+), apply a 0.5x multiplier to the final score. This does not ban them -- it pushes them down from rank #1 to rank #8-12, where they might still appear but won't dominate.

### SQL Implementation

```sql
SET search_path TO f990_2025;

DROP TABLE IF EXISTS cohort_foundation_lists_v2;

CREATE TABLE cohort_foundation_lists_v2 AS
WITH target_combos AS (
    SELECT DISTINCT state, LEFT(ntee_code, 1) AS sector
    FROM nonprofits_prospects2
    WHERE contact_email IS NOT NULL
      AND ntee_code IS NOT NULL
      AND state IS NOT NULL
),

-- Pre-aggregate foundation giving by state+sector
foundation_state_sector AS (
    SELECT
        fg.recipient_state AS state,
        LEFT(dr.ntee_code, 1) AS sector,
        fg.foundation_ein,
        df.name AS foundation_name,
        SUM(fg.amount) AS total_giving_5yr,
        PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY fg.amount) AS median_grant,
        COUNT(DISTINCT fg.recipient_ein) AS unique_recipients,
        COUNT(DISTINCT CASE WHEN fg.tax_year >= 2022 THEN fg.recipient_ein END) AS recent_recipients,
        MAX(fg.tax_year) AS last_active_year,
        COUNT(DISTINCT fg.tax_year) AS active_years
    FROM fact_grants fg
    JOIN dim_recipients dr ON dr.ein = fg.recipient_ein
    JOIN dim_foundations df ON df.ein = fg.foundation_ein
    WHERE fg.tax_year >= 2019
      AND fg.amount > 0
      AND dr.ntee_code IS NOT NULL
    GROUP BY fg.recipient_state, LEFT(dr.ntee_code, 1), fg.foundation_ein, df.name
    HAVING SUM(fg.amount) > 0
),

-- Join with foundation profiles and state distribution for scoring inputs
scored AS (
    SELECT
        fss.state,
        fss.sector,
        fss.foundation_ein,
        fss.foundation_name,
        fss.total_giving_5yr,
        fss.median_grant,
        fss.unique_recipients,
        fss.recent_recipients,
        fss.last_active_year,
        fss.active_years,
        cfp.openness_score AS cfp_openness,
        cfp.accepts_applications,
        cfp.new_recipients_5yr,
        cfp.unique_recipients_5yr AS cfp_unique_5yr,
        cfp.total_giving_5yr AS cfp_total_giving,
        fp.total_assets_eoy_amt AS assets,

        -- 1. GEO SCORE (0-1): How concentrated is this foundation in the target state?
        --    Uses stg_foundation_state_dist.state_pct
        --    0.40+ = very focused, 0.10-0.39 = moderate, <0.10 = unfocused
        COALESCE(fsd.state_pct, 0) AS state_pct,
        LEAST(COALESCE(fsd.state_pct, 0) / 0.40, 1.0) AS geo_score,

        -- 2. SIZE FIT SCORE (0-1): Is median grant appropriate for small/mid nonprofits?
        --    Sweet spot: $5,000 - $200,000. Penalize <$1,000 and >$500,000.
        CASE
            WHEN fss.median_grant BETWEEN 5000 AND 200000 THEN 1.0
            WHEN fss.median_grant BETWEEN 1000 AND 5000 THEN 0.6
            WHEN fss.median_grant BETWEEN 200000 AND 500000 THEN 0.7
            WHEN fss.median_grant > 500000 THEN 0.3
            WHEN fss.median_grant < 1000 THEN 0.2
            ELSE 0.5
        END AS size_fit_score,

        -- 3. OPENNESS SCORE (0-1): Does this foundation fund NEW organizations?
        --    Combines calc_foundation_profiles.openness_score with new_recipients ratio
        CASE
            WHEN cfp.accepts_applications = true
                 AND cfp.new_recipients_5yr::float / NULLIF(cfp.unique_recipients_5yr, 0) >= 0.30
            THEN LEAST(
                0.5 + 0.5 * (cfp.new_recipients_5yr::float / NULLIF(cfp.unique_recipients_5yr, 0)),
                1.0
            )
            WHEN cfp.accepts_applications = true THEN 0.5
            ELSE 0.2
        END AS openness_calc,

        -- 4. RECENCY SCORE (0-1): How recently active?
        CASE
            WHEN fss.last_active_year >= 2023 AND fss.active_years >= 3 THEN 1.0
            WHEN fss.last_active_year >= 2023 THEN 0.9
            WHEN fss.last_active_year >= 2022 THEN 0.7
            WHEN fss.last_active_year >= 2021 THEN 0.5
            ELSE 0.3
        END AS recency_score,

        -- 5. VOLUME SCORE (0-1): Log-scaled total giving in this state+sector
        --    Prevents raw dollars from dominating
        LEAST(LN(fss.total_giving_5yr + 1) / LN(100000000), 1.0) AS volume_score,

        -- MEGA-FOUNDATION PENALTY
        CASE
            WHEN fp.total_assets_eoy_amt >= 1000000000 THEN 0.50
            WHEN fp.total_assets_eoy_amt >= 500000000 THEN 0.75
            ELSE 1.00
        END AS mega_penalty

    FROM foundation_state_sector fss
    JOIN target_combos tc ON tc.state = fss.state AND tc.sector = fss.sector
    LEFT JOIN calc_foundation_profiles cfp ON cfp.ein = fss.foundation_ein
    LEFT JOIN foundation_prospects2 fp ON fp.ein = fss.foundation_ein
    LEFT JOIN stg_foundation_state_dist fsd
        ON fsd.foundation_ein = fss.foundation_ein
        AND fsd.recipient_state = fss.state
    -- Exclude foundations banned for email
    LEFT JOIN ref_foundation_email_exclusions exc ON exc.foundation_ein = fss.foundation_ein
    WHERE exc.foundation_ein IS NULL  -- not banned
      AND (cfp.accepts_applications = true OR cfp.openness_score >= 0.30)
      AND cfp.last_active_year >= 2021
      AND cfp.has_grant_history = true
),

-- Compute final composite score and rank
ranked AS (
    SELECT
        s.*,
        -- Weighted composite score with mega penalty
        (
            0.30 * s.geo_score +
            0.25 * s.size_fit_score +
            0.20 * s.openness_calc +
            0.15 * s.recency_score +
            0.10 * s.volume_score
        ) * s.mega_penalty AS email_fit_score,
        ROW_NUMBER() OVER (
            PARTITION BY s.state, s.sector
            ORDER BY (
                0.30 * s.geo_score +
                0.25 * s.size_fit_score +
                0.20 * s.openness_calc +
                0.15 * s.recency_score +
                0.10 * s.volume_score
            ) * s.mega_penalty DESC
        ) AS foundation_rank
    FROM scored s
)

SELECT
    r.state,
    r.sector AS ntee_sector,
    r.foundation_rank,
    r.foundation_ein,
    r.foundation_name,
    r.total_giving_5yr,
    r.median_grant,
    r.unique_recipients,
    r.recent_recipients,
    r.last_active_year,
    r.assets,
    r.state_pct,
    r.geo_score,
    r.size_fit_score,
    r.openness_calc AS openness_score,
    r.recency_score,
    r.volume_score,
    r.mega_penalty,
    r.email_fit_score,
    r.accepts_applications
FROM ranked r
WHERE r.foundation_rank <= 25;  -- Keep top 25 per combo (up from 15)
```

### What Changes

| Before | After |
|--------|-------|
| Ranked by total_giving_5yr DESC | Ranked by composite email_fit_score |
| Top 15 per combo | Top 25 per combo (wider pool for the email writer) |
| No geographic filter | geo_score penalizes foundations with <10% giving in target state |
| No size filter | size_fit_score penalizes $300 DAF grants and $500K+ strategic grants |
| No new-grantee signal | openness_score rewards foundations that fund 30%+ new recipients |
| No recency beyond 5yr window | recency_score weights 2023-2024 activity higher |
| No mega penalty | 0.5x multiplier for $1B+ asset foundations |
| No exclusion list | LEFT JOIN to ref_foundation_email_exclusions |

### Expected Impact

- Gates Foundation drops from appearing in 184 cohorts to roughly 20-30 (only where it has genuine geographic concentration, like WA)
- Lilly Endowment drops from 117 cohorts to roughly 3-5 (IN and surrounding states)
- Bank of America Charitable drops from 383 cohorts to near zero (median grant $300 gets a 0.2 size_fit_score)
- Mid-tier community foundations with high openness scores rise to ranks #1-#5 in their home states

## 2. Grant Example Selection

### Problem

The current query picks the example grant with `ORDER BY fg.amount DESC LIMIT 1` -- the single biggest grant the foundation gave in that state+sector. This produces absurd comparisons:
- Helpmate Inc (domestic violence, NTEE I40) cited as "similar work" for a disability resource center (NTEE P80)
- Museum of Chinese in America (NTEE A84) cited for a motor racing archive (NTEE A84, but entirely different niche)
- National Urban League ($50M+ budget) cited as comparable for a $400K community org

### Solution: NTEE Sub-Code + Revenue Size Matching

The example grant should be the most RELEVANT grant, not the biggest. Relevance means:
1. **NTEE sub-code match**: Same 2-3 character prefix as the target prospect (e.g., P43 domestic violence matched to P43, not P20 general human services)
2. **Revenue-size similarity**: The example recipient's revenue should be within 3x of the target prospect's revenue
3. **Recency**: Prefer grants from the last 2 years

### SQL Implementation

```sql
-- For a given prospect (state, ntee_code, revenue) and foundation,
-- find the best example grant using a cascading match strategy

CREATE OR REPLACE FUNCTION f990_2025.pick_example_grant(
    p_foundation_ein VARCHAR,
    p_state VARCHAR,
    p_ntee_code VARCHAR,     -- full NTEE code of the target prospect (e.g., 'P43Z')
    p_prospect_revenue NUMERIC  -- target prospect's total_revenue
)
RETURNS TABLE (
    example_recipient_name VARCHAR,
    example_recipient_ein VARCHAR,
    example_grant_amount BIGINT,
    example_grant_purpose VARCHAR,
    example_grant_year INT,
    example_recipient_revenue NUMERIC,
    match_tier INT  -- 1=best, 4=fallback
) AS $$
BEGIN
    -- Tier 1: Same NTEE sub-code (3 chars), similar revenue (within 3x), same state
    RETURN QUERY
    SELECT
        dr.name, dr.ein, fg.amount, fg.purpose_text, fg.tax_year,
        nr.total_revenue, 1 AS match_tier
    FROM f990_2025.fact_grants fg
    JOIN f990_2025.dim_recipients dr ON dr.ein = fg.recipient_ein
    LEFT JOIN LATERAL (
        SELECT n.total_revenue
        FROM f990_2025.nonprofit_returns n
        WHERE n.ein = dr.ein
        ORDER BY n.tax_year DESC NULLS LAST
        LIMIT 1
    ) nr ON true
    WHERE fg.foundation_ein = p_foundation_ein
      AND fg.recipient_state = p_state
      AND fg.tax_year >= 2021
      AND fg.amount > 0
      AND dr.ntee_code IS NOT NULL
      AND LEFT(dr.ntee_code, 3) = LEFT(p_ntee_code, 3)
      AND (
          p_prospect_revenue IS NULL
          OR nr.total_revenue IS NULL
          OR (nr.total_revenue BETWEEN p_prospect_revenue / 3.0 AND p_prospect_revenue * 3.0)
      )
    ORDER BY fg.tax_year DESC, fg.amount DESC
    LIMIT 1;

    IF FOUND THEN RETURN; END IF;

    -- Tier 2: Same NTEE sub-code (2 chars), same state, any size
    RETURN QUERY
    SELECT
        dr.name, dr.ein, fg.amount, fg.purpose_text, fg.tax_year,
        nr.total_revenue, 2 AS match_tier
    FROM f990_2025.fact_grants fg
    JOIN f990_2025.dim_recipients dr ON dr.ein = fg.recipient_ein
    LEFT JOIN LATERAL (
        SELECT n.total_revenue
        FROM f990_2025.nonprofit_returns n
        WHERE n.ein = dr.ein
        ORDER BY n.tax_year DESC NULLS LAST
        LIMIT 1
    ) nr ON true
    WHERE fg.foundation_ein = p_foundation_ein
      AND fg.recipient_state = p_state
      AND fg.tax_year >= 2019
      AND fg.amount > 0
      AND dr.ntee_code IS NOT NULL
      AND LEFT(dr.ntee_code, 2) = LEFT(p_ntee_code, 2)
    ORDER BY
        -- Prefer similar size, then recency, then amount
        ABS(COALESCE(nr.total_revenue, 0) - COALESCE(p_prospect_revenue, 0)) ASC,
        fg.tax_year DESC
    LIMIT 1;

    IF FOUND THEN RETURN; END IF;

    -- Tier 3: Same broad sector (1 char), same state, similar revenue
    RETURN QUERY
    SELECT
        dr.name, dr.ein, fg.amount, fg.purpose_text, fg.tax_year,
        nr.total_revenue, 3 AS match_tier
    FROM f990_2025.fact_grants fg
    JOIN f990_2025.dim_recipients dr ON dr.ein = fg.recipient_ein
    LEFT JOIN LATERAL (
        SELECT n.total_revenue
        FROM f990_2025.nonprofit_returns n
        WHERE n.ein = dr.ein
        ORDER BY n.tax_year DESC NULLS LAST
        LIMIT 1
    ) nr ON true
    WHERE fg.foundation_ein = p_foundation_ein
      AND fg.recipient_state = p_state
      AND fg.tax_year >= 2019
      AND fg.amount > 0
      AND dr.ntee_code IS NOT NULL
      AND LEFT(dr.ntee_code, 1) = LEFT(p_ntee_code, 1)
      AND (
          p_prospect_revenue IS NULL
          OR nr.total_revenue IS NULL
          OR (nr.total_revenue BETWEEN p_prospect_revenue / 5.0 AND p_prospect_revenue * 5.0)
      )
    ORDER BY fg.tax_year DESC, fg.amount DESC
    LIMIT 1;

    IF FOUND THEN RETURN; END IF;

    -- Tier 4: Same broad sector, same state, any size (original behavior minus biggest-grant bias)
    RETURN QUERY
    SELECT
        dr.name, dr.ein, fg.amount, fg.purpose_text, fg.tax_year,
        nr.total_revenue, 4 AS match_tier
    FROM f990_2025.fact_grants fg
    JOIN f990_2025.dim_recipients dr ON dr.ein = fg.recipient_ein
    LEFT JOIN LATERAL (
        SELECT n.total_revenue
        FROM f990_2025.nonprofit_returns n
        WHERE n.ein = dr.ein
        ORDER BY n.tax_year DESC NULLS LAST
        LIMIT 1
    ) nr ON true
    WHERE fg.foundation_ein = p_foundation_ein
      AND fg.recipient_state = p_state
      AND fg.tax_year >= 2019
      AND fg.amount > 0
      AND dr.ntee_code IS NOT NULL
      AND LEFT(dr.ntee_code, 1) = LEFT(p_ntee_code, 1)
    ORDER BY fg.tax_year DESC, fg.amount DESC
    LIMIT 1;
END;
$$ LANGUAGE plpgsql;
```

### Key Design Decisions

**Why NTEE sub-code matters:** The current system matches at the 1-character sector level. Sector "P" (Human Services) includes P20 (general services), P43 (domestic violence), P80 (disability), P85 (homeless services), and dozens of others. A domestic violence shelter wants to see a grant to another DV org, not to a disability center. Matching at the 2-3 character level (P43 to P43) fixes this.

**Why revenue matching matters:** National Urban League ($50M+ budget) is not comparable to a $400K community org. The 3x revenue range (Tier 1) ensures examples are in the same ballpark. A $600K org sees a $200K-$1.8M example; a $5M org sees a $1.7M-$15M example.

**Graceful degradation:** The 4-tier cascade means we never fail to find an example. But the `match_tier` column is stored, so the email writer can use different language:
- Tier 1: "They recently funded [Name], which does similar work to yours, with a $X grant"
- Tier 2: "They've funded [Name] in [state], a [sub-sector] organization, with $X"
- Tier 3: "They've supported [Name] in [state], a [sector] org with $X"
- Tier 4: "They've given $X to [sector] organizations in [state]" (no "similar work" claim)

### Fallback When No Good Match Exists

If even Tier 4 returns no results (the foundation has no grants in that state+sector in the last 5 years), the foundation should NOT appear in that cohort at all. This is a data integrity issue -- the old system could surface a foundation with one stray grant as a "top funder."

Add to the cohort build script:

```sql
-- After scoring, remove foundations with no viable example grant
DELETE FROM cohort_foundation_lists_v2 cfl
WHERE NOT EXISTS (
    SELECT 1 FROM fact_grants fg
    JOIN dim_recipients dr ON dr.ein = fg.recipient_ein
    WHERE fg.foundation_ein = cfl.foundation_ein
      AND fg.recipient_state = cfl.state
      AND LEFT(dr.ntee_code, 1) = cfl.ntee_sector
      AND fg.tax_year >= 2019
      AND fg.amount > 0
);
```

## 3. Foundation Exclusion List

### Problem

The current pipeline has one filter: `accepts_applications = true` from `calc_foundation_profiles`. This lets through every foundation where `only_contri_to_preselected_ind = FALSE` on their 990-PF. But many foundations technically "accept applications" while being effectively invitation-only (Ford, Mellon, Gates) or giving only in a narrow geography (Lilly = 67% Indiana). Their raw volume pushes them to rank #1 in dozens of states where they have no real relevance.

### Solution: ref_foundation_email_exclusions

A permanent exclusion table with explicit reasons, maintainable over time.

### Exclusion Criteria

| Criterion | Logic | Rationale |
|-----------|-------|-----------|
| Patient Assistance Foundations | name ILIKE '%patient assistance%' OR name ILIKE '%patient access%' | These give billions to individuals, not orgs. 10 foundations, $64B total. |
| Trusts passing to single entity | unique_recipients_5yr = 0 AND total_giving_5yr > 100M | Kellogg Trust, BMS Patient Assistance, etc. Pass-through vehicles. |
| Strategic/invitation-only mega-foundations | Curated list of foundations known to be invitation-only | Ford, Gates, Bloomberg, Soros, etc. Cannot be detected by data alone. |
| Geographic monopolies cited outside home state | assets >= 1B AND max_state_pct >= 0.60 AND cited in other state | Lilly (67% IN), Woodruff (100% GA), Duke Endowment (NC only) |
| DAF/corporate giving vehicles with micro-grants | median_grant < 500 AND unique_recipients > 10000 | Bank of America ($300 median, 28K recipients), JPMorgan ($550, 15K recipients) |

### DDL and Initial Population

```sql
CREATE TABLE f990_2025.ref_foundation_email_exclusions (
    foundation_ein VARCHAR(20) NOT NULL,
    foundation_name TEXT,
    exclusion_reason VARCHAR(100) NOT NULL,
    exclusion_category VARCHAR(50) NOT NULL,
    -- Categories: 'invitation_only', 'patient_assistance', 'pass_through',
    --             'geographic_monopoly', 'micro_grant_daf', 'manual'
    assets NUMERIC,
    notes TEXT,
    added_date DATE DEFAULT CURRENT_DATE,
    added_by VARCHAR(100) DEFAULT 'pipeline',
    PRIMARY KEY (foundation_ein)
);

-- 1. Patient assistance foundations (giving to individuals, not orgs)
INSERT INTO f990_2025.ref_foundation_email_exclusions
    (foundation_ein, foundation_name, exclusion_reason, exclusion_category, notes)
SELECT DISTINCT
    cfp.ein,
    df.name,
    'Patient assistance foundation - gives to individuals not organizations',
    'patient_assistance',
    'total_giving_5yr: ' || cfp.total_giving_5yr::text
FROM f990_2025.calc_foundation_profiles cfp
JOIN f990_2025.dim_foundations df ON df.ein = cfp.ein
WHERE (df.name ILIKE '%patient assistance%'
    OR df.name ILIKE '%patient access%'
    OR df.name ILIKE '%patient foundation%')
  AND cfp.unique_recipients_5yr <= 30
ON CONFLICT DO NOTHING;

-- 2. Pass-through trusts (zero or near-zero unique recipients, huge dollars)
INSERT INTO f990_2025.ref_foundation_email_exclusions
    (foundation_ein, foundation_name, exclusion_reason, exclusion_category, notes)
SELECT
    cfp.ein,
    df.name,
    'Pass-through trust with 0 org recipients',
    'pass_through',
    'total_giving_5yr: ' || cfp.total_giving_5yr::text || ', recipients: ' || cfp.unique_recipients_5yr::text
FROM f990_2025.calc_foundation_profiles cfp
```

## 4. Prospect Data Validation

### Problem

The first email batch had multiple data quality failures:
- Email #14: Sojourner House (MA) but email domain sjhpa.org (PA entity)
- Email #20: Contact name "Andrea" but email goes to "sue@..."
- 766 prospects have contact_first_name = 'there' (from "Hi there" fallback)
- 3,492 of 6,372 prospects (54.8%) have generic emails (info@, contact@, office@, admin@, team@)
- Prospects with .gov email addresses (government entities, not nonprofits)

### Solution: Seven Validation Checks

Run these BEFORE selecting prospects for any email cohort. Results populate the pre-generation quality report (Section 7).

```sql
-- VALIDATION 1: Placeholder name detection
-- Flags prospects where contact_first_name is a greeting word, not a name
SELECT ein, organization_name, contact_first_name, contact_email,
    'placeholder_name' AS issue
FROM f990_2025.nonprofits_prospects2
WHERE contact_email IS NOT NULL
  AND contact_first_name IN ('there', 'Hi', 'Hello', 'Dear', 'Sir', 'Madam', 'whom', 'Team')
;
-- IMPACT: 766 "there" records. These should use "Hi there" in emails (acceptable)
-- but should be deprioritized in favor of named contacts.

-- VALIDATION 2: Name-email mismatch detection
-- Flags where contact_first_name comes from a different person than the email
-- Heuristic: if email has a personal prefix, check if first name initial matches
SELECT
    np.ein, np.organization_name,
    np.contact_first_name,
    np.contact_email,
    SPLIT_PART(np.contact_email, '@', 1) AS email_prefix,
    np.ed_ceo_name,
    np.president_name,
    CASE
        WHEN LOWER(SPLIT_PART(np.contact_email, '@', 1)) LIKE LOWER(np.contact_first_name) || '%'
            THEN 'match'
        WHEN SPLIT_PART(np.contact_email, '@', 1) IN ('info','contact','office','admin','team',
            'hello','support','general','inquiries','mail','email','main','reception')
            THEN 'generic_email'
        ELSE 'possible_mismatch'
    END AS name_email_status
FROM f990_2025.nonprofits_prospects2 np
WHERE np.contact_email IS NOT NULL
  AND np.contact_first_name NOT IN ('there', 'Hi', 'Hello')
  AND SPLIT_PART(np.contact_email, '@', 1) NOT IN
      ('info','contact','office','admin','team','hello','support','general',
       'inquiries','mail','email','main','reception')
  AND NOT LOWER(SPLIT_PART(np.contact_email, '@', 1)) LIKE LOWER(np.contact_first_name) || '%'
  AND NOT LOWER(SPLIT_PART(np.contact_email, '@', 1)) LIKE
      LOWER(LEFT(np.contact_first_name, 1)) || '%'
;
-- IMPACT: These should be flagged but not necessarily excluded.
-- The email writer should use the generic "Hi there" instead of the wrong name.

-- VALIDATION 3: Email domain vs website domain mismatch
-- Flags where the email domain has no relationship to the org's website
SELECT
    np.ein, np.organization_name, np.state,
    np.contact_email,
    SPLIT_PART(np.contact_email, '@', 2) AS email_domain,
    np.website,
    'domain_mismatch' AS issue
FROM f990_2025.nonprofits_prospects2 np
WHERE np.contact_email IS NOT NULL
  AND np.website IS NOT NULL
  AND np.website NOT IN ('N/A', 'NONE', '0', '')
  AND LENGTH(np.website) > 5
  -- Extract and compare domains (simplified)
  AND LOWER(SPLIT_PART(np.contact_email, '@', 2))
      NOT LIKE '%' || LOWER(
          REPLACE(REPLACE(REPLACE(REPLACE(
              SPLIT_PART(SPLIT_PART(np.website, '://', 2), '/', 1),
          'www.', ''), 'www2.', ''), 'www3.', ''), 'http:', '')
      ) || '%'
  AND LOWER(
      REPLACE(REPLACE(REPLACE(REPLACE(
          SPLIT_PART(SPLIT_PART(np.website, '://', 2), '/', 1),
      'www.', ''), 'www2.', ''), 'www3.', ''), 'http:', '')
  ) NOT LIKE '%' || LOWER(SPLIT_PART(np.contact_email, '@', 2)) || '%'
;
-- IMPACT: Catches the Sojourner House (MA org with PA email) problem.
-- These are high-risk: the email may reach the WRONG ORGANIZATION entirely.

-- VALIDATION 4: Government email detection
SELECT
    np.ein, np.organization_name, np.state,
    np.contact_email,
    'gov_email' AS issue
FROM f990_2025.nonprofits_prospects2 np
WHERE np.contact_email IS NOT NULL
  AND (
      np.contact_email LIKE '%.gov'
      OR np.contact_email LIKE '%.gov.%'
      OR np.contact_email LIKE '%@us.%'
  )
;
-- IMPACT: Government emails should be excluded entirely.
-- A cold sales email to a .gov address is inappropriate and may violate policy.

-- VALIDATION 5: Stale filing detection
SELECT
    np.ein, np.organization_name,
    np.most_recent_tax_year,
    'stale_filer' AS issue
FROM f990_2025.nonprofits_prospects2 np
WHERE np.contact_email IS NOT NULL
  AND np.most_recent_tax_year < 2022
;
-- IMPACT: Orgs that haven't filed in 3+ years may be defunct.
-- Should be excluded from email cohorts.

-- VALIDATION 6: Missing NTEE code
SELECT
    np.ein, np.organization_name, np.state,
    'missing_ntee' AS issue
FROM f990_2025.nonprofits_prospects2 np
WHERE np.contact_email IS NOT NULL
  AND np.ntee_code IS NULL
;
-- IMPACT: 1,388 prospects (21.8%) have no NTEE code.
-- These cannot be matched to cohorts and should be excluded from
-- sector-specific campaigns.

-- VALIDATION 7: Duplicate email detection (across different EINs)
SELECT
    contact_email,
    COUNT(*) AS org_count,
    STRING_AGG(organization_name, ' | ' ORDER BY organization_name) AS org_names,
    'duplicate_email' AS issue
FROM f990_2025.nonprofits_prospects2
WHERE contact_email IS NOT NULL
GROUP BY contact_email
HAVING COUNT(*) > 1
;
-- IMPACT: If same email appears for multiple orgs, sending to both
-- reveals the mass-email nature of the campaign.
```

### Prospect Quality Tiers

Based on the validations, categorize each prospect:

```sql
ALTER TABLE f990_2025.nonprofits_prospects2
    ADD COLUMN IF NOT EXISTS email_quality_tier VARCHAR(1),
    ADD COLUMN IF NOT EXISTS email_quality_flags TEXT[];

UPDATE f990_2025.nonprofits_prospects2 np SET
    email_quality_flags = ARRAY_REMOVE(ARRAY[
        CASE WHEN np.contact_first_name IN ('there','Hi','Hello','Dear') THEN 'placeholder_name' END,
        CASE WHEN np.contact_email LIKE '%.gov' OR np.contact_email LIKE '%.gov.%' THEN 'gov_email' END,
        CASE WHEN np.most_recent_tax_year < 2022 THEN 'stale_filer' END,
        CASE WHEN np.ntee_code IS NULL THEN 'missing_ntee' END,
        CASE WHEN np.contact_email LIKE 'info@%' OR np.contact_email LIKE 'contact@%'
            OR np.contact_email LIKE 'office@%' OR np.contact_email LIKE 'admin@%'
            OR np.contact_email LIKE 'team@%' THEN 'generic_email' END
    ], NULL),
    email_quality_tier = CASE
        -- Tier A: Named contact, personal email, recent filer, has NTEE
        WHEN np.contact_first_name NOT IN ('there','Hi','Hello','Dear')
            AND np.contact_email NOT LIKE 'info@%'
            AND np.contact_email NOT LIKE 'contact@%'
            AND np.contact_email NOT LIKE 'office@%'
            AND np.contact_email NOT LIKE 'admin@%'
            AND np.contact_email NOT LIKE 'team@%'
            AND np.contact_email NOT LIKE '%.gov'
            AND np.most_recent_tax_year >= 2022
            AND np.ntee_code IS NOT NULL
        THEN 'A'
        -- Tier B: Has named contact OR personal email, recent filer
        WHEN np.most_recent_tax_year >= 2022
            AND np.ntee_code IS NOT NULL
            AND np.contact_email NOT LIKE '%.gov'
            AND (np.contact_first_name NOT IN ('there','Hi','Hello','Dear')
                 OR (np.contact_email NOT LIKE 'info@%'
                     AND np.contact_email NOT LIKE 'contact@%'
                     AND np.contact_email NOT LIKE 'office@%'))
        THEN 'B'
        -- Tier C: Generic email or placeholder name but otherwise valid
        WHEN np.most_recent_tax_year >= 2022
            AND np.contact_email NOT LIKE '%.gov'
        THEN 'C'
        -- Tier D: Has issues (stale filer, gov email, etc.)
        ELSE 'D'
    END
WHERE np.contact_email IS NOT NULL;
```

### Expected Distribution (Estimated)

| Tier | Criteria | Est. Count | Email Strategy |
|------|----------|------------|---------------|
| A | Named contact + personal email + NTEE | ~1,200 | Use "Hi [Name]" -- highest priority |
| B | Named contact OR personal email | ~2,500 | Use name or "Hi there" |
| C | Generic email, placeholder name | ~2,200 | Use "Hi there" -- lower priority |
| D | Stale/gov/broken | ~470 | Exclude from campaigns |

**Rule:** Never send to Tier D. Start campaigns with Tier A, expand to B, then C.

## 5. Cohort Sizing

### Problem

The `cohort_viability` table reports raw `foundation_count` per state+sector combo, ranging from 1 (US Virgin Islands / Civil Rights) to 2,880 (New York / Education). The emails currently say "top 200 list" for any combo with 200+ foundations. This has two problems:
1. "200 foundations" sounds like a data dump, not curated intelligence
2. Small numbers (<50) sound too precise and expose the methodology

### Recommendation

**The number cited in emails should always be between 25 and 75, with a preference for a round number near the true count.**

| True Foundation Count | What to Cite | Rationale |
|----------------------|-------------|-----------|
| 1-24 | Do not cite a number; say "several foundations" | Too small to be impressive |
| 25-49 | Cite the real number rounded to nearest 5 | "27 foundations" feels curated and credible |
| 50-75 | Cite the real number rounded to nearest 5 | "65 foundations" is the sweet spot |
| 76-150 | Say "over 50 foundations" or cite 75 | Large enough to impress, not a data dump |
| 151-500 | Say "75 foundations we think are the best match" | Curated subset > exhaustive list |
| 500+ | Say "75 foundations" (the curated list size) | Never mention 500+; it's obviously machine-generated |

### Implementation

```sql
-- Add display columns to cohort_viability
ALTER TABLE f990_2025.cohort_viability
    ADD COLUMN IF NOT EXISTS display_count INTEGER,
    ADD COLUMN IF NOT EXISTS display_text VARCHAR(100);

UPDATE f990_2025.cohort_viability SET
    display_count = CASE
        WHEN foundation_count < 25 THEN NULL
        WHEN foundation_count BETWEEN 25 AND 49 THEN ROUND(foundation_count / 5.0) * 5
        WHEN foundation_count BETWEEN 50 AND 75 THEN ROUND(foundation_count / 5.0) * 5
        WHEN foundation_count BETWEEN 76 AND 150 THEN 75
        ELSE 75
    END,
    display_text = CASE
        WHEN foundation_count < 25 THEN 'several foundations that fund'
        WHEN foundation_count BETWEEN 25 AND 49 THEN (ROUND(foundation_count / 5.0) * 5)::text || ' foundations that fund'
        WHEN foundation_count BETWEEN 50 AND 75 THEN (ROUND(foundation_count / 5.0) * 5)::text || ' foundations that fund'
        WHEN foundation_count BETWEEN 76 AND 150 THEN 'over 75 foundations that fund'
        ELSE '75 foundations we identified that fund'
    END;
```

### When Small Counts Help Credibility

Small counts (25-67) are actually MORE credible than large ones because:
1. They imply curation ("we filtered to the 35 that matter")
2. They match how grant writers actually work (20-40 prospects/year)
3. They avoid the "data dump" perception
4. Email #13 (Kansas / Housing, 67 foundations) was rated one of the strongest emails

**The number 67 is more persuasive than 200.** It says "we did specific research for your state and sector" rather than "we ran a database query."

## 6. Geographic Relevance Scoring

### Problem

Lilly Endowment gives 39.6% of its grants in Indiana, 37.2% in blank/unknown states, and 1.8% in California. Yet it appeared in 117 state+sector cohorts because it has scattered grants across the country (one grant to Hartford International University in CT triggers inclusion in CT cohorts). The current system has no geographic relevance filter.

### Solution: Three-Tier Geographic Classification

```sql
-- Classify every foundation's relationship to every state
CREATE MATERIALIZED VIEW f990_2025.mv_foundation_geo_relevance AS
SELECT
    fsd.foundation_ein,
    fsd.recipient_state AS state,
    fsd.state_pct,
    fsd.grant_count,
    fsd.total_grants AS state_giving_dollars,

    -- Classify geographic relevance
    CASE
        WHEN fsd.state_pct >= 0.25 THEN 'primary'      -- 25%+ of giving = home state
        WHEN fsd.state_pct >= 0.05 THEN 'secondary'    -- 5-24% = meaningful presence
        WHEN fsd.state_pct >= 0.01
             AND fsd.grant_count >= 5 THEN 'tertiary'  -- 1-5%, 5+ grants = some presence
        ELSE 'incidental'                               -- <1% or <5 grants = noise
    END AS geo_tier,

    -- For foundations with no clear geographic focus (national funders)
    CASE
        WHEN max_pct.max_state_pct < 0.15 THEN true
        ELSE false
    END AS is_national_funder

FROM f990_2025.stg_foundation_state_dist fsd
JOIN (
    SELECT foundation_ein, MAX(state_pct) AS max_state_pct
    FROM f990_2025.stg_foundation_state_dist
    WHERE recipient_state != '' AND recipient_state IS NOT NULL
    GROUP BY foundation_ein
) max_pct ON max_pct.foundation_ein = fsd.foundation_ein
WHERE fsd.recipient_state != '' AND fsd.recipient_state IS NOT NULL;

CREATE UNIQUE INDEX ON f990_2025.mv_foundation_geo_relevance (foundation_ein, state);
```

### Geographic Thresholds

| Geo Tier | State PCT Required | Min Grants | Meaning | Use in Email |
|----------|-------------------|------------|---------|-------------|
| primary | >= 25% | any | This is a home-state funder | "A [State] foundation that..." -- strongest claim |
| secondary | >= 5% | any | Meaningful geographic presence | "...has funded [sector] organizations across [State]" |
| tertiary | >= 1% | >= 5 grants | Some presence, not core | Use with caution; don't claim geographic focus |
| incidental | < 1% or < 5 grants | any | Noise / one-off grants | EXCLUDE from cohort for this state |

### Integration with Cohort Build

The cohort build script (Section 1) already incorporates geographic scoring through the `geo_score` dimension. But we need a hard floor:

```sql
-- In the cohort build, add a HARD filter:
-- Foundations must be at least "tertiary" in the target state
WHERE (
    mgr.geo_tier IN ('primary', 'secondary', 'tertiary')
    OR mgr.is_national_funder = true  -- national funders allowed everywhere
)
```

### Handling National Funders

Some foundations genuinely give everywhere (Kellogg, Robert Wood Johnson, Kresge). Their top state might only be 8-12% of giving. These are classified as `is_national_funder = true` (no state has > 15% of their giving).

National funders ARE allowed in any state's cohort, but:
- They receive a lower `geo_score` (their state_pct is low, so 0.30 * geo_score is small)
- They still benefit from high openness, appropriate size, and recency scores
- They are effectively a "wild card" in the ranking, typically appearing at ranks #10-25 rather than #1-3

This matches reality: Kresge, RWJ, and Kellogg DO fund everywhere and DO accept applications, but they should not be the LEAD foundations in any specific state's email.

### Lilly Endowment Example (Before vs After)

| State | Before (rank) | After (rank) | Why |
|-------|--------------|-------------|-----|
| IN | #1 | #1-#3 | 39.6% state_pct, high geo_score |
| IL | #4 | #12-#15 | 1.8% state_pct, low geo_score + mega penalty |
| CT | #3 | EXCLUDED | 0.25% state_pct = incidental tier |
| NY | #5 | EXCLUDED | 1.65% state_pct with only a handful of grants -- could be tertiary but mega penalty pushes it out of top 25 |

## 7. Pre-Generation Data Quality Report

### Problem

The first email batch was generated without any systematic data quality check. Problems were discovered AFTER emails were written, requiring expensive manual review and rewrites. Every issue found in the 6-perspective review could have been caught by automated pre-flight checks.

### Solution: Materialized View + Validation Script

```sql
-- Run before every email batch generation

-- 7a. PROSPECT QUALITY SUMMARY
CREATE OR REPLACE VIEW f990_2025.v_email_preflight_prospect_quality AS
SELECT
    np.ein,
    np.organization_name,
    np.state,
    np.ntee_code,
    np.contact_email,
    np.contact_first_name,
    np.email_quality_tier,
    np.email_quality_flags,

    -- Flag: Name is placeholder
    CASE WHEN np.contact_first_name IN ('there','Hi','Hello','Dear','Sir','Madam','whom','Team')
         THEN true ELSE false END AS flag_placeholder_name,

    -- Flag: Generic email
    CASE WHEN np.contact_email LIKE 'info@%' OR np.contact_email LIKE 'contact@%'
         OR np.contact_email LIKE 'office@%' OR np.contact_email LIKE 'admin@%'
         OR np.contact_email LIKE 'team@%' OR np.contact_email ILIKE 'INFO@%'
         THEN true ELSE false END AS flag_generic_email,

    -- Flag: Government email
    CASE WHEN np.contact_email LIKE '%.gov' OR np.contact_email LIKE '%.gov.%'
         THEN true ELSE false END AS flag_gov_email,

    -- Flag: Stale filer
    CASE WHEN np.most_recent_tax_year < 2022
         THEN true ELSE false END AS flag_stale_filer,

    -- Flag: Missing NTEE
    CASE WHEN np.ntee_code IS NULL
         THEN true ELSE false END AS flag_missing_ntee,

    -- Flag: Missing mission
    CASE WHEN np.mission_description IS NULL OR LENGTH(np.mission_description) < 10
         THEN true ELSE false END AS flag_missing_mission

FROM f990_2025.nonprofits_prospects2 np
WHERE np.contact_email IS NOT NULL;


-- 7b. COHORT QUALITY SUMMARY
CREATE OR REPLACE VIEW f990_2025.v_email_preflight_cohort_quality AS
SELECT
    cfl.state,
    cfl.ntee_sector,
    cv.foundation_count AS total_available_foundations,
    cv.display_text,
    COUNT(*) AS foundations_in_list,

    -- Flag: All foundations are mega (>$1B assets)
    COUNT(CASE WHEN fp.total_assets_eoy_amt >= 1000000000 THEN 1 END) AS mega_foundation_count,
    CASE WHEN COUNT(CASE WHEN fp.total_assets_eoy_amt >= 1000000000 THEN 1 END)
         = COUNT(*) THEN true ELSE false END AS flag_all_mega,

    -- Flag: Average geo concentration is low
    AVG(mgr.state_pct) AS avg_state_concentration,
    CASE WHEN AVG(mgr.state_pct) < 0.05
         THEN true ELSE false END AS flag_low_geo_relevance,

    -- Foundation diversity: how many distinct foundations?
    COUNT(DISTINCT cfl.foundation_ein) AS distinct_foundations

FROM f990_2025.cohort_foundation_lists cfl  -- or _v2
LEFT JOIN f990_2025.foundation_prospects2 fp ON fp.ein = cfl.foundation_ein
LEFT JOIN f990_2025.mv_foundation_geo_relevance mgr
    ON mgr.foundation_ein = cfl.foundation_ein AND mgr.state = cfl.state
LEFT JOIN f990_2025.cohort_viability cv
    ON cv.state = cfl.state AND cv.ntee_sector = cfl.ntee_sector
GROUP BY cfl.state, cfl.ntee_sector, cv.foundation_count, cv.display_text;


-- 7c. FOUNDATION OVER-CITATION CHECK
-- Any foundation appearing in more than 3 emails in the same batch
CREATE OR REPLACE VIEW f990_2025.v_email_preflight_overcitation AS
SELECT
    cfl.foundation_ein,
    cfl.foundation_name,
    COUNT(DISTINCT (cfl.state, cfl.ntee_sector)) AS cohort_appearances,
    ARRAY_AGG(DISTINCT cfl.state || '-' || cfl.ntee_sector) AS cohorts
FROM f990_2025.cohort_foundation_lists cfl
GROUP BY cfl.foundation_ein, cfl.foundation_name
HAVING COUNT(DISTINCT (cfl.state, cfl.ntee_sector)) > 3;


-- 7d. EXAMPLE GRANT QUALITY CHECK
-- Grants where the example recipient's NTEE sub-code doesn't match the cohort sector
```

### Gate Rule

**No email batch should be generated if the quality report shows:**
1. Any prospect in the batch with `flag_gov_email = true`
2. Any prospect with `email_quality_tier = 'D'`
3. Any foundation cited in more than 5 emails within the same batch
4. Any cohort where `flag_all_mega = true` (all foundations are $1B+)

These are hard blockers. The email writer should receive only clean, validated data.

### 8d. Materialized view for geographic relevance

```sql
CREATE MATERIALIZED VIEW IF NOT EXISTS f990_2025.mv_foundation_geo_relevance AS
SELECT
    fsd.foundation_ein,
    fsd.recipient_state AS state,
    fsd.state_pct,
    fsd.grant_count,
    fsd.total_grants AS state_giving_dollars,
    CASE
        WHEN fsd.state_pct >= 0.25 THEN 'primary'
        WHEN fsd.state_pct >= 0.05 THEN 'secondary'
        WHEN fsd.state_pct >= 0.01 AND fsd.grant_count >= 5 THEN 'tertiary'
        ELSE 'incidental'
    END AS geo_tier,
    CASE
        WHEN max_pct.max_state_pct < 0.15 THEN true
        ELSE false
    END AS is_national_funder
FROM f990_2025.stg_foundation_state_dist fsd
JOIN (
    SELECT foundation_ein, MAX(state_pct) AS max_state_pct
    FROM f990_2025.stg_foundation_state_dist
    WHERE recipient_state != '' AND recipient_state IS NOT NULL
    GROUP BY foundation_ein
) max_pct ON max_pct.foundation_ein = fsd.foundation_ein
WHERE fsd.recipient_state != '' AND fsd.recipient_state IS NOT NULL;

CREATE UNIQUE INDEX ON f990_2025.mv_foundation_geo_relevance (foundation_ein, state);

COMMENT ON MATERIALIZED VIEW f990_2025.mv_foundation_geo_relevance IS
    'Geographic relevance tier per foundation per state. Refresh with: REFRESH MATERIALIZED VIEW CONCURRENTLY f990_2025.mv_foundation_geo_relevance;';
```

### What This Report Does NOT Cover

- **Email copywriting improvements** (template rotation, subject line formulas, CTA optimization) -- these are downstream of the data pipeline
- **CAN-SPAM compliance** (physical address, unsubscribe mechanism) -- infrastructure issue, not data pipeline
- **Sender reputation/deliverability** (SPF/DKIM/DMARC, warm-up cadence) -- email infrastructure
- **A/B testing framework** -- campaign management layer

### Assumptions

- The `stg_foundation_state_dist` and `stg_foundation_sector_dist` tables are up to date and reflect current `fact_grants` data
- The `calc_foundation_profiles` table has been recalculated recently
- `nonprofit_returns.total_revenue` is the most recent filing year's revenue (needs verification per EIN)

### Risks

- The `pick_example_grant()` function uses LATERAL joins which may be slow for large foundations with thousands of grants. Consider adding an index on `fact_grants(foundation_ein, recipient_state, tax_year DESC)` if performance is an issue.
- The geographic monopoly exclusion for foundations with >60% in one state is aggressive. Some foundations (e.g., Heinz) do occasionally fund outside PA. The "allowed in home state" carve-out handles this, but review after the first batch.
- The openness_score calculation combines `accepts_applications` with `new_recipients_5yr / unique_recipients_5yr`. Foundations with only 3-5 recipients may have misleadingly high new-recipient rates. Consider adding a minimum `unique_recipients_5yr >= 10` threshold.

### Key Insight from Data Analysis

The database already contains almost everything needed for better foundation selection. The `stg_foundation_state_dist` table has per-state giving percentages. The `calc_foundation_profiles` table has `new_recipients_5yr`, `openness_score`, and `median_grant`. The `foundation_prospects2` table has `total_assets_eoy_amt` and `app_restrictions`. The problem was never missing data -- it was that the ranking algorithm used only ONE dimension (total_giving_5yr) and ignored all the other signals already computed.
