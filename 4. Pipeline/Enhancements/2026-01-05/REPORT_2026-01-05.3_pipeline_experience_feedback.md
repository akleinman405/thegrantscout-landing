## Executive Summary

Building grant opportunity reports for 5 clients (PSMF, RHF, SNS, HN, FCSD) revealed significant friction between the **algorithmic scoring pipeline** and the **curated research process**. The pipeline is designed to process the top 50-100 machine-scored foundations, but the highest-quality matches often come from **manual research** that identifies foundations not in the scored list.

**Key metrics:**
- 5 clients processed
- 20 total opportunities generated
- ~3 hours of pipeline work (including troubleshooting)
- RHF required complete workaround (0 opportunities initially, 5 after manual snapshot generation)
- 3 template files missing, 8 SQL templates missing (restored from archive)

**Bottom line:** The pipeline works, but it's two separate workflows awkwardly stitched together - algorithmic scoring and manual curation - and they don't communicate well.

## What I Wish I Had Known Going In

### 1. The Scored Foundations List is Incomplete for Curated Matches

**The Problem:** The pipeline scores 12,000+ foundations but only writes the top 100 to `02_scored_foundations.csv`. When you manually research and curate foundations (like I did with Weinberg, Woodruff, Kresge for RHF), those foundations often aren't in the top 100 scored list - so they have no snapshot data.

**What I Had to Do:** Generate snapshots manually using direct SQL queries for the 5 RHF curated foundations, then merge them into the 03_snapshots.json file.

**What Would Have Helped:** Knowing upfront that curated foundations need their own snapshot generation step.

### 2. Template Files Were Missing

**The Problem:** The render script (08) failed because `report_template.md` and `opportunity_template.md` were not in `4. Pipeline/templates/`. They existed in `4. Pipeline/archive/v1/rendering/templates/` but weren't copied during migration.

**What I Had to Do:** Manually copy templates from archive to templates directory.

**What Would Have Helped:** A setup script that validates all required files exist, or a migration checklist.

### 3. SQL Templates Were Missing

**The Problem:** The 03_get_snapshots.py script failed with `KeyError: 'annual_giving'` because the SQL templates weren't in `4. Pipeline/sql/`.

**What I Had to Do:** Copy 8 SQL files from `4. Pipeline/archive/v1/enrichment/sql/` to `4. Pipeline/sql/`.

**What Would Have Helped:** Same as above - validation script or migration checklist.

### 4. The Anthropic API Key Wasn't Set

**The Problem:** Narrative generation (step 06) ran in "dry-run" mode, producing generic fallback text instead of personalized AI-generated narratives.

**What I Had to Do:** Eventually wrote all narratives manually (which was the right call anyway for quality).

**What Would Have Helped:** Clearer indication in .env that API key is required for full functionality.

### 5. The "Viable Foundations" Step Expects Algorithmic Input

**The Problem:** Step 04 (filter_viable.py) expects input from step 03 (enrichment check), but when foundations are manually curated, you need to create 04_viable_foundations.json manually.

**What I Had to Do:** Create 04_viable_foundations.json files for all 5 clients by hand, populating enrichment data from my research.

**What Would Have Helped:** A clear path for "manual curation mode" vs "algorithmic scoring mode."

## Major Friction Points Encountered

### 1. Two Incompatible Workflows

**The Core Problem:** There are really two different workflows:

| Workflow A: Algorithmic | Workflow B: Curated |
|------------------------|---------------------|
| Score 12K foundations | Research specific foundations manually |
| Take top 100 | Select 5-10 best matches from research |
| Generate snapshots for scored list | Generate snapshots for curated list |
| Filter by viability | Already know viability from research |
| Assemble opportunities | Assemble opportunities |

The pipeline is designed for Workflow A, but the highest quality reports come from Workflow B. They need to be better integrated.

### 2. RHF Got Zero Opportunities

**What Happened:** All 5 curated RHF foundations (Weinberg, Woodruff, Kresge, Piper, Hilton) were filtered out in step 05 due to "No grant amount data."

**Root Cause:** Snapshots were generated for the top 50 *scored* foundations. The curated foundations weren't in the scored list, so they had no snapshot data.

**Resolution:** Manually generated snapshots via SQL queries, merged into 03_snapshots.json, re-ran step 05.

**Time Cost:** ~30 minutes of debugging and workaround.

### 3. Client Questionnaires Lack Specificity

**What Happened:** Many client profiles have empty or vague fields:
- `mission_statement`: empty for all 5 clients
- `grant_size_target`: empty for all 5 clients
- `specific_ask_text`: Some have it (RHF, FCSD), others are vague

**Impact:** Makes it harder to match foundations to specific client needs. Had to infer needs from other fields.

### 4. Enrichment Data Storage is Disconnected

**What Happened:** I researched 22 foundations manually and created detailed enrichment data. This data needed to be:
1. Stored in the database (foundation_enrichment table)
2. Added to 04_viable_foundations.json for each client
3. Referenced by step 05 when assembling opportunities

**The Problem:** No clear documentation on how enrichment data flows through the pipeline.

## Database and Data Quality Issues

### 1. Snapshot Data for Large Foundations is Incomplete

**Issue:** For major foundations like Weinberg ($505M giving), Woodruff ($681M), Kresge ($781M), the snapshot data was missing several fields:
- recipient_profile (all zeros)
- funding_trend (Unknown)

**Cause:** These foundations were queried directly via SQL, not through the full snapshot generation process which calculates these derived metrics.

**Recommendation:** Create a robust `generate_snapshot_for_ein()` function that produces complete data regardless of whether the foundation is in the scored list.

### 2. Column Name Confusion

**Issue:** fact_grants table has `recipient_name_raw`, not `recipient_name`. Had to debug this during snapshot generation.

**Recommendation:** Document table schemas more prominently, or create a data dictionary view that's easy to reference.

### 3. Contact Information Quality

**Issue:** Many foundations have outdated or missing contact info in pf_returns:
- Phone numbers without formatting
- Website URLs that may not be current
- No email addresses

**Recommendation:** Enrichment process should validate/update contact info from current 990s or website scraping.

## The Curated vs. Scored Foundation Problem

This deserves its own section because it's the fundamental architectural tension in the pipeline.

### The Scoring Model's Limitation

The LASSO V6.1 scoring model does an excellent job of **ranking foundations by likelihood of fit**. But:

1. It only surfaces the top 100 foundations
2. Major national funders (Weinberg, Woodruff, Kresge) often aren't in the top 100 for specific clients
3. These major funders are often the **best** matches because they have:
   - Open application processes
   - Documented funding priorities
   - Larger grant sizes
   - Professional staff

### The Curation Advantage

Manual curation can identify foundations that:
- Have explicit sector focus (e.g., "senior housing" for RHF)
- Have geographic alignment (e.g., Atlanta-based Woodruff for RHF's DeKalb project)
- Have recently announced new priorities (e.g., Woodruff's $75M Affordable Housing Fund)
- Accept unsolicited applications

### The Integration Problem

Currently, curated and scored foundations are processed through the same pipeline, but the pipeline assumes everything comes from the scoring model. This causes:

1. Curated foundations missing from snapshots
2. Curated foundations missing "connections" (board overlaps, shared funders)
3. Quality filter failures for curated foundations

### Proposed Solution

Create a parallel track:

```
Scored Track (automated):
  01 -> 02 -> 03 -> 04 -> 05

Curated Track (manual research):
  Research -> 03_curated -> 04_curated -> 05

Merge Track:
  05_scored + 05_curated -> 05_combined -> 06 -> 07 -> 08 -> 09
```
