# Analysis: Openness Feature Multicollinearity

**Date:** 2025-01-03
**Issue:** Three openness-related features are 86-99.8% correlated
**Impact:** LASSO will assign arbitrary coefficients, model will be unstable

---

## The Correlation Matrix

| Feature | openness_score | first_time_rate | one_time_rate |
|---------|----------------|-----------------|---------------|
| **openness_score** | 1.00 | **0.998** | 0.86 |
| **first_time_rate** | **0.998** | 1.00 | 0.86 |
| **one_time_rate** | 0.86 | 0.86 | 1.00 |

**Critical:** 99.8% correlation means these are measuring the same underlying construct.

---

## What Each Feature Measures

### `openness_score` (Composite Metric)
**Likely formula:**
```
openness_score = weighted_avg(
    first_time_rate * 0.4,
    (1 - repeat_rate) * 0.3,
    one_time_rate * 0.2,
    accepts_applications * 0.1
)
```

**Interpretation:** "How willing is this foundation to fund new organizations?"

**Range:** 0.0 (closed, only funds existing grantees) to 1.0 (fully open)

### `first_time_rate` (Component Metric)
**Formula:**
```
first_time_rate = (# grants to first-time recipients) / (# total grants)
```

**Interpretation:** "What % of this foundation's grants go to organizations they've never funded before?"

**Example:**
- Foundation gave 100 grants
- 30 went to orgs they'd never funded
- first_time_rate = 0.30

### `one_time_rate` (Component Metric)
**Formula:**
```
one_time_rate = (# recipients funded exactly once) / (# unique recipients)
```

**Interpretation:** "What % of this foundation's relationships are one-and-done?"

**Example:**
- Foundation funded 50 unique orgs
- 25 got funded only once (never repeated)
- one_time_rate = 0.50

---

## Why 99.8% Correlation?

**Hypothesis:** `openness_score` is *derived from* `first_time_rate` as the primary component.

If the formula is:
```python
openness_score = 0.7 * first_time_rate + 0.3 * other_stuff
```

Then the two features will be nearly identical (high weight on first_time_rate).

---

## What LASSO Will Do (Wrong)

With 99.8% correlation, LASSO's optimization becomes unstable:

### Scenario 1: Picks first_time_rate, zeros out openness_score
```
Coefficients:
  f_first_time_rate: +2.34
  f_openness_score: 0.00
  f_one_time_rate: +0.15
```

### Scenario 2: Picks openness_score, zeros out first_time_rate
```
Coefficients:
  f_first_time_rate: 0.00
  f_openness_score: +2.38
  f_one_time_rate: +0.12
```

### Scenario 3: Splits the weight arbitrarily
```
Coefficients:
  f_first_time_rate: +1.20
  f_openness_score: +1.18
  f_one_time_rate: -0.05  # WRONG SIGN!
```

**Problem:** Small changes in data (different train/test split) will produce wildly different coefficients.

---

## Impact on Model

### Model Performance
- **Predictive power:** Likely unaffected (AUC will be similar)
- **Why:** All three features carry the same information

### Model Interpretation
- **Coefficient stability:** ❌ Unstable across runs
- **Feature importance:** ❌ Can't say which matters
- **Sign interpretation:** ❌ May get wrong signs (positive feature with negative coefficient)

### Production Deployment
- **Reproducibility:** ⚠️ Re-training may flip coefficients
- **Client explanation:** ❌ Hard to explain "why we recommended this foundation"

---

## Recommended Fix

### Option 1: Keep Composite, Drop Components (RECOMMENDED)

**Keep:**
- `f_openness_score`

**Drop:**
- `f_first_time_rate`
- `f_one_time_rate`

**Rationale:**
- Openness score is validated in V5 (coefficient +0.46, correct sign)
- Conceptually meaningful to clients ("this foundation is open to new applicants")
- Single feature is easier to interpret

**SQL change:**
```sql
-- Comment out these lines in SQL_2025-01-02_v6_training_data.sql

-- NEW: First-time rate (historical tendency to fund new recipients)
-- COALESCE(cf.first_time_rate, 0.3) as f_first_time_rate,  -- REMOVED: 99.8% correlated

-- NEW: One-time rate (high = more willing to try new orgs once)
-- COALESCE(cf.one_time_rate, 0.5) as f_one_time_rate,  -- REMOVED: 86% correlated
```

---

### Option 2: Keep Components, Drop Composite

**Keep:**
- `f_first_time_rate`
- `f_one_time_rate`

**Drop:**
- `f_openness_score`

**Rationale:**
- More granular understanding of foundation behavior
- Can see if "tries once" is different from "funds new orgs"
- Components are raw data, composite is derived

**Problem:**
- Components are still 86% correlated
- May still get wrong signs

**Only choose this if:** You don't trust the openness_score formula and want to let LASSO re-learn the weights.

---

### Option 3: Principal Component Analysis

**Process:**
1. Compute PCA on [openness_score, first_time_rate, one_time_rate]
2. Take PC1 (explains ~95% of variance)
3. Use PC1 as single "openness" feature

**Rationale:**
- Mathematically optimal (captures shared variance)
- Guaranteed orthogonal to other features

**Cons:**
- Harder to interpret ("PC1" doesn't mean anything to clients)
- Requires extra preprocessing step

**Only choose this if:** You need to publish in a journal and want mathematical rigor.

---

### Option 4: Ridge Regression Instead of LASSO

**Change:** Use Ridge penalty (L2) instead of LASSO (L1)

**How Ridge handles multicollinearity:**
- Shrinks correlated features together
- Doesn't zero out features
- Produces stable coefficients

**Cons:**
- Gives dense models (all features have non-zero coefficients)
- Harder to deploy (can't prune unimportant features)

**Only choose this if:** You want to keep all features and don't care about sparsity.

---

## My Recommendation

**Go with Option 1: Keep `openness_score`, drop components.**

**Why:**
1. You already validated openness_score in V5 (correct sign, interpretable)
2. Single feature is cleaner for production deployment
3. Easier to explain to clients ("foundations open to new applicants")
4. Components don't add new information (99.8% correlation)

**Expected result:**
- `f_openness_score` coefficient: +0.5 to +1.5 (positive, as intended)
- Model interpretation: "Open foundations are X% more likely to fund new orgs"
- Stable across re-training runs

## Real-World Interpretation

### If `openness_score` coefficient = +0.8

**Meaning:**
- A 0.1 increase in openness score (e.g., 0.5 → 0.6) increases log-odds by 0.08
- In probability terms: ~2% absolute increase in funding probability
- Foundation with openness = 0.8 is 2x more likely to fund new org than openness = 0.4

**Client-facing explanation:**
> "We prioritize foundations with high 'openness scores' - those that historically
> fund many new organizations each year rather than only re-funding existing grantees."

## Conclusion

**Action:** Remove `f_first_time_rate` and `f_one_time_rate` from V6 training data.

**Benefit:**
- Eliminates 99.8% correlation
- Produces stable, interpretable coefficient
- Cleaner model for production

**Trade-off:**
- Lose granularity (can't see first-time vs. one-time separately)
- But: information is already captured in composite metric

**Expected V6 improvement:**
- More stable coefficients across CV folds
- Easier to interpret feature importance
- No negative coefficients on openness features
