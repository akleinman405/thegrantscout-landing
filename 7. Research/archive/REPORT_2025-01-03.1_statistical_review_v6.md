## Critical Issue #1: No Negatives in Validation/Test Sets (FATAL)

### The Problem

According to your description:
- **Training (2017-2022):** Has both positives and negatives
- **Validation (2023):** 94,597 positives, **0 negatives**
- **Test (2024):** 11,894 positives, **0 negatives**

### Why This Is Fatal

1. **Cannot compute AUC:** The Area Under the ROC Curve requires both classes. With only positives, you cannot calculate True Positive Rate vs False Positive Rate.

2. **Cannot evaluate calibration:** You won't know if a predicted probability of 0.3 actually means "30% chance of funding" without negative examples.

3. **Cannot detect overfitting:** High accuracy on 100% positive data is meaningless - the model could just predict 1.0 for everything and achieve "perfect" accuracy.

4. **Production deployment impossible:** When you score new foundation-nonprofit pairs, you'll have no idea if the model generalizes beyond 2022.

### Root Cause

From the SQL script (line 156-158):

```sql
CREATE TABLE stg_foundations_funded_new_recently AS
SELECT DISTINCT foundation_ein
FROM stg_first_time_grants
WHERE tax_year >= 2023;
```

Your positives are defined as first-time grants. For 2023-2024, you only collected **actual grants that occurred**, not the universe of pairs that **could have been funded but weren't**.

### Solutions (Ranked by Statistical Validity)

**Option A: Temporal holdout with synthesized negatives for 2023-2024 (RECOMMENDED)**

```sql
-- For validation/test, sample negatives from foundations active in those years
-- matched with recipients who DIDN'T receive funding

INSERT INTO stg_v5_training_pairs (foundation_ein, recipient_ein, tax_year, label, sample_type)
SELECT
    f.ein as foundation_ein,
    r.ein as recipient_ein,
    2023 as tax_year,
    0 as label,
    'hard_negative_2023' as sample_type
FROM (
    SELECT DISTINCT foundation_ein as ein
    FROM fact_grants
    WHERE tax_year = 2023
    LIMIT 10000  -- Sample of active foundations
) f
CROSS JOIN (
    SELECT ein
    FROM dim_recipients
    WHERE ein IN (SELECT recipient_ein FROM fact_grants WHERE tax_year <= 2022)
    ORDER BY RANDOM()
    LIMIT 50  -- 50 candidates per foundation
) r
LEFT JOIN fact_grants fg
    ON f.ein = fg.foundation_ein
    AND r.ein = fg.recipient_ein
WHERE fg.id IS NULL  -- Exclude pairs that actually got funded
LIMIT 500000;  -- Match positive count
```

**Pros:**
- Maintains temporal split (trains on past, predicts future)
- Realistic evaluation scenario
- Can detect concept drift

**Cons:**
- Need to carefully match negative sampling to positive distribution
- Risk of label leakage if recipients got funded elsewhere

**Option B: Random train/val/test split (ignore temporal ordering)**

Split the 2017-2022 data randomly into 60/20/20 splits.

**Pros:**
- Standard ML practice
- Guarantees balanced classes in all splits
- Easy to implement

**Cons:**
- **Loses temporal validation** - doesn't test "can we predict 2024 from 2017-2022 data?"
- May overestimate performance if foundation behavior changes over time
- Not representative of production use case

**Option C: Use 2017-2021 for train, 2022 for validation, 2023-2024 for qualitative review**

Keep temporal split but only evaluate quantitatively on 2022.

**Pros:**
- Still some temporal validation
- Avoids synthesizing negatives

**Cons:**
- Wastes 2023-2024 data
- Smaller validation set
- Doesn't answer "will this work in production?"

### My Recommendation

**Use Option A with hard negative sampling.** Your model is designed to predict "will this foundation fund this nonprofit in the next cycle?" - that's inherently a temporal prediction task. You need temporal validation.

For hard negatives in 2023-2024:
1. Sample foundations that were active (made grants) in those years
2. Sample recipients that existed and were "gettable" (filed 990s, similar size)
3. Exclude pairs where funding actually occurred
4. Match the negative sampling distribution to your training negatives (similar foundation size, sector, geography)

## Critical Issue #2: Feature Leakage in `f_funded_new_recently`

### The Problem

From SQL (line 218):
```sql
CASE WHEN fnr.foundation_ein IS NOT NULL THEN 1 ELSE 0 END as f_funded_new_recently
```

Where `fnr` is defined as (line 156):
```sql
SELECT DISTINCT foundation_ein
FROM stg_first_time_grants
WHERE tax_year >= 2023
```

### Why This Is Leakage

**For validation/test sets (2023-2024):**
- Every positive example is by definition a first-time grant from 2023-2024
- Therefore, `f_funded_new_recently = 1` for 100% of positives in 2023-2024
- If negatives were added for 2023-2024, they would likely have `f_funded_new_recently = 0` unless the foundation funded *other* new recipients

**Result:** The model will learn "if `f_funded_new_recently = 1`, then label = 1" with near-perfect correlation in validation/test sets.

### Is It Actually Leakage?

**Yes, for temporal splits.** Consider the causal timeline:

1. Foundation-nonprofit pair exists at start of 2023
2. Foundation decides whether to fund the nonprofit
3. **Grant occurs** → becomes a positive in your training data
4. **Grant is recorded** → foundation appears in `stg_first_time_grants` for 2023
5. Feature `f_funded_new_recently = 1` is set

The feature is **derived from the outcome you're trying to predict**. In production, you won't know if a foundation will fund new recipients in 2024 when you're making predictions in early 2024.

### Temporal Availability Test

Ask: "At the time we want to make a prediction, is this feature knowable?"

- **Training (2017-2022):** Using 2023-2024 data? **Not available at prediction time** ❌
- **Validation (2023):** Using 2023 data to predict 2023 outcomes? **Circular** ❌
- **Production (2025):** Using 2024 data? **Requires 1-year lag** ⚠️

### Fix Options

**Option 1: Lag the feature by 1 year**
```sql
-- Use 2021-2022 for training rows, 2022-2023 for validation, etc.
CASE WHEN EXISTS (
    SELECT 1 FROM stg_first_time_grants ftg
    WHERE ftg.foundation_ein = tp.foundation_ein
      AND ftg.tax_year BETWEEN (tp.tax_year - 2) AND (tp.tax_year - 1)
) THEN 1 ELSE 0 END as f_funded_new_recently
```

This gives you "did the foundation fund new orgs in the *past* 2 years?" rather than "in the same year we're predicting."

**Option 2: Drop the feature**

If you already have `f_first_time_rate` (historical rate of funding new recipients), `f_funded_new_recently` may be redundant.

**Option 3: Keep but use only pre-2023 data**

```sql
CASE WHEN EXISTS (
    SELECT 1 FROM stg_first_time_grants ftg
    WHERE ftg.foundation_ein = tp.foundation_ein
      AND ftg.tax_year BETWEEN 2021 AND 2022  -- Fixed window
) THEN 1 ELSE 0 END as f_funded_new_recently_2021_2022
```

This makes it clear the feature is frozen at 2022 knowledge.

### My Recommendation

**Use Option 1 with a 1-2 year lag.** The insight is valuable ("foundations recently funding new orgs are more likely to continue"), but you need temporal separation between feature generation and label.

## Critical Issue #3: Extreme Multicollinearity

### The Problem

You reported:
- `openness_score ↔ first_time_rate`: **99.8% correlation**
- `openness_score ↔ one_time_rate`: **86% correlation**
- `first_time_rate ↔ one_time_rate`: **86% correlation**

### Statistical Implications

**For LASSO regression:**

1. **Coefficient instability:** With 99.8% correlation, small changes in data will cause coefficients to swing wildly between the two features.

2. **Arbitrary selection:** LASSO will randomly pick one feature and zero out the other, or assign seemingly random weights (e.g., +2.5 to one, -2.3 to the other).

3. **Interpretation impossible:** You won't be able to say "openness_score matters more than first_time_rate" because they're measuring the same thing.

### Root Cause Analysis

How is `openness_score` calculated? If it's derived from `first_time_rate`, that explains the 99.8% correlation.

Looking at your SQL (line 190):
```sql
cf.openness_score as f_openness_score,
```

This comes from `calc_foundation_features`. Without seeing that definition, I suspect:

```sql
-- Likely definition (hypothesized)
openness_score =
    0.4 * first_time_rate +
    0.3 * (1 - repeat_rate) +
    0.2 * one_time_rate +
    0.1 * (accepts_applications::int)
```

If `first_time_rate` is weighted heavily in the composite, you get near-perfect correlation.

### Solutions

**Option 1: Drop redundant features (RECOMMENDED)**

Keep only `openness_score` if it's a validated composite metric. Drop `first_time_rate` and `one_time_rate`.

**Rationale:**
- Simpler model
- Easier interpretation ("openness" is conceptually meaningful)
- LASSO will give stable coefficient

**Option 2: Drop the composite, keep components**

Keep `first_time_rate` and `one_time_rate`, drop `openness_score`.

**Rationale:**
- More granular understanding
- Can see which dimension (first-time vs one-time) matters more
- But: they're 86% correlated, so still multicollinear

**Option 3: Principal Component Analysis**

Create a single "openness" factor from the three features.

**Rationale:**
- Captures shared variance
- Reduces 3 correlated features to 1 orthogonal component

**Cons:**
- Harder to interpret
- Requires additional preprocessing

**Option 4: Ridge regression instead of LASSO**

Ridge handles multicollinearity better by shrinking correlated coefficients together.

**Cons:**
- Won't zero out unimportant features
- Gives less sparse model (harder to deploy)

### My Recommendation

**Option 1: Keep only `openness_score`.** You spent V3-V5 fixing its sign and making it meaningful. The component features (`first_time_rate`, `one_time_rate`) are mechanically similar and add noise, not signal.

**Exception:** If you find `openness_score` is actually *derived from* the components, then drop `openness_score` and keep the components (Option 2).

## Additional Statistical Concerns

### 5. Training Set Composition (Positives Only from Accessible Foundations)

From SQL (line 40):
```sql
FROM stg_first_time_grants ftg
JOIN stg_accessible_foundations af ON ftg.foundation_ein = af.ein
```

**Issue:** Your positives are pre-filtered to "accessible" foundations (accepts applications, not preselected only).

**Consequence:** The model learns "among accessible foundations, which ones fund new orgs?" not "among all foundations, which ones fund new orgs?"

**Is this a problem?**

**No, if:** Your production use case only scores accessible foundations anyway.

**Yes, if:** You want to identify foundations that *aren't* marked as accessible but still fund new orgs (discovery use case).

### 6. Negative Sampling Strategy Not Documented

From SQL (line 73-75):
```sql
IF NOT EXISTS (SELECT 1 FROM information_schema.tables WHERE table_name = 'stg_v5_training_pairs') THEN
    RAISE EXCEPTION 'stg_v5_training_pairs not found. Run SQL_2025-01-02_v5_training_data_fast.sql first.';
END IF;
```

**Issue:** The V6 script assumes V5 negatives already exist but doesn't document how they were sampled.

**Why this matters:**
- If negatives are random pairs → easy negatives, inflated AUC
- If negatives are hard (same state, sector) → realistic difficulty, lower AUC
- If negatives are "rejected applications" → ideal but unavailable

**Need to verify:** What's the negative sampling strategy in `SQL_2025-01-02_v5_training_data_fast.sql`?

### 7. Class Imbalance (Assumed)

You mention positives in validation/test but didn't specify train class balance.

**Typical grant matching task:**
- Positives (actual grants): ~0.01-0.1% of all possible pairs
- Negatives (non-grants): ~99.9-99.99%

**If you sampled 1:1 positives to negatives:**
- Training AUC will be high (easy to distinguish)
- Production performance will be lower (class imbalance not reflected)

**If you kept natural imbalance:**
- Training will be harder
- Need to use weighted loss or resampling

**Recommendation:** Document the positive:negative ratio and justify the choice.
