# Create a Session State File (Handoff Document)

This command WRITES a new session state file. It does NOT read or load existing session files.

**This is NOT the /resume command.** Do NOT search for or read existing SESSION_STATE or CONTINUITY files. Your job is to CREATE a new one based on the current git state and conversation.

Task description: $ARGUMENTS

## Instructions

**STEP 0: Validate arguments.** If `$ARGUMENTS` is empty or blank, ask the user: "What task should I capture session state for?" Do NOT proceed until you have a task description.

**STEP 1: Gather git state** by running these in parallel:
- `git diff` to see all unstaged changes
- `git diff --cached` to see staged changes
- `git status` to list all modified/new/deleted files
- `git log --oneline -5` for recent commits

**STEP 2: Review this conversation** (the messages above in this chat session) to extract:
- Decisions made during this session (with rationale)
- Approaches that were tried and failed (and why)
- What's working vs. what's still broken
- Any partially implemented features
- Database queries that worked well
- Key data findings or insights

If this is a fresh conversation with no prior messages, skip this step and note "No conversation context available" in the Decisions section.

**STEP 3: Write the session state file** to `4. Pipeline/Enhancements/YYYY-MM-DD/SESSION_STATE_YYYY-MM-DD_description.md` (create directory if needed, use snake_case description from $ARGUMENTS):

```markdown
# SESSION STATE: [Task Description from $ARGUMENTS] - YYYY-MM-DD

**Last Updated:** YYYY-MM-DD
**Purpose:** Preserve context across sessions for [task description]
**Status:** In Progress / Blocked / Ready for Review

---

## CURRENT STATUS - WHERE WE ARE

### Current Step
[Exactly what the next session should do first. Be specific — file names, function names, line numbers, SQL queries.]

### Remaining Steps
1. [Most important — first thing to do]
2. [Second priority]
3. [Third priority]

---

## What's Done

## What's Partially Done
- [In-progress items — what works, what doesn't yet]

## What's Not Started

---

---

---

---

```

**STEP 4: Confirm** by showing the user the file path and a brief summary of what was captured (file count, next step #1).
